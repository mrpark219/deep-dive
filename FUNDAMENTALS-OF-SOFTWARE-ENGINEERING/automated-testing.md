# 자동 테스트: 좋은 소프트웨어를 만드는 최고의 습관

## 1. 자동 테스트 (Automated Testing)

- 유지보수하기 좋고 높은 품질의 소프트웨어를 만드는 일도 코드 작성, 리뷰, 테스트 과정에서 좋은 습관을 꾸준히 실천한 결과이다.
- 높은 품질의 소프트웨어를 작성하는 일은 규칙적인 연습이 필요한 훈련이지만, 시간이 지나면 테스트에 들인 노력이 동료와 미래의 자신 모두가 고마워할 훌륭한 **안전장치**가 되어줄 것이다.

## 2. 자동 테스트의 이점

- 기존 코드를 검증하려고 추가 코드를 작성하는 일에 그만큼 노력을 들일 가치가 있는지 의문이 생길 수 있다.
- 하지만 자동 테스트는 다음과 같은 중요한 가치를 지닌다.

### 2.1. 문서 역할을 하는 테스트

- 이미 진행 중인 프로젝트에 합류하는 경우, 프로젝트 관련 문서가 제대로 갖춰진 경우는 드물다.
- 운이 좋다면 선임 개발자나 도메인 전문가를 통해 온보딩할 수도 있다.
- 하지만 이러한 상황이 늘 가능한 것은 아니며, 문서가 빈약하거나 아예 없을 때 **테스트**가 있다면 큰 도움이 된다.
- 예를 들어 특정 카드 결제 처리 문제를 해결해야 하는 상황이라고 가정한다.
  - 적절한 문서나 테스트가 없다면 낯설고 복잡해 보이는 코드를 일일이 뒤지거나 동료의 도움에 의존해야 한다.
  - 이때 전체 기능을 검증하는 포괄적인 **테스트 모음(Test Suite)** 이 있다면, **모듈(Module)** 이 올바르게 작동하는지 파악하고 관련 코드가 어디에 있는지 쉽게 알 수 있다.
  - 실패하는 테스트를 따라가다 보면 전체 코드 안에서 문제를 일으킨 원인을 쉽게 찾을 수 있다.

### 2.2. 유지보수성 향상

- 유지보수하기 좋은 코드를 작성하는 기술을 익히려면 보통 수년이 걸린다.
- 테스트를 코드보다 먼저 작성하는 것은 말하기 전에 먼저 생각하는 것과 같다.
- 생각을 정리하고 설계 구조(Architecture)를 잡는 데 도움을 주기 때문이다.
- 초보자의 주된 목표는 대개 코드를 돌아가게 만드는 것뿐이지만, 테스트를 작성하면 개선이 필요한 부분을 훨씬 빨리 발견할 수 있다.
- 코드를 어떻게 테스트할지 고민하다 보면 자연스럽게 설계상의 결함이 무엇인지 파악하게 된다.
- 이 과정을 통해 비대해진 로직을 별도의 클래스로 **리팩터링**할 수 있으며, 결과적으로 테스트와 유지보수가 훨씬 쉬워진다.
- 예를 들어 하나의 메서드(Method)가 여러 기능을 수행한다면 테스트하기가 까다롭다.
- 이처럼 테스트 가능성을 고려하다 보면 설계 문제를 쉽고 빠르게 파악할 수 있으며, 테스트를 기반으로 안전한 리팩터링이 가능해진다.

### 2.3. 자신감 상승

- 자동 테스트를 구축하면 자유롭게 코드를 작성할 수 있다는 자신감이 생긴다.
- 소프트웨어 개발은 끊임없이 코드를 쓰고 실험하며 리팩터링하는 반복적인 과정이다.
- 테스트가 없다면 기존 기능을 망가뜨릴까 봐 걱정하며 코드를 작성할 수밖에 없다.
- 또한 새로운 기술이나 창의적인 해결책을 시도하기 두려워질 수도 있다.
- 하지만 탄탄한 테스트 모음이 있다면 신뢰할 수 있는 **안전망**을 얻은 셈이다.
- 코드 변경과 리팩터링은 시스템을 완벽히 이해하지 못했거나 **부작용(Side Effect)** 을 예측하기 어려워 스트레스가 심한 일이다.

#### 단단한 테스트 모음이 있다면

- 확신을 가지고 코드 리팩터링을 수행할 수 있다.
- 코드 변경이 기존 기능을 망가뜨리지 않았는지 바로 확인할 수 있다.
- **운영 환경**에 배포하기 전에 잠재적인 결함을 파악할 수 있다.

#### 포괄적인 테스트 모음을 작성하고 유지한다면

- 시스템 동작을 더 명확하게 이해할 수 있다.
- 개발 초기 단계에 버그를 잡아내어 수정 비용과 노력을 줄일 수 있다.
- **회귀 버그(Regression Bug, 이전 버전에서 잘 작동하던 기능에 다시 버그가 생기는 현상)의 발생 가능성을 줄인다**.
- 자동 테스트를 통해 기능이 **요구사항(Specification)** 을 충족하는지 검증하고 코드 품질을 유지할 수 있다.

### 2.4. 일관성과 반복 가능성

- **수동 테스트(Manual Testing)** 방식은 실수하기 쉽고, 반복하기 어려우며, 많은 인력과 시간이 필요하다.
- 자동 테스트는 언제나 일관되고 반복 가능한 결과를 제공한다.
- 사람과 달리 **테스트 스크립트(Test Script)** 는 매번 실수나 누락 없이 정확히 똑같은 단계를 수행한다.

## 3. 자동 테스트 유형

- 테스트를 해야 하는 이유를 알았다면 이제 무엇을 테스트해야 하고, 무엇을 테스트하지 말아야 할지 알아야 한다.
- 자동 테스트에는 **UI 테스트**, **종단 간(E2E) 테스트**, **통합 테스트**, **단위 테스트** 등 다양한 유형이 포함된다.
- **테스트 피라미드**는 소프트웨어 테스트에서 중요한 개념이지만, 종종 잘못 알려지거나 완전히 무시되기도 한다.
  - 테스트는 **단위 테스트**, **통합 테스트**, **종단 간 테스트** 순으로 더 많이 수행해야 한다.
- 자동 테스트의 3가지 주요 유형은 다음과 같다.
  - **단위 테스트(Unit Test)**: 개별 구성 요소나 함수를 시스템의 나머지 부분과 격리한 상태에서 테스트한다. 각 요소 자체가 올바르게 작동하는지 확인하는 것이 목적이다.
  - **통합 테스트(Integration Test)**: 시스템의 각기 다른 구성 요소나 모듈이 하나의 결합된 단위로서 어떻게 함께 연결되어 작동하는지 검증한다.
  - **종단 간 테스트(E2E Test)**: 가장 앞단의 사용자 인터페이스(UI)부터 가장 뒷단의 백엔드 시스템까지 애플리케이션 전체 흐름을 테스트한다.
- 이러한 피라미드 구조는 소프트웨어 테스트의 근본적인 **트레이드오프(Trade-off)** 관계에 따라 형성된다.
- 피라미드 위로 올라갈수록 테스트 실행 속도는 느려지고, 유지보수 비용은 비싸지며, 테스트가 깨질 가능성(Brittle)은 커진다.

### 3.1. 단위 테스트(Unit Test)

- 단위 테스트는 **테스트 피라미드(Test Pyramid)** 의 기반을 형성하며, 테스트 모음에서 가장 큰 부분을 차지한다.
- 단위 테스트는 코드가 지켜야 할 **계약(Contract)** 이라고 생각해야 한다.
- 코드의 기능이 바뀌면 해당 단위 테스트가 깨지면서 잠재적인 문제를 알려줘야 한다.
- 단위 테스트는 개별 구성 요소나 함수를 격리해서 검사하며, 각 단위 코드 자체가 올바르게 작동하는지 확인한다.
- 단위 테스트는 **정적 분석(Static Analysis)**, **코드 리뷰(Code Review)** 와 함께 결함에 맞서는 최전방 방어선 역할을 한다.
- 빠르게 작성하고 실행할 수 있어야 하며, 개발 과정에서 피드백을 즉시 제공해야 한다.
- 단위 테스트가 실패하면 대개 특정 함수나 코드 줄을 가리키기 때문에 디버깅이 수월하다.
- 이러한 속도와 정밀함 덕분에 단위 테스트는 기능 개발과 **지속적 통합(CI, Continuous Integration)** 과정에서 핵심적인 역할을 한다.

### 3.2. 통합 테스트(Integration Test)

- 통합 테스트는 시스템의 각기 다른 구성 요소나 모듈이 한 단위로 결합되어 어떻게 맞물려 작동하는지 철저히 검증하는 과정이다.
- 단위 테스트가 격리된 개별 코드 단위에 집중한다면, 통합 테스트는 이러한 단위들 사이의 상호작용을 검사하며 더 넓은 범위를 다룬다.
- 하지만 전체 시스템의 성능과 기능을 평가하는 완전한 **시스템 테스트(System Test)** 에 비하면 여전히 범위가 좁다.
- 통합 테스트는 단위 테스트로 개별 검증된 구성 요소가 서로 결합될 때 발생하는 문제를 잡는 데 유용하며, 통합된 시스템이 올바르고 효율적으로 작동하는지 확인해 준다.
- 범위와 복잡성 때문에 통합 테스트는 실행 시간이 더 오래 걸리고 실행 빈도도 낮다.

### 3.3. 종단 간 테스트(E2E Test, End-to-End Test)

- 종단 간 테스트는 가장 앞단의 사용자 인터페이스(UI)에서 시작해 가장 뒷단의 백엔드 시스템까지 애플리케이션 전체를 다루는 포괄적인 테스트 절차다.
- 이러한 테스트는 실제 사용자 시나리오를 시뮬레이션하도록 설계되어 애플리케이션이 실제 환경에서 예상대로 작동하는지 확인한다.
- 본질적으로 복잡하고 철저하기 때문에 종단 간 테스트는 대개 느리고 자원과 시간이 많이 든다.
- 종단 간 테스트는 깨지기 쉽고(Brittle) **거짓 음성(False Negative)** 에 더 취약할 수 있다.
  - 기능 자체의 문제보다는 UI 요소의 작은 변경 때문에 테스트가 실패할 수 있기 때문이다.
- 따라서 종단 간 테스트가 실패하면 문제를 파악하고 수정하는 데 상당한 디버깅 노력이 필요할 수 있다.

### 3.4. 테스트하지 말아야 할 것

- 포괄적인 테스트가 소프트웨어 품질에 필수적이지만, 무엇을 테스트할지 전략적으로 판단하는 일도 그만큼 중요하다.
  - 우리가 작성한 코드에 집중해야 하며, 언어 자체의 기능이나 프레임워크 코드를 테스트할 필요는 없다.
  - **게터(Getter)**, **세터(Setter)**, 빌더 메서드, 자동 생성된 **데이터 전송 객체(DTO, Data Transfer Object)** 처럼 단순하게 생성된 코드는 테스트하지 말아야 한다.
  - **비공개 메서드(Private Method)** 를 테스트하는 일은 피해야 한다. 대신 이 비공개 메서드를 호출하는 **공개 인터페이스(Public Interface)** 를 테스트하는 데 집중해야 한다.
  - 테스트가 외부 서비스에 의존하지 않게 작성해야 한다. 단위 테스트에는 **모의 객체(Mock Object)** 를, 통합 테스트에는 **테스트 대역(Test Double)** 을 대신 사용해야 한다.

## 4. 코드 실행률(Code Coverage)

- **코드 실행률(Code Coverage)** 은 전체 코드 중에서 테스트가 수행되는 동안 실행된 코드의 비율을 측정하는 지표다.
- 코드 실행률이 높다는 것은 대부분의 코드를 검증했다는 의미이며, 코드 실행률이 낮다는 것은 버그가 숨을 수 있는 사각지대가 존재한다는 뜻이다.
- 코드 실행률이 중요하기 때문에 다양한 **통합 개발 환경(IDE, Integrated Development Environment)** 에서 이 기능을 기본적으로 지원한다.
- **CI/CD 파이프라인(CI/CD Pipeline)** 에 통합되어 최소 코드 실행률 임계치(Threshold)를 강제하는 도구들도 있다.
- 실행률이 임계치 이하로 떨어지면 기준 미달로 인해 **코드 병합(Code Merge)** 이 차단된다.
- 코드 실행률 측정 도구는 테스트 실행 결과를 분석하여 어떤 줄(Line), 어떤 분기(Branch), 어떤 함수가 실행되었는지 정확히 보여주는 리포트를 생성한다.
- **행 단위 실행률(Line Coverage)** 85%나 **분기 단위 실행률(Branch Coverage)** 72% 같은 백분율을 통해 테스트 중에 코드가 얼마나 실행되었는지 정량적으로 파악할 수 있다.
- 코드 실행률 지표가 유용한 통찰을 제공하기는 하지만, 단순히 실행률 수치만 높이기 위해 테스트를 작성하는 경우에는 한계가 있다.
- 메서드를 호출하기만 할 뿐, 유효한 검증(Assertion)을 전혀 수행하지 않는 의미 없는 테스트도 실행률 수치에는 반영되기 때문이다.
- 100% 실행률이라는 수치 자체를 목표로 삼기보다는, 실행률을 우리가 작성하는 테스트의 품질을 개선하는 피드백 수단으로 활용해야 한다.
- 핵심적인 **비즈니스 로직(Business Logic)** 영역에서 실행률이 낮다면, 이는 더 주의를 기울여야 할 코드 영역이라는 뜻일 수 있다.
- 반면 단순히 데이터를 전달하는 클래스나 설정 코드를 테스트하느라 실행률이 높은 것은, 테스트가 굳이 필요 없는 영역에 비용을 쓰고 있다는 신호일 수 있다.
- 결국 동작을 올바르게 검증하는 의미 있는 테스트를 작성하는 데 집중해야 하며, 코드 실행률은 추가적인 관심이 필요한 코드로 개발자를 안내하는 나침반 용도로 사용해야 한다.

## 5. 테스트 작성

### 5.1. 시작 방법

- 어떤 프로그래밍 언어나 프레임워크, 혹은 메타 프레임워크를 사용하든 간에 다양한 테스트 도구를 사용할 수 있다.
- 프로젝트를 생성한 후에는 어떤 테스트 프레임워크를 사용할지 결정해야 한다. **자바(Java)** 환경이라면 **JUnit**이 표준이다.
- 테스트를 작성하는 방식은 크게 두 가지로 나뉜다.
  - **테스트 선행 방식(Test-First)** 과 **테스트 후행 방식(Test-Last)** 이다.
  - **TDD(Test-Driven Development, 테스트 주도 개발)** 같은 테스트 선행 개발 방법은 **'레드-그린-리팩터링(Red-Green-Refactor)'** 주기를 따른다.
    - 먼저 실패하는 테스트(Red)를 작성하고, 테스트를 통과할 수 있는 최소한의 코드(Green)를 구현한 뒤, 마지막으로 동작을 바꾸지 않으면서 코드를 개선(Refactor)한다.
  - 테스트 후행 방식은 기능을 먼저 구현한 다음에 테스트 코드를 작성한다.
- 테스트는 **인공지능(AI)** 의 도움을 받아 작성하기에 매우 적합한 영역이다.
  - AI를 활용하면 팀의 확립된 컨벤션과 패턴을 유지하면서도 테스트 절차의 속도를 획기적으로 높일 수 있다.

### 5.2. 단정문(Assertion)

- **단정문(Assertion)** 은 프로그램 실행 중에 특정 조건이 충족되는지 검증하기 위해 사용하는 함수나 메서드다.

### 5.3. 단위 테스트 작성

- 단위 테스트는 독립적으로 실행되며 수행 속도가 매우 빠른 격리된 테스트다.
- 테스트 중인 시스템의 특정 컴포넌트, 모듈 또는 소프트웨어의 일부를 가리켜 **테스트 대상 시스템(SUT, System Under Test)** 이라고 부른다.
- 단위 테스트는 SUT만을 집중적으로 테스트하는 것을 목표로 삼는다.
- 이때 정상적인 흐름뿐만 아니라 **경계 조건(Boundary Condition)** 까지 확실하게 검증해야 한다.

### 5.4. 모의 객체(Mock Object)

- 소프트웨어 테스트에서 **모의 객체(Mock Object)** 는 SUT가 의존하는 구성 요소를 시뮬레이션하는 대체 버전이며, 이러한 맥락에서 **테스트 대역(Test Double)** 이라고도 부른다.
- 모의 객체를 활용하면 코드를 테스트하기 좋게 격리된 상태로 만들 수 있다.
- 단위 테스트를 작성할 때 **의존 구성 요소(Dependency)** 란 테스트 대상 시스템이 작업을 완료하기 위해 의존하는 다른 모든 클래스나 모듈을 의미한다.
- 단위 테스트에서는 모의 객체가 의존 구성 요소를 격리하는 목적으로 단순하고 유용하게 쓰이지만, 통합 테스트에서는 이보다 훨씬 까다로운 문제가 된다.
  - 통합 테스트는 의존 관계에 있는 여러 구성 요소가 서로 어떻게 맞물려 돌아가는지 검증하는 것이 목적이기 때문이다.
- 여러 의존 관계를 관리하고, 각각 올바른 상태인지 확인하며, 구성 요소 간의 상호작용을 처리하다 보면 통합 테스트의 복잡성이 급격히 증가할 수 있다.
- 자바 생태계에서는 대표적인 모의 객체 프레임워크로 **모키토(Mockito)** 를 사용한다.

### 5.5. 통합 테스트 작성

- 단위 테스트를 작성할 때는 의존 요소를 모의 객체로 대체하여 대상 클래스를 격리했다.
- 반면 통합 테스트에서는 다른 의존 요소와 실제로 어떻게 상호작용하는지 검증해야 한다.
- 보통 접미사나 폴더명에 **Integration Test(INT)** 를 명시하여 통합 테스트임을 나타낸다.
- 실제 애플리케이션에서는 **의존성 주입(DI, Dependency Injection)** 프레임워크를 사용해 인스턴스를 주입받지만, 경우에 따라 테스트 환경 구성을 위해 수동으로 객체를 생성하기도 한다.

### 5.6. 종단 간 테스트 작성

- 백엔드 영역에서의 종단 간 테스트는 주로 **API 엔드포인트(API Endpoint)** 검증에 집중한다.
- 이 테스트를 실행하려면 실제 운영 환경과 유사한 서블릿 컨테이너가 필요한데, 스프링 생태계에서는 **톰캣(Tomcat)** 서버가 내장된 **스프링 부트(Spring Boot)** 를 활용한다.
- 목표는 실제 서버가 구동 중인지 확인하고, HTTP 요청을 보내어 응답과 상태를 검증하는 것이다. 주요 과정은 다음과 같다.
  - 무작위 포트(Random Port) 설정을 통한 포트 충돌 방지
  - **MockMvc** 또는 **WebTestClient**를 사용한 서버 측 스프링 MVC 테스트 지원 기능 활용
  - 각 테스트 실행 전에 데이터베이스(DB) 상태를 초기화
  - 사용자 등록 등의 API 요청 시뮬레이션 수행
  - 반환된 응답 결과와 실제 데이터베이스의 데이터 상태를 상호 검증
- 이러한 방식을 통해 API 요청부터 데이터베이스 상호작용까지의 전체 흐름을 테스트함으로써 시스템이 종단 간에 올바르게 작동하는지 확인할 수 있다.

## 6. 실천해보기

- **오픈소스 라이브러리에서 학습**:
  - 잘 만들어진 오픈소스 라이브러리의 테스트 코드를 읽어본다.
  - 새로 알게 된 라이브러리의 기능이나 자신이 기존에 가졌던 테스트 관행과의 차이점을 분석하고, 유용한 테스트 전략을 프로젝트에 도입하는 것을 고려한다.
- **테스트를 문서로 활용**:
  - 전체 코드에서 익숙하지 않은 도메인이나 모듈을 작업하게 되었을 때는 기존 테스트 코드를 이정표로 삼는다.
  - 만약 테스트가 없거나 부족하다면 코드를 파악해 나가는 과정에서 새로운 테스트를 추가하며 문서를 보완한다.
- **기존 코드 개선**:
  - 프로젝트에 테스트가 부족하다고 느껴진다면 모든 문제를 한꺼번에 해결하려 하지 말고 점진적으로 개선해 나간다.
  - 팀원들이 하루에 최소 하나의 새로운 테스트 코드를 작성하도록 독려하는 문화가 도움이 된다.
- **테스트와 함께 새로운 기능 작업 시작**:
  - 새로운 요구사항이나 기능을 맡게 되면 구현에 앞서 테스트 작성부터 시작하는 습관을 들인다.
- **테스트로 버그 해결**:
  - 생산 환경이나 QA 과정에서 버그를 발견하면, 해당 버그를 똑같이 재현하는 실패하는 테스트부터 작성한다.
  - 이 테스트는 버그가 수정되기 전까지 실패(Red) 상태를 유지하므로, 결함 수정 여부를 확실하게 검증할 수 있는 수단이 된다.
  - 이 과정에서 코드의 가독성과 유지보수성을 높이기 위한 **리팩터링(Refactoring)** 기회도 확보할 수 있다.
- **테스트 옹호**:
  - 적절한 테스트가 있었다면 막을 수 있었던 장애나 결함을 발견했을 때는 이를 팀의 학습 기회로 삼는다.
  - 문제를 잡아내는 테스트를 즉시 추가하고, 향상된 코드 실행률과 테스트 자산이 미래에 비슷한 문제를 어떻게 예방할 수 있는지 동료 및 관리자와 논의한다.
- **연습**:
  - **코드 카타(Code Kata)** 와 같이 구조적이고 반복적인 연습 방법을 통해 테스트 작성 기술과 근육을 꾸준히 키워나간다.
