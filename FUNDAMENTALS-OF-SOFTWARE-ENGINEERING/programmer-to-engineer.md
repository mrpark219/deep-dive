# 프로그래머에서 이제는 엔지니어로

## 1. 소프트웨어 엔지니어

- 소프트웨어 엔지니어가 되려면 비즈니스 요구사항 분석, 아키텍처 설계, 유지보수 가능한 코드 구현, 테스트 작성, UI 개발, 데이터 관리, CI/CD 기반의 운영 환경 배포까지 전 영역에 걸친 공학적 기술을 익혀야 한다.
- 단순히 코드를 작성하는 일(Coding)을 넘어 소프트웨어 생명 주기(SDLC) 전반의 공학 체계를 이해하고 적용해야 진정한 엔지니어링 영역으로 진입할 수 있다.
- 프로그래머가 코드 생성이라는 단일 작업에 집중하고, 개발자가 소프트웨어 전달의 큰 그림과 수명 주기를 탐구한다면, **소프트웨어 엔지니어는 전체 수명 주기에 공학적 접근법을 적용해 확장성, 신뢰성, 효율성, 보안을 극대화하는 사람**이다.
- 알고리즘과 설계를 피상적으로 이해하는 수준을 넘어, 규칙을 명확히 알 뿐만 아니라 **언제 그 규칙을 유연하게 적용하고 깨뜨려야 하는지**까지 본질적으로 이해해야 한다.

## 2. 기초의 중요성

- 코드 작성의 기본기는 나와 협업할 동료의 작업을 단순화하고 비용을 줄여주는 강력한 도구다. 기술은 끊임없이 변하므로 새로운 패러다임을 빠르게 습득할 수 있는 **기반 지식**이 중요하다.
- 많은 프로그래머가 경력 성장에 필수적인 **소프트 스킬(Soft Skills)** 과 변하지 않는 기본 원칙을 무시한 채, 최신 기술 유행이나 신규 언어 스펙에만 집착하는 경향이 있다.
- 탄탄한 기본기는 당장 화려해 보이지 않더라도 정체된 경력과 더 넓은 범위의 책임을 가질 기회 사이의 차이를 만드는 핵심 요인이다.
- 소프트웨어 산업에서의 성공은 단순 기술 습득이 아닌 **문제 해결 능력, 실험 정신, 그리고 창의적인 사고방식**에 달려 있다.

## 3. 교육받은 지식과 알아야 할 지식

- 대부분의 개발자는 교육 과정이나 이상적인 환경의 그린필드(Greenfield) 프로젝트를 통해 코드를 새롭게 작성하는 방법 위주로 배운다.
- 그러나 실제 현업 소프트웨어 엔지니어는 **대부분의 시간을 기존 애플리케이션, 즉 레거시 코드(Legacy Code)를 분석하고 다루는 데 소모**한다. 여기에는 과거에 내려진 결정으로 인한 오버헤드와 기술 부채가 필연적으로 포함된다.
- 따라서 타인과 협업하는 방법, 가독성 높은 코드를 읽고 쓰는 능력, **레거시 코드를 안전하고 효과적으로 리팩터링하는 기술**이 현업에서 살아남기 위해 가장 필요한 진짜 지식이다.

## 4. 게으른 프로그래머 정신의 수용

- '게으른 프로그래머 정신'은 나태함이 아니라 **생산성과 효율성에 초점을 맞춘 공학적 철학**이다.
- 신입 엔지니어는 본능적으로 눈앞의 코드 구현에 급급하지만, 전략적으로 한 걸음 물러서서 생각할 시간을 확보하는 것이 궁극적으로 더 나은 해결 방안을 만드는 지름길이다.
- 우리가 마주한 특정 문제는 이미 전 세계의 누군가가 해결했을 가능성이 매우 크다.
- 오픈소스 라이브러리나 검증된 기존 디자인 패턴을 탐색하는 데 시간을 투자하여 **바퀴를 재발명하지 않는 것(Don't reinvent the wheel)** 이 효율성의 핵심이다.

## 5. 새로운 시각의 가치

- 조직에 새로 합류한 신입 개발자나 구성원에게는 과거의 관성에 얽매이지 않는 **편견 없는 새로운 시각**이 존재한다.
- 이러한 시각은 과거의 비효율적인 결정을 발견하고 오래된 가정에 의문을 제기함으로써, 기존 시스템을 개선하는 혁신적인 해결책의 시발점이 된다.
- 도메인 내에서 이상하거나 이해되지 않는 구조를 마주했을 때 질문하기를 주저하지 말고, 상황과 맥락에 대해 팀원들에게 적극적으로 설명을 요청하는 태도가 필요하다.

## 6. 성급한 해결책 제시의 위험성

- 문제를 맞닥뜨렸을 때 너무 빨리 코딩(해결책)으로 뛰어들려는 충동을 억제해야 한다. 임시방편의 해법은 당장의 증상만 가릴 뿐, 장기적으로 시스템에 더 큰 부작용을 야기한다.
- 결함을 바로잡는 비용을 최소화하려면 **문제를 최대한 프로세스의 앞 단계에서 조기에 발견(Shift-Left)** 해야 한다.

| 활동                | 엔지니어링 관점의 가치 전환                                                                |
| :------------------ | :----------------------------------------------------------------------------------------- |
| **페어 프로그래밍** | 며칠 후에 진행되던 비동기 코드 리뷰를 **실시간 품질 보장 활동**으로 전환한다.              |
| **지속적인 테스트** | 버그가 발생한 시점과 가장 가까운 타이밍에 문제를 식별하여 **오류 수정 비용을 최소화**한다. |
| **지속적 통합(CI)** | 코드를 조기에 자주 병합함으로써 형상 관리 중 발생하는 **코드 충돌의 고통을 경감**한다.     |

- 소프트웨어 개발에서 가장 빈번하고 치명적인 난제는 기술적 오류가 아닌 **요구사항에 대한 오해**에서 비롯된다.
- 오해로 인한 실패 비용을 줄이는 유일한 방법은 **해결해야 하는 문제 본질에 집중**하는 것이다. 고객이 요청하는 명시적 요구사항 이면의 **근본적인 필요(Root Cause)** 와 제약 조건을 파악하는 데 충분한 시간을 투자해야 한다.
- 개발자는 종종 해결책을 지나치게 복잡하게 오버 엔지니어링(Over-engineering)하는 경향이 있으므로, 비즈니스 맥락을 끊임없이 주시하며 문제에 적합한 **가장 단순하고 올바른 도구**를 선택해야 한다.

## 7. 소프트웨어의 황금률 적용

- 비즈니스와 일상 전반에는 **남에게 대접받고 싶은 대로 남을 대접하라**는 황금률이 존재하며, 이는 소프트웨어 엔지니어링 세계에도 동일하게 적용된다.
- 코드를 작성할 때는 언제나 **내 뒤에 이 코드를 이어받아 유지보수할 동료 개발자**를 배려해야 한다. 가독성이 떨어지는 코드는 동료의 생산성을 파괴하는 기술 부채가 된다.
- 가장 중요한 사실은 내 코드를 이어받을 **다음 개발자가 실은 '미래의 나'일 가능성이 매우 크다는 점**이다. 내가 짠 코드라도 몇 달이 지나면 기억이 희미해지기 마련이다.
- 코드는 컴퓨터에 내리는 명령문이기 전에, 동료들과 비즈니스 맥락을 공유하는 **궁극의 의사소통 수단**이다.
- 따라서 컴파일러의 해석 속도가 아닌 **사람의 인지 속도에 맞춰 읽기 쉬운 코드를 작성(Optimize for readability)** 해야 한다.
- 시장에서 대체 불가능한 탁월한 소프트웨어 엔지니어로 차별화되고 싶다면, 기교 넘치는 복잡한 코드보다 이 황금률을 묵묵히 따르는 것이 지름길이다.

## 8. 정리

- 소프트웨어 엔지니어로서의 커리어는 모니터를 응시하며 기계적으로 코드를 쏟아내는 행위보다 훨씬 더 넓고 깊은 공학적 역량을 의미한다.
- 문법적으로 완벽한 코드를 생산하는 프로그래머를 넘어, **전체 수명 주기를 제어하고 비즈니스 문제를 해결하는 엔지니어**로 성장해야 한다.
- 향후 새로운 기능 개발이나 버그 수정 태스크를 할당받는다면, 즉시 키보드에 손을 올리고 타이핑을 시작하던 과거의 습관을 잠시 멈춰야 한다.
- 코드를 치기 전 **내가 해결하려는 문제의 본질과 제약 조건을 완벽히 이해했는지** 스스로 검증하는 단계를 반드시 거친다.
- 이미 검증된 기존 라이브러리나 디자인 패턴이 있는지 탐색하고, 화이트보드에 도메인 개념을 스케치하거나 **컴포넌트 간의 관계를 가볍게 모델링**해 보는 습관을 지녀야 한다.
- 필요하다면 경험이 풍부한 동료와 설계 선택지를 논의하여 최적의 방향을 정립한 뒤 비로소 구현에 착수하는 것이 올바른 엔지니어의 자세다.
