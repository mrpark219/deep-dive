# 전개(Deployment) 파이프라인

- **전개 파이프라인(Deployment Pipeline)** 은 소스 코드 저장소에 커밋된 최신 코드를 프로덕션 운영 인프라에 이르기까지 안전하고 정합성 있게 이동시키기 위한 자동화된 로드맵이다.
- 성공적인 지속적 인도 아키텍처를 구축하기 위해, 파이프라인은 다음과 같이 긴밀하게 상호 연결된 4가지 핵심 구성 요소로 설계된다.

## 1. 전개 환경(Deployment Environments)

- 격리된 로컬 개발 장비부터 최종 고객이 실시간으로 애플리케이션과 상호작용하는 물리 운영 인프라에 이르기까지, 코드가 단계별로 배포되어 생명력을 갖는 독립된 인프라 영역이다.
  - **개발 환경(Local/Dev Environment)**: 개별 엔지니어가 자신의 로컬 PC나 샌드박스 서버에서 기능을 구현하고 빠르게 피드백을 확인하는 단계다.
  - **테스트/QA 환경(Test/QA Environment)**: 빌드된 산출물의 기능적 결함, 버그, 비즈니스 유스케이스 명세 적합성을 전문 QA 팀이나 자동화 테스트 스위트를 통해 정밀 검증하는 환경이다.
  - **스테이징 환경(Staging/Pre-live Environment)**: 실제 프로덕션 서버와 100% 동일한 하드웨어 스펙, 네트워크 토폴로지, 그리고 마스킹된 운영 복제 데이터셋을 구성하여 배포 프로세스 자체의 아키텍처적 결함을 최종 검어하는 릴리스 직전 단계다.
  - **운영 환경(Production Environment)**: 실제 서비스 트래픽을 수용하며 최종 사용자에게 비즈니스 가치를 실시간으로 제공하는 물리 인프라 최상위 환경이다.

## 2. 버전 관리 전략(Branching Strategy)

- 대규모 멀티 워커 개발 조직에서 다수의 엔지니어가 서로의 소스 코드를 덮어쓰거나 충돌(Conflict)을 일으키지 않으면서, 단일 프로젝트 저장소의 형상을 안정적으로 제어할 수 있도록 조율하는 브랜치 관리 워크플로우다.
- 견고한 버전 관리 전략을 도입하면 현재 실행 중인 실시간 프로덕션 서비스에 영향을 주지 않으면서 원하는 시점에 기민하게 비즈니스 기능을 릴리스할 수 있고, 전개 중 예기치 못한 시스템 장애 발생 시 특정 커밋 시점으로 초고속 **롤백(Rollback)** 대응이 가능하며, 컴플라이언스 규제 준수 목적의 상세한 코드 변경 기록 및 감사 이력(Audit Trail)을 온전히 보존할 수 있다.
- 대표적인 형상 관리 전략으로는 엄격한 브랜치 생명주기를 가져가는 **Git-Flow** 시스템이나, 메인 브랜치 중심의 지속적 통합에 최적화된 가벼운 **GitHub-Flow** 등이 존재한다.

## 3. 전개 자동화(Deployment Automation)

- 소스 코드가 격리된 각 전개 환경 사이를 이동할 때 엔지니어가 수동으로 명령어를 입력하거나 인프라를 조작하면서 발생하는 휴먼 에러(Human Error) 가능성을 원천적으로 배포 파이프라인에서 제거하는 기술이다.
- 전개 자동화 시스템은 사람이 개입하지 않아도 매 배포 요청마다 동일한 스크립트 시퀀스에 따라 신뢰할 수 있는 무중단 절차를 기계적으로 대리 실행한다.
- 자동화 파이프라인 내부에서 일관되게 처리하는 주요 프로세스는 다음과 같다.
  - 소스 코드 컴파일, 라이브러리 의존성 분석 및 배포 가능한 바이너리 아카이브 패키징
  - 단위 테스트, 통합 테스트 및 소스 코드 정적 보안 취약점 유효성 검증
  - 대상 환경의 대상 서버 인프라 프로비저닝 및 런타임 환경 변수 동적 바인딩
  - 실제 대상 인스턴스로의 패키지 전개 실행 및 구버전 프로세스 스와핑
  - 배포 완료 직후 시스템 헬스 체크(Health Check) 및 통합 관측 지표 검증
- 이 모든 과정은 셸 스크립트나 코드 형태의 인프라(IaC) 구성을 통한 **전개 스크립팅(Deployment Scripting)** 을 통해 명세화된다.
- 전개 프로세스를 완벽히 자동화하더라도 운영 환경의 네트워크 타임아웃 등 예기치 못한 장애 변수가 상존하므로, 배포 비정상 실패 감지 시 즉각 이전의 안전했던 빌드 상태로 시스템을 되돌리는 자동화된 **원상복구(Rollback) 스크립트** 를 반드시 파이프라인 내부에 기본 자산화해 두어야 한다.

### 3.1. 실무 핵심 전개 전략(Deployment Strategies)

- **일괄 전개(Big Bang / Recreate)**:
  - 현재 가동 중인 구버전 애플리케이션 인스턴스를 한꺼번에 완전히 중단(Down)시킨 후, 새 버전의 패키지를 전체 서버에 통째로 교체 적재하는 방식이다.
  - 아키텍처 구현과 데이터베이스 스키마 하위 호환성 관리가 매우 직관적이고 간단하지만, 배포 프로세스 진행 도중 일시적인 서비스 다운타임(Downtime)이 불가피하게 발생하며, 새 버전에 치명적인 버그가 있을 경우 전체 유저가 즉시 장애 서킷에 노출된다는 높은 위험성을 안고 있다. 시스템 일시 중단이 비즈니스적으로 허용되는 내부 어드민 도메인이나 소규모 토이 프로젝트에 적합하다.
- **점진적 전개(Phased / Progressive Approach)**:
  - 새 버전의 기능적 안정성을 프로덕션 환경에서 먼저 점진적으로 검증해 나가며 출시 범위를 단계적으로 확장하는 고도화된 배포 방식이다.
  - 이를 통해 초기 배포 단계에서 잠재적 버그를 조기에 감지하고 전체 시스템으로 장애가 전파되는 반경(Blast Radius)을 최소화하며 방향을 수정할 수 있다.
  - 대표적으로 전체 인프라 노드 중 아주 소량의 서버(예: 5%)에만 신버전을 먼저 배포하여 실 트래픽 반응을 살피는 **카나리 전개(Canary Deployment)** 와, 구버전 인스턴스를 하나씩 순차적으로 새 버전으로 교체해 나가는 **롤링 전개(Rolling / 순차 전개)** 방식이 존재한다.
  - 두 기법 모두 구버전과 신버전 인프라를 동시에 띄워두고 로드 밸런서(Load Balancer) 레벨에서 트래픽 가중치를 동적으로 스위칭하는 **블루-그린(Blue-Green)** 아키텍처와 결합하여, 유저가 인프라 교체 순간을 전혀 인지하지 못하도록 완벽한 **무중단 배포(Zero-Downtime Deployment)** 를 달성한다. 대규모 엔터프라이즈 서비스나 정밀한 모니터링이 필요한 핵심 도메인에 필수적으로 요구된다.

## 4. CI/CD(지속적 통합 및 지속적 인도)

- 엔지니어가 로컬에서 구현한 코드를 Git 형상 관리 시스템에 푸시(Push)하여 풀 리퀘스트(PR)를 제출하는 바로 그 순간부터, 최종 운영 인프라에서 안정적으로 구동될 때까지의 전 과정을 단 하나의 자동화 파이프라인으로 연결하는 핵심 데브옵스(DevOps) 패러다임이다.
- **지속적 통합(CI, Continuous Integration)**:
  - 전체 개발 팀원들이 각자 변경한 소스 코드를 메인 공유 브랜치에 매일 정기적으로 자동 통합하고, 빌드 및 테스트 스위트를 구동하여 코드 정합성을 상시 검증하는 프로세스다. 통합 시점에 발생하는 아키텍처적 충돌과 논리적 버그를 조기에 발견할 수 있도록 돕는다.
- **지속적 전달(CD, Continuous Delivery)**:
  - CI 단계를 무사히 통과하여 검증된 소스 코드를 언제든 프로덕션 운영 환경에 릴리스할 수 있도록 준비된 상태(Artifact 빌드 완료)를 유지하는 프로세스다. 운영 전개 직전 단계까지 파이프라인이 자동 제어하며, 최종 프로덕션 배포 스위치는 담당자의 수동 승인을 거쳐 실행된다.
- **지속적 전개(CD, Continuous Deployment)**:
  - 지속적 전달 단계에서 한 걸음 더 나아가, CI 테스트를 성공적으로 통과한 아티팩트를 별도의 유인 승인 절차 없이 시스템 판단하에 프로덕션 운영 환경까지 100% 자동으로 실시간 출시(Release)하는 극대화된 자동화 단계다.
- CI/CD 파이프라인 아키텍처 구축을 통해 기업은 환경 간 배포 일관성 확보, 휴먼 에러 없는 상시 자동화 테스트 수행, 개발 팀의 배포 스트레스 대폭 감소, 통합 병목 문제의 조기 차단, 그리고 비즈니스 아이디어가 실제 운영 환경에 반영되기까지의 소요 시간(Lead Time)을 기하급수적으로 단축할 수 있는 막대한 비즈니스 경쟁력을 확보하게 된다.
- 일반적인 파이프라인은 `[Source Code] -> [Build] -> [Test] -> [Package] -> [Deploy]`의 연속적인 파이프라인 스테이지 시퀀스를 따르며, 현대 백엔드 생태계에서는 **GitHub Actions** 나 **Jenkins(젠킨스)** 등의 전문 CI/CD 도구를 도입하여 파이프라인 인프라를 구성 및 자동화한다.
