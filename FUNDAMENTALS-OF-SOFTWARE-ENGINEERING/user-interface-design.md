# 엔지니어라면 꼭 알아야 할 UI 설계 기초

## 1. 사용자 인터페이스(UI, User Interface)

- 많은 경우에 **사용자 인터페이스(UI, User Interface)** 설계는 다른 엔지니어링 요소에 대한 깊은 고려 없이, 그저 입력 항목을 빈 공간에 적당히 배치하는 일로 치부되곤 했다.
- 그러나 UI는 **사용자 경험(UX, User Experience)** 의 실체이며, 거의 모든 사용자에게는 눈에 보이는 UI가 곧 애플리케이션 그 자체임을 잊지 말아야 한다.

## 2. 모두를 위한 디자인(Universal Design)

- **접근성(Accessibility)**, **사용성(Usability)**, **포용성(Inclusivity)** 은 사용자의 배경이나 신체적 능력과 상관없이 다양한 사람이 제품을 불편 없이 사용할 수 있도록 만드는 것을 목표로 삼는다.
- 참고로 기술 표준 전파 과정에서 접근성은 **A11y**, **지역화(Localization)** 는 **L10n**, **국제화(Internationalization)** 는 **I18n**과 같이 영문 글자 수를 활용한 약어로 축약하여 표기하기도 한다.
- 이러한 요소들은 애플리케이션의 시장 성패를 결정할 뿐만 아니라, 국가에 따라 법률이나 장애인 차별 금지 규제의 대상이 되기도 하므로 설계 단계부터 반드시 반영해야 한다.

### 2.1. 사용성(Usability)

- 사용성은 소프트웨어 세계의 치실과 같다.
- 모두가 중요하다고 입을 모아 말하지만, 실제 프로젝트 일정 압박 속에서 가장 먼저 무시되곤 하기 때문이다.
- 사용성은 "사용자가 이 애플리케이션을 얼마나 쉽게 사용할 수 있는가?"라는 한 가지 질문으로 요약된다.
- 이를 평가하는 세부 기준은 다음과 같다.
  - **학습 용이성(Learnability)**: 사용자가 애플리케이션의 인터페이스와 조작 방식을 처음 접했을 때 얼마나 배우기 쉬운가?
  - **효율성(Efficiency)**: 사용자가 원하는 목적을 달성하기 위해 얼마나 효율적(최소한의 클릭 수, 불필요한 화면 이동의 최소화)으로 작업할 수 있는가?
  - **기억 용이성(Memorability)**: 한동안 애플리케이션을 쓰지 않다가 다시 돌아왔을 때, 인터페이스의 사용법을 얼마나 쉽게 다시 기억해 낼 수 있는가?
  - **발견 용이성(Discoverability)**: 시스템이 제공하는 다양한 핵심 기능을 사용자가 화면 안에서 얼마나 직관적으로 찾아낼 수 있는가?
  - **오류 처리(Error Handling)**: 사용자가 잘못된 입력을 하거나 시스템에 문제가 생겼을 때, 애플리케이션이 오류를 안내하고 복구하도록 돕는 방식이 매끄러운가?
  - **사용자 만족도(Satisfaction)**: 애플리케이션의 전반적인 상호작용을 경험한 후 사용자가 느끼는 주관적인 만족도와 기분은 어떠한가?
  - **접근성(Accessibility)**: 시각, 청각 또는 지체 장애를 가진 사용자를 위해 인터페이스가 스크린 리더나 키보드 전용 내비게이션 등의 보조 수단을 얼마나 충실히 지원하는가?

### 2.2. 접근성(Accessibility)

- **접근성(Accessibility)** 은 사용자의 신체적·인지적 능력이 서로 달라도 차별 없이 소프트웨어를 활용할 수 있도록 만드는 데 집중한다.
- 모든 사용자가 인터페이스를 탐색하고 유기적으로 상호작용할 수 있어야 한다.
- 소프트웨어의 접근성을 높이는 것은 단순히 장애가 있는 사람뿐만 아니라, 일시적 제약을 겪는 사람을 포함한 모든 사용자에게 결론적으로 도움이 된다.
- UI 설계 시 주로 다음과 같은 제약 조건을 염두에 두어야 한다.
  - **시각적 제약**: 전맹(실명)부터 저시력, 색약 및 색맹 등
  - **청각적 제약**: 청각 장애, 난청, 또는 소리를 들을 수 없는 환경적 제약 등
  - **기동성 제약**: 마우스를 정밀하게 조작하기 어려운 운동 장애나 반복성 긴장 장애(RSI) 등
  - **인지적 제약**: 자폐증이나 난독증 같은 질환부터 언어 읽기 능력 및 의사소통 능력이 부족한 경우까지

### 2.3. 국제화(I18n)와 지역화(L10n)

- **국제화(I18n, Internationalization)** 는 애플리케이션이 다양한 언어와 문화적 배경에 유연하게 적응할 수 있도록 아키텍처를 설계하고 구축하는 과정이다.
  - 예를 들어 화면에 노출되는 텍스트를 소스 코드에 하드코딩(Hardcoding)하지 않고, 다국어 속성 파일(Properties/JSON 등)에서 키(Key) 값으로 불러오도록 UI 프레임워크를 구조화하는 작업이 이에 해당한다.
- 반면 **지역화(L10n, Localization)** 는 그렇게 국제화된 인터페이스를 특정 국가나 시장의 언어와 문화에 맞게 실제로 최적화하는 구체적인 변환 과정이다.
  - 즉, 현지 정서에 맞는 문구로 번역을 진행하고 현지 화폐 단위, 날짜 및 시간 표기법, 도량형(측정 단위) 등을 정확하게 적용하는 실무 작업을 뜻한다.

### 2.4. 사용자 파악

- 인터페이스를 설계할 때는 제품을 사용할 대상(Target Audience)과 그들이 처한 실제 환경을 정확히 이해해야 한다.
- 주로 사용자들이 도메인 지식이나 컴퓨터 조작에 익숙한지, 어떤 운영체제(OS)와 디바이스를 주로 사용하는지, 신체적 제약이 있는지, 저해상도 모니터나 화면 확대(Zoom) 상태에서 레이아웃이 깨지지 않는지 등을 검토해야 한다.
- 시스템은 **"사용하기 쉬운 방법이 곧 올바른 방법"** 이 되도록 직관적으로 설계해야 한다.
- 애플리케이션이 유용하고 가치 있을수록 사용자의 만족도는 비례하여 상승하며, 모든 사용자는 시스템으로부터 존중받을 자격이 있음을 명심해야 한다.

### 2.5. 2차 사용자(Secondary User)

- 제품의 시스템을 직접 조작하는 주 사용자(Primary User)에게 집중하는 것도 좋지만, 시스템의 결과물에 간접적으로 영향을 받는 **2차 사용자(Secondary User)** 의 경험도 간과해서는 안 된다.
- 예를 들어 우리가 고객 서비스 상담원에게 전화로 문의하거나 자동차 정비소에서 정비 예약을 진행할 때, 직원이 조작하는 내부 시스템의 편의성에 따라 최종 서비스를 받는 우리(콜센터 고객, 차량 소유주)가 바로 2차 사용자가 된다.

### 2.6. 개발자는 사용자가 아니다

- 실제 엔드 유저(End User)들은 개발자라면 전혀 하지 않을 만한 예외적인 행동을 실생활에서 기꺼이 행한다.
- 예를 들어 글씨가 잘 보이지 않는다는 이유로 브라우저나 OS 화면을 200% 이상 과도하게 확대하여, UI 요소의 일부 영역이 화면 밖으로 밀려나 보이지 않는 상태로 앱을 사용하기도 한다. 개발자의 기준과 상식으로 사용자의 행동을 재단해서는 안 된다.

### 2.7. 문화의 영향

- 아랍어처럼 오른쪽에서 왼쪽으로 글을 읽는 RTL(Right-to-Left) 언어권 국가를 대상으로는 전체 UI 레이아웃의 방향과 흐름을 완전히 뒤집어야(Mirroring) 한다.
- 또한 우리에게는 지극히 당연하고 무해한 제스처나 표현이 특정 문화권의 사용자에게는 심각한 모욕이나 금기 사항이 될 수 있다.
- 이러한 문화적 차이는 아이콘이나 일러스트의 시각적 디자인에도 큰 영향을 미친다.
- 따라서 글로벌 소프트웨어를 출시할 때는 사용자가 누구인지, 소프트웨어를 어떤 환경에서 어떻게 쓸지 미리 철저하게 조사해야 사용자를 불편하게 만드는 설계 오류를 피할 수 있다.

## 3. 사용성 극대화

- 위에서 언급한 모든 사용성 요소를 극대화하기에 앞서, "이 애플리케이션이 궁극적으로 어떻게 사용될 것인가?"라는 본질적인 질문을 던져야 한다.
- 예컨대 **CAD(Computer-Aided Design)** 프로그램처럼 도구 자체가 태생적으로 복잡하고 전문가가 하루 종일 상주하며 써야 하는 시스템이라면, 초기 학습 비용이 들더라도 숙련자의 생산성을 높여주는 **효율성(Efficiency)**, **기억 용이성(Memorability)**, **사용자 만족도(Satisfaction)**가 최우선 지표가 된다.
- 반면 대중을 타깃으로 하는 **소셜 네트워크 서비스(SNS)**나 커머스 앱이라면 첫 진입 장벽을 낮추는 **학습 용이성(Learnability)**과 **발견 용이성(Discoverability)**이 훨씬 중요하다. 인터페이스가 직관적이지 않으면 사용자는 즉시 서비스를 이탈하기 때문이다.
- 즉, 핵심 타깃 유저의 구성과 사용 시나리오를 바탕으로 애플리케이션의 비즈니스 목적과 사용 맥락을 철저하게 고려하여 설계해야 한다.

## 4. 디자인 원칙

### 4.1. 대비(Contrast)

- **대비(Contrast)** 는 시각적 위계를 세우고 가독성을 확보하기 위해 사용할 수 있는 가장 강력한 디자인 도구 중 하나다.
- 흔히 대비라고 하면 두 색상의 톤 차이만을 떠올리기 쉽지만, 서로 다른 속성을 가진 두 가지 요소라면 무엇이든 대비를 만들어낼 수 있다.
  - 큰 글꼴(Title)과 작은 글꼴(Body)의 크기 차이
  - 굵은 두께(Bold)와 얇은 두께(Regular)의 굵기 차이
  - 어두운 배경과 밝은 오브젝트의 명도 차이
  - 작은 이미지와 거대한 이미지의 스케일 차이
- 대비를 효과적으로 사용하면 화면 디자인이 사용자의 시각적 흥미를 강하게 끌게 할 수 있다. 이때 주의할 점은 애매한 차이를 두지 말고, 확실하게 구별되도록 과감하게 속성 차이를 부여해야 한다는 것이다.
- 또한 대비는 화면 정보의 구조를 잡아주는 조직화 도구의 역할도 수행한다. 사용자는 페이지를 가볍게 훑어보는 스캐닝(Scanning) 과정에서 대비를 통해 디자인의 우선순위와 중요도를 즉각 파악할 수 있어야 한다.

### 4.2. 반복(Repetition)

- 특정 핵심 시각 요소가 시스템 전반에 걸쳐 일관되게 **반복(Repetition)** 되면, 애플리케이션에 친숙함과 단단한 통일감을 부여할 수 있다.
- 상단에 고정된 **네비게이션 바(Navigation Bar)**, 일관된 스타일의 버튼 구조, 규칙적인 헤더 레이아웃 등에서 가장 자주 볼 수 있으며, 그 효과는 사용자가 인지하지 못할 정도로 미묘하고 자연스럽게 스며든다.
- 반복은 사용자에게 자신이 여전히 동일한 시스템 영역 내에 안전하게 머무르고 있다는 심리적 안정감을 제공한다.
- 사용자는 반복되는 레이아웃과 컴포넌트 스타일에 금방 익숙해지므로, 일상적인 작업 중에 굳이 신경 쓰지 않고 무시해도 되는 영역을 자연스럽게 필터링하여 인지 비용을 아낄 수 있다.

### 4.3. 정렬(Alignment)

- **정렬(Alignment)** 은 실천하기 가장 쉬우면서도 효과가 확실한 원칙으로, 화면의 모든 시각 요소를 격자(Grid) 구조 위에 신중하게 배치하는 것을 의미한다.
- 요소를 규칙적으로 정렬하면 화면이 전체적으로 깔끔하고 세련된 인상을 주며, 산만하게 흩어져 있던 시각 요소들이 시각적으로 하나로 단단히 묶이게 된다.
- 정렬 원칙을 엄격하게 적용하면, 화면 안에서 서로 멀리 떨어져 있는 항목이라 할지라도 보이지 않는 선으로 연결된 듯한 정돈된 느낌을 줄 수 있다.

### 4.4. 근접성(Proximity)

- **근접성(Proximity)** 은 사용자가 화면의 정보를 인지하고 관계를 평가하는 방식에 매우 중요한 역할을 미친다.
- 게슈탈트(Gestalt) 심리학 법칙에 따르면, 시각적으로 물리적인 거리가 가깝게 묶여 있는 항목들은 설령 기능적으로 아무런 관련이 없더라도 서로 밀접한 관계가 있는 하나의 그룹으로 인식된다.
- 따라서 논리적으로 연관성이 없는 항목들은 절대로 함께 붙여서 배치하지 말아야 한다. 화면 안에서 정보의 구분이 필요한 시각적 단위를 만들 때는, 반드시 그 사이에 충분한 빈 여백(White Space)을 확보하여 경계를 명확히 나누어 주어야 한다.

## 5. 좋은 오류 메시지의 중요성

- 애플리케이션의 전반적인 사용성에서 오류 상황에 시스템이 얼마나 적절하게 대응하는지는 서비스의 신뢰도를 결정하는 핵심 요인이 된다.
- 가장 좋은 설계는 애초에 오류가 발생하지 않도록 UI 레벨에서 예방하는 것이겠지만, 사용자가 불가피하게 정상 경로에서 벗어났다면 명확하고 의미 있는 메시지로 다음 행동을 안내해야 한다.
- 잘 작성된 오류 메시지는 사용자가 현재 겪고 있는 문제를 스스로 디버깅하고 해결할 수 있도록 돕는 훌륭한 가이드가 된다.
- 오류 메시지를 구성할 때는 단순히 에러가 발생했다는 사실만 전하는 것을 넘어, **무엇이 잘못되었는지(Cause)**를 명확히 알리고 문제를 해결할 수 있는 **대안(Actionable Solution)**을 함께 제시해야 한다.

## 6. 파괴적인 작업(Destructive Action)

- 사용자가 모든 작업을 물 흐르듯 쉽게 처리할 수 있게 만들라는 일반적인 UX 원칙에서, 거의 유일하게 예외가 인정되는 영역이 바로 **파괴적인 작업(Destructive Action)**을 처리할 때다.
- 데이터 영구 삭제나 계정 탈퇴와 같이 되돌릴 수 없는 치명적인 결정에 관해서는, 사용자의 실수를 막기 위해 의도적으로 확인 과정을 몇 단계 추가하는 것이 올바른 설계다.
- 이때 시각적 **대비(Contrast)**를 역으로 활용하는 것도 좋은 방법이다. 예를 들어 일반적인 확인 버튼은 무채색으로 두고, 데이터가 삭제되는 버튼만 강렬한 빨간색 경고 톤으로 배치하여 사용자가 행동의 위험성을 인지하도록 유도해야 한다.
- 추가적인 팝업이나 텍스트 입력 확인 절차가 다소 과도해 보일 수 있으나, 사소한 부주의로 인해 소중한 데이터가 실수로 삭제되어 돌이킬 수 없는 피해를 입는 최악의 상황을 방지하는 것이 훨씬 이득이다.
