# 데이터 이전과 변환

- 소프트웨어 엔지니어링 환경에서 개발 팀은 비즈니스 성장에 발맞추어 주기적으로 데이터베이스 시스템을 마이너/메이저 업그레이드하거나, 도메인 특성에 맞는 다른 유형의 데이터베이스로 전환하거나, 외부 서비스와 시스템을 연동하거나, 데이터 분석을 위해 여러 원천(Source)의 대규모 데이터를 하나로 결합하는 등의 변환 작업을 정기적으로 수행해야 한다.

## 1. 데이터 이동 기초

- 단일 인스턴스 환경에서 한 시스템에서 다른 시스템으로 데이터를 단순히 복사하는 문제는 표면적으로 꽤 간단해 보일 수 있다.
- 하지만 프로덕션 운영 환경에서 데이터 유실 없는 성공적인 데이터 이전을 달성하기 위해서는 데이터 구조에 대한 정밀한 분석과 더불어 신중한 사전 계획, 단계별 실행, 데이터 정합성 검증(Validation), 그리고 장애 발생 시 즉각 대응할 수 있는 **원상복구(Rollback)** 계획이 필수적으로 수반되어야 한다.
- 아키텍처 상황에 따라 선택할 수 있는 대표적인 이전 전략은 다음과 같다.

### 1.1. 일괄 이전 대 점진적 이전

- **일괄 이전(Big Bang Migration / 빅뱅 마이그레이션)**:
  - 시스템 운영을 일시적으로 중단하고, 약속된 점검 시간(Downtime) 동안 모든 데이터를 한 번에 통째로 목적지 저장소로 이관하는 방식이다.
  - 데이터 동기화 구조가 단순하여 기술적으로는 상대적으로 쉬운 접근 방식일 수 있지만, 마이그레이션 도중 예기치 못한 예외가 발생했을 때 롤백 비용이 매우 크고 대규모 시스템일수록 비즈니스 공백 리스크가 높다는 위험이 따른다.
- **점진적 이전(Phased Migration / 단계적 마이그레이션)**:
  - 거대한 도메인 데이터셋을 범주나 유저 그룹별로 세분화하여 단계별로 나누어 이관하는 방식이다.
  - 일괄 이전처럼 모든 데이터를 한 번에 옮기지 않고 여러 구간으로 나누어 차례대로 이전하기 때문에, 전체 서비스를 오랫동안 중단해야 하는 부담을 줄일 수 있다. 경우에 따라서는 기존 시스템을 계속 운영하면서 일부 데이터나 일부 사용자 그룹만 먼저 새 시스템으로 이전하는 방식도 가능하다.
  - 이전을 안전하게 완수하기 위해서는 정밀한 스케줄링 계획을 세워야 하며, 최초 이관 시에는 문제가 발생하더라도 전체 서비스 및 비즈니스에 미치는 영향이 가장 적은 소규모 테스트 그룹이나 비핵심 도메인 영역부터 시작하는 보수적인 접근이 권장된다.

### 1.2. ETL(Extract, Transform, Load) 절차

- **ETL(Extract, Transform, Load)** 은 서로 다른 이기종 시스템 간에 대규모 데이터를 유기적으로 이동시키는 핵심 엔지니어링 프로세스다.
  - **추출(Extract)**: 하나 또는 여러 원천 시스템(Source System)으로부터 필요한 원천 데이터를 안전하게 읽어오는 단계다.
  - **변환(Transform)**: 추출한 원천 데이터를 타겟 시스템의 비즈니스 명세, 스키마 형식, 데이터 품질 요구사항에 맞게 정제, 가공, 매핑하는 단계다.
  - **적재(Load)**: 최종 변환이 완료된 정형 데이터를 목적지 시스템(Target System)의 데이터베이스나 데이터 웨어하우스에 안정적으로 인서트(Insert)하는 단계다.
- ETL 파이프라인의 각 단계는 단일 책임 원칙에 따라 명확히 격리되어 작동하지만, 대규모 데이터를 다루는 특성상 단계별로 고유한 아키텍처적 난제가 발생한다.
  - **추출 단계의 난제**: 원천 시스템의 API 속도 제한(Rate Limit)에 걸리거나, 과도한 조회 쿼리로 인해 현재 실시간 트래픽을 처리 중인 프로덕션 운영 시스템의 성능에 악영향을 미치는 문제가 발생할 수 있다.
  - **변환 단계의 난제**: 소규모 데이터로 진행한 **개념 검증(PoC, Proof of Concept)** 테스트 단계에서는 발견되지 않았던 데이터 포맷 누락, 자료형 불일치, 깨진 문자열 등의 예기치 못한 데이터 품질(Data Quality) 문제가 대량 연산 중에 종종 터져 나온다.
  - **적재 단계의 난제**: 목적지 데이터베이스 엔진에 설정된 외래 키(FK) 제약 조건, 고유성(Unique) 제약 조건, 또는 스키마 레벨의 유효성 검사(Validation) 실패가 발생하여 전체 적재 파이프라인이 중단될 수 있다.

### 1.3. 데이터 동기화(Data Synchronization)

- 기존 레거시 시스템에서 신규 시스템으로 아키텍처를 점진적으로 전환하는 과도기에는, 서비스 중단 없이 여러 이기종 데이터 저장소를 실시간으로 동기화해야 하는 요구사항이 발생한다.
- 이때 시스템 간의 데이터 정합성을 유지하기 위해 다음과 같은 검증된 엔지니어링 시나리오와 솔루션을 도입할 수 있다.
  - **메시지 큐(Message Queue) 활용**: 카프카(Kafka) 나 래빗MQ(RabbitMQ) 같은 메시징 인프라를 구축하여, 발생한 데이터 변경 이벤트를 비동기적(Asynchronous)으로 전파하고 소비함으로써 데이터 저장소 간의 결합도를 낮추고 동기화 흐름을 보장한다.
  - **변경 데이터 포착(CDC, Change Data Capture)**: 데이터베이스 엔진의 로우 레벨 트랜잭션 로그(예: MySQL의 Binlog 등)를 실시간으로 추적·감지하여 데이터 변경 사항을 추출하는 **CDC** 도구(Debezium 등)를 도입한다. 애플리케이션 레이어에 부하를 주지 않고도 소스 DB와 타겟 DB를 초고속으로 동기화할 수 있다.
  - **데이터 조정(Reconciliation) 배치 프로세스**: 두 시스템 간의 데이터 유실이나 불일치를 원천 차단하기 위해, 주기적으로 양측 저장소의 스냅샷 데이터를 상호 비교 검증하고 유실된 데이터를 보정 및 동기화해 주는 별도의 **조정(Reconciliation)** 배치 프로그램을 스케줄링하여 운영한다.

## 2. 스키마 변경 처리(Schema Migration)

- 비즈니스가 성장하고 기능이 고도화됨에 따라 소프트웨어는 끊임없이 발전하며, 이에 맞춰 프로덕션 데이터베이스의 기본 스키마 구조도 불가피하게 변경(DDL 실행)해야 한다.
- 레거시 데이터의 영속성과 무결성을 완벽히 보존하면서 이러한 스키마 변경 과정을 안전하게 제어하는 능력은 백엔드 엔지니어의 핵심 역량이다.
- 이를 달성하기 위해서는 데이터베이스 스키마의 형상 관리 개념을 도입하고, **플라이웨이(Flyway)** 같은 자동화 마이그레이션 도구를 활용해야 하며, 무중단 배포를 위한 하위 호환성 유지 기술이 요구된다.

### 2.1. 데이터 구조에 버전 관리(Version Control) 사용하기

- 데이터베이스 구조에 형상 관리를 적용하지 않으면, 과거 스키마 아키텍처의 변경 이력을 유실하고, 멀티 워커 환경에서 동료 개발자가 수동으로 수행한 DDL 작업 내용을 덮어써 버리거나, 프로덕션 배포 중 장애가 발생했을 때 이전 상태로 복구할 수 있는 안정적인 수단을 잃게 되는 치명적인 위험에 직면한다.
- 자바 생태계의 **플라이웨이(Flyway)**, 리퀴베이스(Liquibase), 혹은 루비 온 레일즈 마이그레이션(Rails Migration) 같은 전문 도구는 데이터베이스 스키마의 버전 관리 프레임워크를 제공한다.
- 이러한 도구들은 전용 메타데이터 테이블을 통해 어떤 마이그레이션 스크립트 파일이 현재 인프라 환경에 반영되었는지 고유 번호(Version)로 정확히 추적함으로써 개발(Local), 스테이징(QA), 운영(Prod) 등 전반적인 인프라 환경의 스키마 일관성을 완벽히 보장한다.
- 데이터베이스 스키마 버전 관리 아키텍처를 운용할 때 준수해야 할 필수 지침은 다음과 같다.
  - **멱등성(Idempotency) 확보**: 작성하는 모든 마이그레이션 DDL/DML 스크립트는 어떤 환경에서 몇 번을 반복해서 실행하더라도 항상 동일한 결과 상태를 도출하도록 **멱등성**을 갖추어야 한다. (예: `CREATE TABLE IF NOT EXISTS` 구문 활용 등)
  - **하위 호환성(Backward Compatibility) 유지**: 운영 중인 웹 서비스의 무중단 배포(Rolling Deployment 등)를 지원할 수 있도록, 새로운 컬럼 추가 시 기본값(Default Value)을 명시하거나 구버전 애플리케이션 코드가 실행 중인 상태에서도 데이터베이스 접근에 실패하지 않도록 이전 버전 스키마와의 호환성을 철저히 보장해야 한다.
  - **단일 원자적 스크립트 구성**: 특정 기능 변경에 필요한 데이터베이스 아키텍처 구조의 변경(Schema DDL)과 레거시 데이터의 정제 및 이관 작업(Data DML)은 반드시 동일한 마이그레이션 버전 스크립트 내에 묶어서 점진적으로 배포해야 한다.

### 2.2. 데이터 변환 관리

- 스키마 변경 작업 시 테이블 구조 변경뿐만 아니라 기존 레거시 데이터의 포맷 변환(예: 문자열 컬럼을 파싱하여 JSON 형식 데이터로 재구축 등)이 수반되는 경우, 이 변환 로직 역시 스키마 변경 파일과 함께 철저히 버전 관리 시스템에 편입시켜야 한다.
- 이렇게 설계해야만 모든 격리된 인프라 환경에서 데이터베이스 구조와 실제 적재된 데이터 포맷이 한 치의 오차도 없이 동기화된 상태를 유지할 수 있다.
- 예를 들어 플라이웨이(Flyway) 프레임워크의 경우, 단순한 순수 SQL 스크립트 작성 방식 외에도 프로그래밍 언어의 런타임 능력을 활용할 수 있는 **자바 기반 마이그레이션(Java-based Migration)** 기능을 강력하게 지원한다.
- 이처럼 자바 코드로 마이그레이션을 구현하는 방식을 채택하면, 애플리케이션 내부의 도메인 엔티티 구조나 암호화 라이브러리, 유틸리티 로직을 그대로 활용할 수 있고, 복잡한 문자열 구문 분석(Parsing) 처리가 용이해지며, 마이그레이션 진행 도중 SQL 만으로는 구현하기 까다로운 비즈니스 명세 수준의 고도화된 데이터 유효성 검증(Validation)을 실행할 수 있어 매우 유연하다.

#### 스키마와 데이터 변환을 함께 버전 관리할 때의 장점

- 아키텍처 구조의 변경과 데이터 포맷 변환 작업이 일관되게 단일 버전 주기로 관리되므로 소스 코드와의 결합 정합성이 유지된다.
- 전체 데이터베이스 마이그레이션이 **원자성(Atomicity)** 을 확보한다. 즉, 테이블 구조 변경과 내포된 데이터 가공 연산이 모두 완벽히 성공하거나, 중간에 실패할 경우 전체 작업이 통째로 롤백(Rollback)되어 유효한 이전 상태를 보존한다.
- 플라이웨이의 마이그레이션 이력 테이블(`flyway_schema_history`)을 조회하여 복잡한 데이터 변환이 정확히 어느 시점에 어떤 환경까지 안정적으로 반영되었는지 인프라 수준에서 투명하게 추적할 수 있다.

#### 대규모 데이터셋 기반의 복잡한 변환 처리 시 보완 아안

- **비동기 이면 작업(Background Job) 실행**: 마이그레이션 대상 데이터 레코드가 수천만 건 이상으로 너무 방대하여 서비스 점검 시간 내에 완료가 불가능한 경우, 동기식 DDL 배포를 최소화하고 백엔드 영역에서 **스프링 배치(Spring Batch)** 같은 별도의 비동기 배치 프로세스를 구동하여 프로덕션 시스템의 IOPS 성능에 영향을 주지 않도록 이면에서 점진적으로 변환 작업을 수행해야 한다.
- **이중 쓰기(Double Write) 아키텍처 구현**: 레거시 스키마 데이터 저장소와 신규 아키텍처 데이터 저장소를 동시에 운영하는 전환 기간 동안, 애플리케이션 레이어에서 양쪽 시스템에 데이터를 실시간으로 동시 적재하는 **이중 쓰기(Double Write)** 로직을 임시 구현한다. 이후 백그라운드 데이터 마이그레이션이 최종 완료되면 레거시 저장소로의 쓰기 경로를 안전하게 차단(Deprecation)하는 방식을 취한다.
- **철저한 원상 복구(Rollback) 계획 수립**: 복잡한 데이터 인코딩 및 변환 도중 예기치 못한 비즈니스 예외나 데이터 손상이 관찰될 경우를 대비하여, 마이그레이션 실행 직전 물리적인 데이터베이스 스냅샷(Snapshot) 백업을 선행하거나 변경 사항을 즉시 원복할 수 있는 하위 마이그레이션 스크립트(Undo Migration)를 완벽히 자산화해 두어야 한다.
