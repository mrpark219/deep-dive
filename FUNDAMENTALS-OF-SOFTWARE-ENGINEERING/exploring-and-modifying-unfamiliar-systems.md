# 낯선 시스템에 빨리 친숙해지기

## 1. 낯선 시스템에 빨리 친숙해지기

- 많은 개발자가 아무것도 없는 백지에서 새로 시작하는 그린필드(Greenfield) 프로젝트를 선호하지만, 소프트웨어 개발의 냉혹한 현실은 이미 구축된 브라운필드(Brownfield) 시스템에서 대부분의 업무가 이루어진다는 점이다.
- 신규 프로젝트에 새로 합류했거나 새로운 코드베이스를 탐색해야 하는 상황이라면, 잘 모르는 레거시 시스템을 빠르게 파악하고 안전하게 수정하는 기술을 반드시 키워야 한다.

## 2. 낯선 코드 이해

### 2.1. 출발점, 큰 그림

- 프레임워크의 상세 버전이나 라이브러리 설정 같은 세부 사항에만 함몰되면, 프로젝트의 궁극적인 비즈니스 목표나 시스템의 전반적인 맥락 같은 더 큰 그림을 놓치게 된다.

#### 프로젝트 이해

- 코드를 깊이 파악하기 전에 먼저 프로젝트의 존재 목적과 비즈니스 의도를 이해해야 한다.
- 상위 수준(High-Level)에서 시작해야 한다. 이 프로젝트가 존재하는 이유는 무엇인가? 주요 이해관계자(Stakeholder)는 누구인가? 회사나 사업에서 이 프로젝트는 어떤 의미와 비중을 가지는가?
- 이를 파악하기 위해서는 제품에 대해 깊이 아는 **제품 관리자(PM, Product Manager)** 나 기획자를 먼저 찾아가는 것이 좋다. 이들과의 미팅을 통해 도메인 지식을 배우는 것이 효과적이며, 화상 회의를 진행한다면 나중에 복습할 수 있도록 녹화를 요청하는 것도 좋은 방법이다.
- 그 다음에는 프로젝트를 담당하고 있는 동료 소프트웨어 엔지니어에게 연락하여, 현재의 아키텍처나 설계가 결정된 당시의 배경과 통찰을 얻어야 한다.
- 이 제품의 실제 고객이 누구인지 정의된 자료나 공개 정보를 찾아봐야 한다. 철저히 고객의 관점에서 시스템을 바라보려고 노력해야 하며, 조직에 따라 가상의 사용자 프로필인 **사용자 페르소나(User Persona)** 를 활용하는 것도 도움이 된다.

#### 기존 문서 검토

- 이해관계자들과 이야기를 나눈 후에는 **아키텍처 결정 기록(ADR, Architecture Decision Record)** 을 포함하여 사내 위키나 저장소에 남겨진 읽을 수 있는 모든 문서를 샅샅이 살펴봐야 한다.
- 프로젝트의 큰 그림을 머릿속에 그린 후에 이러한 문서들을 읽어야 비로소 맥락이 의미 있게 다가온다.
- 이 단계에서 읽는 가장 대표적인 자료는 **온보딩 문서(Onboarding Document)** 다. 여기에는 로컬 개발 환경에서 프로젝트를 실행하고 빌드하는 데 필요한 모든 정보가 담겨 있다. 최근에는 **도커(Docker)** 덕분에 환경 구성이 단순해져 온보딩 문서의 품질이 전보다 나아졌지만, 여전히 유실되거나 누락된 정보가 많아 까다로울 수 있다.
- 문서의 함정을 항상 경계해야 한다. 과거에 작성된 문서와 현재 실제 동작하는 코드는 다를 수 있기 때문에, 문서와 코드를 교차 검증하며 차이점을 발견해야 한다. 막히는 부분이 생기면 동료에게 바로 질문하여 불필요한 시간 낭비를 줄여야 한다.

#### 아키텍처와 프로젝트 구조 이해

- 전반적인 프로젝트 배경을 파악하고 기존 문서를 읽었다면, 이제 전체 코드의 아키텍처 패러다임과 폴더 구조를 살펴봐야 한다.
- 이는 코드의 세부 구현이나 설정 파일을 자세히 분석하기 전에 거쳐야 할 매우 중요한 단계다.
- 프로젝트의 디렉터리 구조는 생각보다 많은 정보를 드러내며, 코드를 탐색할 때 길을 잃지 않도록 돕는 전체적인 시야를 제공한다.
- 대부분의 백엔드 시스템은 아래의 아키텍처 패턴 중 하나를 따른다.
  - **계층별 패키지(Package by Layer)**:
    - 기술적인 역할과 책임에 따라 코드를 수평적으로 배치하는 방식이다.
    - 주로 `controllers`, `services`, `repositories`, `models` 같은 직관적인 최상위 디렉터리 구조를 가지게 된다.
  - **기능별 패키지(Package by Feature)**:
    - 비즈니스 지원 능력이나 도메인 기능을 중심으로 코드를 수직적으로 배치하는 방식이다.
    - `user`, `products`, `orders` 같이 비즈니스 도메인을 나타내는 최상위 디렉터리를 보게 된다.
    - 이 방식은 특정 기능과 관련된 모든 기술 요소를 함께 캡슐화하므로, 하나의 비즈니스 흐름을 이해하기 쉽게 해준다.
  - **헥사고날 아키텍처(Hexagonal Architecture / Ports and Adapters)**:
    - 핵심 비즈니스 로직(도메인)을 프레임워크이나 데이터베이스 같은 외부 관심사와 철저히 분리하도록 코드를 구성하는 방식이다.
    - 내부의 핵심 도메인 모델과 인터페이스(포트)를 외부 세계와 연결하는 어댑터(Adapter) 코드가 감싸고 있는 형태를 띤다.
    - 이 패턴은 기술적인 구현 세부 사항보다 비즈니스 규칙의 순수한 격리를 강조한다.
  - **마이크로서비스 아키텍처(MSA, Microservice Architecture)**:
    - 전체 시스템의 기능을 독립적으로 배포 가능한 여러 개의 작은 서비스 단위로 분리하는 방식이다.
    - 마이크로서비스 환경에서 작업할 때는 개별 서비스 내부의 아키텍처뿐만 아니라, 이 서비스들이 네트워크를 통해 전체 생태계에 어떻게 녹아들고 상호작용하는지 메시지 브로커나 API 게이트웨이 등의 인프라 구조까지 함께 이해해야 한다.

### 2.2. 실행 흐름 파악

- **실행 흐름(Execution Flow)** 은 프로그램이 실행되는 동안 따르는 명령의 순차적인 경로다.
- 여기에는 어떤 코드가 어떤 순서로 실행될지 결정하는 모든 조건문, 반복문, 함수 호출이 포함된다.
- 이 단계에서는 코드를 직접 수정하거나 지나치게 깊이 분석하지 않고, 전반적인 코드의 흐름을 이해하는 데 집중한다.

#### 애플리케이션 진입점(Entry Point) 찾기

- 애플리케이션에는 진입점 역할을 하는 여러 개의 관문이 존재한다.
- 이러한 관문은 애플리케이션의 `main` 메서드, 공개 API, 웹 UI 등의 형태로 존재한다.
- 진입점을 찾은 후에는 코드의 실행 흐름을 단계별로 따라가면 된다.
- 이때 **IDE의 디버거(Debugger)** 를 활용하면 프레임워크 내부에서 어떤 작업이 일어나는지 통찰을 얻을 수 있으며, 전체 코드를 파악하는 훌륭한 시작점 역할을 해준다.
- 시스템의 진입점 유형을 더 자세히 분류하면 다음과 같다.
  - **메인/부트스트랩 메서드(Main/Bootstrap Method)**: 애플리케이션 실행이 최초로 시작되는 전통적인 지점이다.
  - **공개 API/컨트롤러(Public API/Controller)**: 외부 시스템에 기능을 노출하는 최전방 엔드포인트다.
  - **이벤트 핸들러/리스너(Event Handler/Listener)**: 시스템 내부외의 특정 이벤트나 트리거에 반응하여 실행되는 코드다.
  - **스케줄러/배치 잡(Scheduler/Batch Job)**: 크론탭(Crontab)처럼 미리 정해진 시간 간격에 따라 주기적으로 실행되는 기능이다.
  - **수명 주기 훅(Lifecycle Hook)**: 컴포넌트의 생성, 시작, 종료 시점에 자동으로 호출되는 메서드다.
  - **플러그인/확장 포인트(Plugin/Extension Point)**: 애플리케이션 기능을 유연하게 확장하기 위해 설계된 인터페이스다.
  - **메시지 컨슈머(Message Consumer)**: 카프카(Kafka) 나 래빗MQ(RabbitMQ) 같은 메시지 브로커의 큐에 담긴 메시지에 응답하는 로직이다.
  - **명령행 인자 처리기(CLI Argument Processor)**: 프로그램 시작 시 전달되는 파라미터를 처리하는 로직이다.
  - **데이터베이스 트리거/저장 프로시저(DB Trigger/Stored Procedure)**: 데이터의 변경에 반응하여 데이터베이스 관리 시스템(DBMS) 서버 내부에서 실행되는 코드다.

#### 데이터에 따른 요청 여정 추적

- 기존 코드베이스, 특히 웹 애플리케이션이나 API 환경에서 작업할 때 시스템을 관통하는 **요청의 여정(Request Journey)** 을 추적하는 것은 코드의 실제 작동 방식과 연동 시스템을 이해하는 가장 확실한 방법이다.
- 요청 여정 추적은 다음과 같은 장점을 가진다.
  - 여러 컴포넌트와 외부 시스템을 거치는 실제 런타임 실행 경로를 명확히 드러낸다.
  - 데이터 처리에 관여하는 아키텍처의 모든 레이어(Layer)를 식별하도록 도와준다.
  - 파이프라인 진행 과정에서 발생하는 데이터의 변환 및 정제 과정을 노출한다.
  - 코드 속에 숨겨진 비즈니스 로직과 복잡한 유효성 검사(Validation) 규칙을 발견하게 해준다.
  - 예외 상황 발생 시 실제로 에러가 어떻게 핸들링되고 전파되는지 보여준다.
- 이때 실행 흐름을 추적하기 위해 활용할 수 있는 도구와 방법은 다음과 같다.
  - **브라우저 개발자 도구(Browser DevTools)**: 네트워크 탭을 통해 프론트엔드와 백엔드 간에 오가는 HTTP 요청과 응답을 직접 확인한다.
  - **API 테스트 도구(Postman / Insomnia)**: API를 직접 호출해 보며 필수 필드와 자료형, 입력값 유효성 검사 동작, 인증 및 권한 부여, 응답 구조 및 상태 코드, 에러 메시지의 품질, 성능 특성 등을 파악한다.
  - **애플리케이션 로그(Application Log)**: 시스템이 출력하는 로그 콘솔을 통해 컴포넌트의 내부 동작을 파악하고 이슈를 디버깅한다.
  - **런타임 디버깅(Runtime Debugging)**: IDE의 디버그 모드를 활성화하여 라인 단위로 코드를 실행해 본다. 각 조건문이 어떻게 평가되는지, 런타임 객체의 상태와 변수값은 무엇인지, 메서드 호출 결과가 어떻게 반환되는지 생생하게 추적할 수 있다.

#### 외부 의존 관계 파악

- 애플리케이션이 단독으로 실행되는 경우는 드물기 때문에, 실행 흐름을 이해하는 과정에서 애플리케이션이 통신하는 다른 API, 데이터베이스(DB), 캐시(Cache) 등의 외부 의존 시스템을 모두 파악하는 것은 매우 중요하다.
- 이 단계에서는 인프라 및 환경 설정 구성 파일을 통해 파악하는 것이 효과적이다. 대표적으로 `docker-compose.yml`, 쿠버네티스 매니페스트(Kubernetes Manifest) 파일 등을 살펴보면 전체 시스템의 유기적인 연결 구조를 쉽게 파악할 수 있다.

#### 내부 프레임워크와 라이브러리 파악

- 자체 제작한 사내 프레임워크나 라이브러리는 기존 코드를 탐색할 때 마주하는 가장 큰 진입 장벽 중 하나다.
- 이는 특정 비즈니스 요구나 기술적 문제를 해결하기 위해 사내에서 직접 개발한 솔루션이다.
- 최초 제작 당시에는 최선의 결정이었을 수 있으나, 코드를 새로 탐색하고 수정해야 하는 신규 팀원에게는 커다란 장애물이 되기도 한다.
- 일반적으로 외부 라이브러리에서 제공하지 않는 특수한 기능이 필요했거나, 특별한 보안 및 성능 요구사항이 있을 때, 혹은 외부 솔루션의 도입 비용이 너무 비싸 이를 대체하고자 도입된다. 때로는 당시 시장에 적절한 솔루션이 존재함을 인지하지 못해 수동으로 구현한 바퀴가 그대로 굳어진 경우도 많다.

### 2.3. 점진적인 정신 모델(Mental Model) 구축

- **정신 모델(Mental Model)** 이란 시스템이 작동하는 방식에 대해 개발자의 머릿속에 그려지는 추상적인 그림이다. 탄탄한 정신 모델을 구축하려면 시간과 의도적인 연습이 필요하다.
- 복잡한 레거시 시스템의 정신 모델을 효과적으로 형성하려면 시스템을 한 번에 소화하려 하지 말고, 단계를 나누어 점진적으로 분해해 나가야 한다.
- 복잡한 시스템을 작게 쪼개고, 결합 관계를 시각화하며, 도메인 지식을 점진적으로 확장하는 접근 방식이 권장된다.

#### 복잡한 시스템 분해하기

- 시스템의 모든 컴포넌트와 비즈니스 로직을 한꺼번에 이해하려고 시도하는 것은 인지 과부하를 일으키며, 매우 비효율적이다.

#### 정신 모델 시각화

- 코드와 컴포넌트 간의 관계를 시각 자료로 표현하면 정신 모델을 구축하는 데 큰 도움이 된다.
- 순서도(Flowchart), **개체-관계 다이어그램(ERD, Entity-Relationship Diagram)**, **컴포넌트 다이어그램(Component Diagram)**, 마인드맵(Mind Map) 등을 적극적으로 활용하여 시스템을 구조화해야 한다.

#### 점진적 범위 확장

- 탐색 범위를 넓혀갈 때는 다음과 같은 단계적 접근법을 취한다.
  1. 격리된 단일 메서드의 동작을 분석하는 것부터 시작한다.
  2. 해당 메서드에서 사용하는 클래스와, 그 클래스가 직접 의존하고 있는 구성 요소로 시야를 넓힌다.
  3. 여러 구성 요소와 레이어에 걸쳐 있는 전반적인 비즈니스 기능 흐름을 따라가며 연쇄적으로 파악한다.
  4. 마지막으로 시스템 전반에 녹아 있는 보안(Authentication/Authorization)이나 트랜잭션(Transaction) 처리와 같은 **횡단 관심사(AOP, Cross-cutting Concerns)** 를 이해한다.

## 3. 안전한 수정

### 3.1. 안전한 리팩터링(Refactoring)

#### 기존 테스트에 의존하라

- 테스트 코드는 시스템의 비즈니스 명세서 역할을 하여 코드를 이해하는 데 도움을 줄 뿐만 아니라, 코드를 안전하고 과감하게 리팩터링할 수 있도록 돕는 가장 강력한 도구다.
- 테스트는 리팩터링 과정에서 든든한 **안전망(Safety Net)** 역할을 하므로, 코드를 수정한 후 테스트를 실행해 봄으로써 프로젝트의 다른 도메인 영역에 예상치 못한 사이드 이펙트(Side Effect)를 미치지 않았는지 즉각 검증할 수 있다.

#### 리팩터링 전에 테스트를 추가하라

- 만약 변경하려는 코드 영역이 기존 테스트의 커버리지 범위 밖이라면, 기존 코드를 수정하기 전에 결함이 없음을 증명할 테스트부터 보완해야 한다.
- 이를 **테스트 주도 리팩터링(Test-Driven Refactoring)** 이라고 부르며, 다음과 같은 절차로 진행된다.
  1. 현재 코드가 수행하고 있는 실제 동작을 문서화하는 테스트 코드를 작성한다.
  2. 현재 구현체가 작성한 테스트를 안정적으로 통과(Green)하는지 확인한다.
  3. 외부 동작을 변경하지 않으면서 내부 코드를 깨끗하게 리팩터링한다.
  4. 새롭게 개선된 구현체가 여전히 동일한 테스트를 통과하는지 최종 검증한다.

### 3.2. 보이스카우트 규칙(Boy Scout Rule)

- "코드를 처음 발견했을 때보다 항상 더 깨끗하게 남겨두라"는 **보이스카우트 규칙(Boy Scout Rule)** 은 소프트웨어 장인 정신의 핵심 원칙이다.
- 이 규칙의 실천 방안으로는 누락된 문서나 주석 보완, 모호한 변수 및 메서드 이름 개선, 지나치게 비대하고 복잡한 메서드 분해, 사용되지 않는 데드 코드(Dead Code) 제거, 발견한 잠재적 버그 수정 등이 있다.
- 다만, 리팩터링에 너무 몰입한 나머지 원래 작업하려던 본래의 스프린트 기능 구현이나 이슈 해결 임무를 완수하지 못하는 상황이 발생하지 않도록 일정과 범위를 조율해야 한다.

- **리팩터링을 즉시 고려해야 하는 경우**:
  - 현재 작업 중인 코드가 너무 난해하여 도저히 이해하기 어려울 때
  - 해당 영역에 조만간 새로운 기능을 추가해야 하는 요구사항이 있을 때
  - 코드에 내재된 특정 버그를 수정하는 중일 때
  - 해당 코드가 시스템 전체의 성능 병목을 일으키고 있을 때
  - 여러 개발자가 이 코드 영역을 자주 변경하고 있어 충돌이 잦을 때

- **리팩터링을 뒤로 미루어야 하는 경우**:
  - 코드가 안정적으로 잘 작동하며, 향후 비즈니스적으로 수정될 가능성이 거의 없을 때
  - 프로젝트 배포 마감 기한의 압박이 극심할 때
  - 코드를 잘못 건드려 발생할 위험 부담이 리팩터링으로 얻을 이득보다 클 때
  - 리팩터링을 수행하기 위해 현재 할당된 작업 범위를 한참 벗어나는 광범위한 코드 아키텍처 변경이 필요할 때
  - 변경 후의 안전성을 보장할 만한 적절한 테스트 코드가 구축되어 있지 않을 때

### 3.3. 되돌릴 수 있는 작은 변경

- 장애 발생 시 언제든 쉽게 롤백(Rollback)할 수 있는 작은 단위의 변경을 점진적으로 적용하는 것이, 전체 시스템에 충격을 주지 않고 기존 레거시 코드를 수정하는 가장 안전한 방법이다.

#### 변경 관리 전략

- **변경 관리 전략(Change Management Strategy)** 은 소프트웨어 팀이 애플리케이션의 수정 및 배포 과정을 통제하기 위해 사용하는 구조화된 접근 방식이다.
- 기본적으로 코드의 품질과 팀의 얼라인먼트를 유지하면서 수정 사항을 제안, 검토, 구현, 추적하는 시스템을 의미하며, 다음과 같은 수칙을 준수해야 한다.
  - 변경 사항을 투명하게 가시화하여 팀 전체에 공유해야 한다.
  - 자동화된 테스트 및 코드 리뷰 등의 검증 체계를 견고히 구축해야 한다.
  - 에러 발생 시 즉각 대처할 수 있는 확실한 원상복구(Rollback) 계획을 수립해야 한다.
  - 배포 후 시스템의 성능 및 동작 효과를 모니터링 시스템(APM 등)을 통해 면밀히 주시해야 한다.

#### 테스트 가능한 작은 증분(Increment)

- 테스트 가능한 작은 증분(Increment) 단위로 작업을 쪼개어 진행하는 사고방식을 가지면, 거대한 레거시 코드베이스도 두려움 없이 수정할 수 있다.
  - 각 변경 사항의 범위를 최소한으로 제한함으로써 장애 위험을 획기적으로 줄여준다.
  - 테스트 코드가 다루어야 할 범위가 좁아지므로 테스트의 집중도와 효과가 높아진다.
  - 수정 사항이 명확하여 동료들의 **코드 리뷰(Code Review)** 부담이 줄어들고 수월해진다.
  - 예기치 못한 문제가 발생했을 때 원인이 되는 코드 줄을 찾기가 훨씬 쉬워진다.
  - 한 번에 거대한 빅뱅(Big Bang)식 변경을 적용하는 것보다, 작고 성공적인 변경을 자주 릴리스하는 것이 훨씬 안전하다.

#### 버전 관리 권장 사용법(Git Best Practices)

- 하나의 커밋(Commit) 단위는 그 자체로 독립적이며, 논리적으로 단일한 변경 사항만을 의미해야 한다.
- 커밋의 목적과 변경 이유를 명확히 알 수 있도록 의미 있는 **커밋 메시지(Commit Message)** 를 작성해야 한다.
- 작업 내용을 잃어버리지 않고 추적할 수 있도록 자주 커밋하는 습관을 들여야 한다.
- 기능 개발 시 반드시 독립적인 피처 브랜치(Feature Branch)를 생성하여 작업하되, 코드 동기화를 위해 각 브랜치의 수명은 짧게 유지해야 한다.
- 메인 브랜치에 병합하기 전에 최신 변경 사항을 **리베이스(Rebase)** 하여 커밋 히스토리를 깔끔하게 정리해야 한다.
- **풀 리퀘스트(PR, Pull Request)** 를 보낼 때는 단 하나의 명확한 문제나 기능에만 집중하는 작은 단위의 PR을 지향해야 한다.
