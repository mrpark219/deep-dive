# 다양한 데이터 기술을 적절하게 사용하는 방법

## 1. 데이터(Data)

- 소프트웨어 엔지니어로 경력을 시작할 때 프로그래밍 언어, 프레임워크, 개발 도구를 배우는 데만 매몰되기 쉽다.
- 하지만 실제 엔지니어링 환경에서는 다양한 형태의 데이터를 다루는 데 대부분의 시간을 보내게 되며, 데이터는 우리가 구현하는 모든 비즈니스 로직의 근간이다.
- 애플리케이션은 데이터를 수집하고, 저장하고, 변환하고, 최종적으로 사용자에게 의미 있는 방식으로 되돌려주는 생명 주기를 중심으로 모델링된다.
- 시스템 개발에서 데이터를 다룬다는 것은 원시 자료형(Primitive Type)의 이해부터 시작하여 적절한 저장소(Storage) 선택, 효율적인 데이터 모델 설계, 조회(Query) 최적화, 데이터 무결성(Integrity) 보장, 시간에 따른 스키마 진화 관리 등 훨씬 더 넓고 깊은 엔지니어링 영역을 포함한다.

## 2. 자료형과 형식의 이해

- 아키텍처의 규모와 상관없이 프로젝트의 성공은 도메인 데이터를 어떻게 조직하고, 검증하며, 효율적으로 처리하느냐에 달려 있다.
- 시스템 전반의 자료형과 데이터 형식을 설계할 때는 반드시 아래의 요소를 종합적으로 고려해야 한다.
  - **소비 대상**: 누가 이 데이터를 읽거나 처리하는가? 사람인가, 아니면 시스템(기계)인가?
  - **성능 요구사항**: 데이터의 크기(Payload Size)나 직렬화/역직렬화 처리 속도가 얼마나 중요한가?
  - **호환성**: 어떤 이기종 시스템이 이 데이터를 소비하고 연동하는가?
  - **복잡성**: 데이터의 구조가 얼마나 깊게 중첩되거나 가변적인가?
  - **검증 필요성**: 엄격한 스키마 유효성 검증(Schema Validation)이 필요한가?

### 2.1. 정형 데이터와 비정형 데이터

#### 정형 데이터(Structured Data)

- **정형 데이터(Structured Data)** 는 스프레드시트나 관계형 테이블처럼 명확하게 구조화되어 카테고리와 연관 관계가 선언적인 데이터를 말하며, 데이터베이스 저장과 검색이 매우 용이하다.
- 모든 데이터가 일관된 필드 타입과 관계를 유지하며, 미리 정의된 데이터 모델이나 엄격한 **스키마(Schema)** 에 따라 정돈된다.
- 백엔드 애플리케이션 코드에서는 주로 엔티티(Entity)나 데이터 모델을 정의하는 클래스를 통해 이 구조를 표현한다.
- 정형 데이터는 다음과 같은 명확한 특징을 지닌다.
  - **일관된 형식**: 모든 레코드가 사전에 약속된 동일한 스키마를 엄격히 따른다.
  - **명확한 관계**: 외래 키(FK) 등을 통해 데이터 개체(Entity) 간의 연결 고리가 명확하다.
  - **용이한 조회**: 인덱스(Index)를 활용한 정밀한 검색, 필터링, 정렬 작업을 강력하게 지원한다.
  - **효율적인 저장**: 고정폭 필드를 사용하는 관계형 데이터베이스(RDB) 환경에 최적화되어 있다.
- 정형 데이터는 엄격한 데이터 무결성, 복잡한 조인(Join) 질의문, 확립된 비즈니스 규칙이 필수적인 금융 거래(트랜잭션), 재고 관리, 사용자 프로필 도메인 등에 주로 사용된다.
- 한계점으로는, 새로운 필드를 추가하거나 변경하려면 데이터베이스 마이그레이션(DDL 실행)과 애플리케이션 코드를 동시에 수정해야 하므로 상대적으로 유연성이 떨어진다.
- 또한 계층 구조가 복잡하거나 동적으로 중첩된 정보를 효율적으로 표현하는 데는 구조적인 어려움을 겪기도 한다.

#### 비정형 데이터(Unstructured Data)

- **비정형 데이터(Unstructured Data)** 는 일반 텍스트, 사진, 오디오, 비디오, 비구조화된 문서 등을 포함하며, 사전에 정의된 데이터 모델이나 구조가 없어 행과 열에 딱 들어맞지 않는 데이터를 뜻한다.
- 이메일 본문, 소셜 미디어(SNS) 게시글, 채팅 기록 같은 자유로운 형식의 텍스트 데이터가 대표적이다.
- 가공되지 않은 raw 데이터 특성을 가지기 때문에, 비정형 데이터를 다룰 때는 정형 데이터와는 완전히 다른 기술 스택으로 접근해야 한다.
- 예를 들어 텍스트 분석을 위해서는 **자연어 처리(NLP, Natural Language Processing)** 기술이 필요하며, 이미지 및 오디오 레이어를 다루기 위해서는 **컴퓨터 비전(Computer Vision)** 이나 딥러닝 인프라가 요구된다.
- 비정형 데이터는 다음과 같은 특징을 지닌다.
  - **가변적인 형식**: 일관된 스키마나 고정된 구조가 존재하지 않는다.
  - **풍부한 콘텐츠**: 컨텍스트 속의 맥락, 감정, 의견, 시각적 세부 사항 등 미묘하고 방대한 정보를 포함한다.
  - **조회의 어려움**: 단순 SQL 질의로는 검색이 불가능하며, **전문 검색(Full-Text Search)** 이나 벡터 검색(Vector Search) 같은 특수한 기법이 필요하다.
  - **저장의 어려움**: 대용량 바이너리 객체(BLOB)에 최적화된 대규모 스토리지 시스템이 요구된다.
- 비정형 데이터를 안정적으로 적재하고 활용하기 위해서는 전통적인 관계형 데이터베이스가 아니라, **엘라스틱서치(Elasticsearch)** 같은 전문 검색 엔진, NoSQL 데이터베이스, 또는 대용량 원천 데이터를 유연하게 보관하기 위해 설계된 **데이터 레이크(Data Lake)** 나 오브젝트 스토리지(AWS S3 등)를 사용해야 한다.

### 2.2. 일반적인 데이터 형식

#### JSON(JavaScript Object Notation)

- **JSON(JavaScript Object Notation)** 은 가독성이 높고 구조가 단순하여 현대 소프트웨어 생태계에서 데이터 교환의 사실상 표준으로 자리 잡았다.
- 특정 프로그래밍 언어에 종속되지 않으며 객체(Map), 배열(List), 문자열, 숫자, 불리언, 널(Null) 값을 직관적으로 지원한다.
- 웹 API 통신 파이프라인(REST API), 시스템 설정 파일, **문서 지향 NoSQL 데이터베이스(Document DB)** 에 아주 훌륭한 선택지다.
- 다만 자체적인 주석(Comment) 스펙을 지원하지 않고, 독립적인 날짜/시간(Date) 형식이 따로 없으며, 베이스64(Base64) 같은 별도의 인코딩 없이는 이진 데이터(Binary Data)를 효율적으로 다루기 어렵다는 한계가 있다.

#### XML(eXtensible Markup Language)

- **XML(eXtensible Markup Language)** 은 태그 기반의 계층 구조를 가지며, **XML 스키마 정의(XSD, XML Schema Definition)** 나 문서 형식 정의(DTD)를 통해 데이터 구조에 대한 매우 강력한 검증 방편을 제공한다.
- XSD를 활용하면 복잡한 데이터 자료형, 값의 제약 조건, 스키마의 구조적 규칙을 명확하게 명시할 수 있다.
- 또한 **엑스패스(XPath)** 표현식을 사용하여 방대한 데이터 구조 내에서 특정 요소를 정밀하게 탐색·선택할 수 있고, **XSLT**를 사용하면 XML 문서를 다른 데이터 형식으로 유연하게 변환할 수 있다.

#### CSV(Comma-Separated Values)

- **CSV(Comma-Separated Values)** 는 데이터 필드를 쉼표(,)로 구분하고 줄 바꿈(New Line)으로 레코드를 구분하여, 표 형태의 정형 데이터를 단순한 평문 텍스트(Plain Text)로 저장하는 형식이다.
- 대개 파일의 첫 번째 줄에는 데이터의 스키마를 암시하는 열 머릿글(Header)이 위치한다.
- 구조가 지극히 단순하여 파싱 비용이 적게 들지만, 내장된 자료형에 대한 정의가 없고 문자열 내에 포함된 쉼표나 큰따옴표 등의 특수 문자를 처리하는 일관된 이스케이프(Escape) 규칙이 부족하다는 단점이 있다.
- 주로 대용량 데이터 내보내기(Export) 및 가져오기(Import), 시스템 간의 단순한 데이터 교환, 그리고 **스프레드시트(Spreadsheet)** 애플리케이션과의 호환성을 확보할 때 가장 적합하다.

#### YAML(YAML Ain't Markup Language)

- **YAML(YAML Ain't Markup Language)** 은 JSON이나 XML에 비해 사람이 직관적으로 읽고 쓰기 가장 쉬운 구문을 제공하며, 문서 내 주석 처리, 객체 참조, 다중 행 텍스트(Multi-line String) 표현을 강력하게 지원한다.
- 괄호나 태그 대신 오직 **들여쓰기(Indentation)** 만으로 데이터의 계층 구조를 정의하므로 시각적 가독성이 매우 뛰어나다.
- 다만 공백(Space) 하나만 잘못 입력해도 구문 분석(Parsing) 오류가 발생할 수 있어 작성 시 주의가 필요하다.
- 인간 친화적인 특성 덕분에 애플리케이션 설정 파일, 특히 **도커 컴포즈(Docker Compose)** 설정, 쿠버네티스 매니페스트, CI/CD 파이프라인 정의 등 현대 **데브옵스(DevOps)** 인프라 도구의 표준 설정 형식으로 폭넓게 사용된다.

### 2.4. 기타 특수 사항

#### 이진 데이터(Binary Data)

- **이진 데이터(Binary Data)** 는 이미지, 컴파일된 문서, 오디오 및 비디오 파일, 그리고 암호화된 콘텐츠에 이르기까지 텍스트 형식을 제외한 모든 형태의 데이터를 포함한다.
- 일반적인 텍스트 기반 형식과 달리, 이진 데이터는 포맷에 맞는 적절한 인코딩 및 디코딩 파이프라인 없이는 인간이 직접 읽거나 프로그램으로 조작할 수 없다.
- 예를 들어 이미지 파일을 **바이트 배열(Byte Array)** 로 메모리에 읽어 들인 후에는 클라이언트 GUI 화면에 렌더링하기, 이미지 프로세싱을 통한 픽셀 조작, 외부 클라우드 오브젝트 스토리지에 업로드, HTTP 멀티파트(Multipart) 요청을 통한 전송, EXIF 같은 **메타데이터(Metadata)** 추출 등의 후속 처리를 수행할 수 있다.
- 이진 데이터를 안전하게 처리하기 위해서는 다음과 같은 엔지니어링 원칙을 준수해야 한다.
  - 특정 파일 포맷에 맞춰 고도로 최적화된 전용 검증 라이브러리를 사용해야 한다.
  - 이진 데이터를 JSON이나 XML 같은 텍스트 기반 전송 프로토콜에 포함해야 할 때는 데이터 손실을 막기 위해 **베이스64(Base64)** 인코딩을 활용해야 한다.
  - 분석 및 로우 레벨(Low-Level) 디버깅 단계에서 바이트 단위의 흐름을 직접 관찰해야 할 때는 **16진수(Hexadecimal)** 표현식을 사용해야 한다.
  - 대용량 이진 파일은 메모리 오버플로우(OOM)를 유발하기 쉬우므로 세심한 메모리 버퍼 관리가 필요하며, 파일 손상이나 네트워크 유실 가능성에 대비해 엄격한 예외 처리(Exception Handling) 구문을 반드시 구현해야 한다.

#### 날짜와 시간 데이터

- 날짜와 시간 데이터는 글로벌 서비스 환경에서 **시간대(Timezone)** 분기, **서머타임(DST, Daylight Saving Time)** 조정, 국가별 달력 시스템의 차이, 복잡한 포맷팅 변환 등으로 인해 수많은 런타임 버그를 유발하는 까다로운 영역이다.
- 비즈니스 도메인에서 날짜와 시간 데이터를 정확하게 다루기 위한 핵심 수칙은 다음과 같다.
  - 애플리케이션 서버 내부 및 데이터베이스 레이어에서는 모든 날짜와 시간 데이터를 항상 **협정 세계시(UTC, Coordinated Universal Time)** 기준으로 통합하여 저장하고, 최종 사용자 화면에 노출할 때만 클라이언트의 현지 시간대로 변환해야 한다.
  - 이기종 시스템 간의 통신 및 데이터 교환 파이프라인에서는 글로벌 표준인 **ISO 8601** 형식(`YYYY-MM-DDTHH:mm:ssZ`)을 철저히 준수해야 한다.
  - 날짜 계산 로직을 직접 수동으로 구현하지 말고, 자바의 `java.time` 패키지(JSR-310)와 같이 내부적으로 윤초와 시간대 변경이 검증된 최신 날짜/시간 표준 라이브러리를 활용해야 한다.
  - 다국적 사용자의 혼란을 방지하기 위해 사용자가 마주하는 인터페이스(UI) 상에 현재 적용 중인 시간대 기준(예: KST, EST 등)을 명확히 명시해야 한다.

#### 대규모 데이터셋(Large Dataset)

- 비즈니스가 성장하고 시스템 트래픽이 확장되면 애플리케이션이 감당해야 하는 데이터셋의 규모도 기하급수적으로 늘어난다.
- 과거 소규모 환경에서는 단순하게 작동하던 로직이, 이제는 메모리에 한꺼번에 올리기 조차 불가능한 대규모 데이터셋으로 발전하면 아키텍처 전반을 완전히 다른 방식으로 재설계해야 한다.
- 대규모 데이터셋을 효율적이고 안정적으로 다루기 위해서는 다음 요소를 아키텍처에 반영해야 한다.
  - 대용량 데이터를 한 번에 메모리에 로드하지 않고, 청크(Chunk) 단위로 점진적으로 읽어 들이는 **스트리밍(Streaming)** 프로그래밍 모델이나 배포 파이프라인을 구축해야 한다.
  - 대량의 목록을 반환하는 API 응답 및 프론트엔드 인터페이스에는 스크롤이나 버튼 기반의 **페이지네이션(Pagination)** 아키텍처를 필수로 구현해야 한다.
  - 읽기 성능 극대화를 위해 적절한 **데이터베이스 인덱스(Index)** 설정 및 쿼리 실행 계획(Execution Plan) 분석을 통한 SQL 튜닝을 상시 진행해야 한다.
  - 온라인 트랜잭션 처리(OLTP)의 한계를 넘어서는 가공 및 통계 데이터셋의 경우, **데이터 웨어하우스(Data Warehouse)** 나 분산 컴퓨팅 프레임워크(Apache Spark 등) 같은 빅데이터 전문 에코시스템 도입을 적극적으로 검토해야 한다.
  - 멀티스레드 환경에서 여러 워커가 대규모 데이터에 동시에 접근할 때는 데이터 무결성을 유지하고 레이스 컨디션(Race Condition)을 방지하기 위해, 적절한 데이터베이스 락(Lock), 원자적 연산(Atomic Operation), 또는 **불변 데이터 구조(Immutable Data Structure)** 를 활용한 동시성 제어(Concurrency Control)를 완벽히 구현해야 한다.
  - 시스템 오픈 후 예상치 못한 병목을 예방하기 위해, 개발 초기 단계부터 실제 프로덕션 운영 환경과 동일한 벤치마크 규모의 더미 데이터를 적재하여 성능 테스트를 선행해야 한다.
