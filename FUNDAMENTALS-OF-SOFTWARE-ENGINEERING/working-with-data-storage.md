# 효과적인 데이터 저장

- 데이터 구조를 정의했다면, 그다음 단계로는 그 데이터를 안전하고 효율적으로 담을 수 있는 최적의 저장 방식을 선택해야 한다.

## 1. 데이터베이스 유형과 사용 사례

- 저장 솔루션(Storage Solution)을 선택할 때는 시스템의 요구사항에 맞춰 다음과 같은 핵심 요소를 반드시 고려해야 한다.
  - **데이터 구조의 복잡성**: 데이터 엔티티 간의 연관 관계가 얼마나 복잡하게 얽혀 있는가?
  - **읽기 대 쓰기 패턴(Read/Write Pattern)**: 애플리케이션의 트래픽 중 조회(Read)와 생성·수정(Write) 중 어느 작업이 더 지배적인가?
  - **질의 복잡성(Query Complexity)**: 비즈니스 로직을 구현하기 위해 데이터베이스에 어떤 종류의 복잡한 질문(조인, 집계 등)을 던져야 하는가?
  - **확장성 요구사항(Scalability)**: 시간이 흐름에 따라 트래픽과 데이터양이 얼마나 늘어날 것이며, 이에 대한 스케일 아웃(Scale-out)이 용이한가?
  - **일관성 요구사항(Consistency)**: 애플리케이션 비즈니스 도메인에서 데이터의 실시간 정합성과 일관성이 얼마나 치명적인가?

### 1.1. 관계형 데이터베이스(RDBMS)

- **관계형 데이터베이스(RDBMS, Relational Database Management System)** 는 데이터를 행(Row)과 열(Column)로 이루어진 2차원 테이블 구조로 관리하며, 기본 키(PK)와 외래 키(FK)를 사용하여 테이블 간의 정형적인 관계를 맺는다.
- 대표적인 솔루션으로는 PostgreSQL, MySQL, Oracle, Microsoft SQL Server 등이 있다.

#### 트랜잭션(Transaction)과 ACID

- 관계형 데이터베이스는 **ACID 속성**을 엔진 레벨에서 철저하게 준수함으로써 데이터의 강력한 무결성(Integrity)을 보장한다.
  - **원자성(Atomicity)**:
    - 트랜잭션은 '올 오어 낫싱(All or Nothing)'의 원칙을 따르는 논리적 작업 단위다.
    - 트랜잭션 내부에 포함된 모든 변경 사항이 완벽하게 성공(Commit)하거나, 단 하나라도 실패하면 아예 아무것도 실행되지 않은 상태로 롤백(Rollback)되어야 한다.
  - **일관성(Consistency)**:
    - 데이터베이스는 각 트랜잭션이 수행되기 전과 성공적으로 완료된 후 모두 내부적으로 유효한 상태(Valid State)를 유지해야 한다.
    - 데이터베이스에 설정된 모든 도메인 제약 조건, 외래 키 관계, 고유성(Unique) 규칙 등이 엄격하게 지켜지므로 무효한 데이터가 적재되는 상황을 원천 차단한다.
  - **격리성(Isolation)**:
    - 동시에 실행되는 여러 트랜잭션은 서로의 중간 작업 과정에 간섭하거나 영향을 미치지 않는다.
    - 격리 수준(Isolation Level) 설정에 따라 제어되며 동시 다발적인 트랜잭션 요청 속에서도 **더티 리드(Dirty Read)**, **논리피터블 리드(Non-repeatable Read)**, **팬텀 리드(Phantom Read)** 같은 정합성 오류를 방지한다.
  - **지속성(Durability)**:
    - 트랜잭션이 성공적으로 완료되어 커밋되면, 그 결과는 데이터베이스 스토리지에 영구적으로 기록된다.
    - 커밋 직후에 예기치 못한 시스템 장애, 서버 다운, 또는 전원 차단 사태가 발생하더라도 데이터는 완벽하게 보존된다.

#### 관계형 데이터베이스를 사용하는 대표적인 예시

- 엔티티 간에 1:N, N:M 등 명확하고 정형적인 관계가 맺어져 있을 때
- 비즈니스 로직 상 다중 테이블 간의 복잡한 질의와 **조인(Join)** 연산이 빈번하게 필요할 때
- 금융 결제, 자금 이체, 예약 시스템처럼 데이터의 정합성과 **트랜잭션(Transaction)** 의 일관성이 절대적으로 중요할 때
- 스키마 구조가 확립되어 있어 데이터 구조의 변화가 그리 자주 발생하지 않을 때

### 1.2. 문서 데이터베이스(Document Database)

- **문서 데이터베이스(Document Database)** 는 관계형 데이터베이스의 엄격한 테이블 구조를 탈피하여, 유연한 **JSON** 또는 BSON 형태의 문서(Document) 단위로 데이터를 저장하는 대표적인 NoSQL 저장소다.
- 고정된 스키마가 없으므로 하나의 컬렉션(Collection) 안에서도 각 문서마다 서로 전혀 다른 구조의 필드를 가질 수 있어 유연성이 매우 뛰어나다.
- 대표적인 솔루션으로는 MongoDB, Couchbase, 그리고 파이어베이스(Firebase)의 클라우드 파이어스토어(Cloud Firestore) 등이 있다.

#### 문서 데이터베이스를 사용하는 대표적인 예시

- 각 레코드마다 보관해야 하는 데이터의 항목이나 구조가 제각각 다를 때
- 데이터 트래픽과 용량이 방대하여 노드 추가를 통한 **수평 확장(Scale-out / Sharding)** 이 필수적일 때
- 비즈니스 요구사항과 서비스의 스펙이 극심하게 변경되어 스키마 변경 비용을 최소화해야 할 때
- 구조가 동적으로 변하는 콘텐츠 관리 시스템(CMS)이나 커스터마이징이 잦은 이커머스 상품 카탈로그를 구축할 때

### 1.3. 키-값 저장소(Key-Value Store)

- **키-값 저장소(Key-Value Store)** 는 가장 단순하고 직관적인 형태의 NoSQL 데이터베이스로, 고유한 키(Key)와 이에 대응하는 데이터 값(Value)의 쌍으로 이루어진 데이터 구조를 가진다.
- 키를 통한 일대일(O(1)) 매핑 구조를 취하므로 초고속 읽기/쓰기 성능을 자랑하지만, 관계형 데이터베이스와 같은 복잡한 인덱싱이나 조건 절 질의(Query) 기능은 매우 제한적이다.
- 대표적인 솔루션으로는 인메모리(In-Memory) 아키텍처 기반의 레디스(Redis), AWS의 관리형 NoSQL 서비스인 다이나모DB(DynamoDB) 등이 있다.

#### 키-값 저장소를 사용하는 대표적인 예시

- 밀리초(ms) 또는 마이크로초(µs) 단위의 극단적으로 빠른 읽기/쓰기 처리 성능이 요구될 때
- 검색 조건이 복잡하지 않고, 오직 특정 키를 통해 데이터를 조회하는 단순한 데이터 접근 패턴을 가질 때
- 데이터베이스의 부하를 줄이기 위한 임시 **캐시 레이어(Cache Layer)**, 대규모 유저 접속을 관리하는 **세션 저장소(Session Store)**, 분산 시스템의 실시간 환경 설정(Configuration) 도구를 구축할 때
- 다중 테이블 조인이나 집계 연산 같은 복잡한 질의 기능보다, 압도적인 시스템 처리 속도와 수평적 확장성이 훨씬 더 중요할 때

### 1.4. 그래프 데이터베이스(Graph Database)

- **그래프 데이터베이스(Graph Database)** 는 네트워크 형태로 서로 복잡하게 연결된 대규모 데이터를 효율적으로 저장하고 질의하도록 설계된 NoSQL 저장소다.
- 데이터의 정보를 **노드(Node, 개체)** 와 **에지(Edge, 개체 간의 관계)** 로 표현하며, 각 노드와 에지는 고유한 속성(Property)을 가질 수 있다.
- 테이블과 외래 키를 사용하는 관계형 데이터베이스와 달리, 그래프 데이터베이스는 관계(Edge) 자체를 데이터 모델의 **일급 객체(First-class Citizen)** 로 취급하여 물리적으로 연결해 둔다.
- 덕분에 데이터 구조 내에 존재하는 특정 패턴을 빠르게 찾아내고, 수많은 연결 고리를 막힘없이 추적할 수 있도록 돕는 특화된 그래프 질의 언어(Cypher 등)를 지원한다.
- 대표적인 솔루션으로는 Neo4j, JanusGraph, AWS의 관리형 서비스인 아마존 냅튠(Amazon Neptune) 등이 있다.

#### 그래프 데이터베이스를 사용하는 대표적인 예시

- 비즈니스 도메인의 데이터가 다대다(N:M) 형태로 고도로 얽히고 연결되어 있을 때
- 데이터 속성 자체만큼이나 개체 간의 유기적인 관계를 파악하는 것이 비즈니스 핵심일 때
- 다단계 조인을 넘어선 복잡한 관계 순회(Traversing)나 최단 경로 찾기(Shortest Path) 연산을 실시간으로 수행해야 할 때
- 방대한 인맥 레이어의 **소셜 네트워크 서비스(SNS)**, 고도화된 연관 **추천 엔진(Recommendation Engine)**, 지식 데이터간의 상관관계를 정의하는 **지식 그래프(Knowledge Graph)** 를 구축할 때

### 1.5. 벡터 데이터베이스(Vector Database)

- **벡터 데이터베이스(Vector Database)** 는 AI 모델이 생성한 고차원 수치 벡터(Numerical Vector) 데이터를 안정적으로 적재하고, 이들 간의 **유사도 검색(Similarity Search)** 을 초고속으로 수행하도록 설계된 특수 목적형 데이터베이스 시스템이다.
- 문자나 숫자 같은 이산적인(Discrete) 값을 고정된 필드에 저장하는 전통적인 데이터베이스와 달리, 벡터 데이터베이스는 비정형 데이터의 맥락적·의미론적 의미를 포착한 수학적 표현체인 **임베딩(Embedding)** 데이터를 다룬다.
- 대표적인 솔루션으로는 클라우드 네이티브 서비스인 파인콘(Pinecone), 웨비에이트(Weaviate), 오픈소스 기반의 크로마(Chroma), Qdrant 등이 있다.

#### 벡터 임베딩(Vector Embedding)

- 텍스트, 이미지, 오디오 등의 원천 데이터를 AI 인코더 모델을 거쳐 수백에서 수천 차원의 고차원 공간상에 숫자의 배열로 변환하여 나타내는 방식이다.
- 비즈니스 의미나 맥락이 서로 비슷한 항목일수록 이 고차원 공간 속에서 서로 가까운 거리(밀접한 벡터)를 가지며, 전혀 다른 성격의 항목들은 수학적으로 멀리 떨어지게 배치된다.

#### 유사도 검색(Similarity Search)

- 벡터 데이터베이스의 데이터 접근 패러다임으로, 유저가 던진 질의와 가장 가깝거나 유사한 맥락을 가진 항목들을 찾아내는 핵심 작업이다.
- 전통적인 데이터베이스처럼 쿼리와 정확히 일치하는(Exact Match) 텍스트를 찾는 것이 아니라, "이 질문의 의도와 가장 의미가 비슷한 콘텐츠를 순서대로 보여달라"고 요청하는 방식(코사인 유사도, 유클리드 거리 등 활용)이다.

#### 벡터 데이터베이스를 사용하는 대표적인 예시

- **의미론적 검색(Semantic Search)**: 특정 키워드가 본문에 직접 포함되어 있지 않더라도, 질문의 문맥과 의미를 기반으로 연관 문서를 찾아내야 할 때
- **추천 시스템**: 유저가 소비한 콘텐츠의 벡터 성향을 분석하여, 시각적·의미론적으로 유사한 제품이나 미디어를 맞춤형으로 제안할 때
- **검색 증강 생성(RAG, Retrieval-Augmented Generation)**: **대규모 언어 모델(LLM)** 이 최신 정보나 기업 내부 지식을 바탕으로 거짓 정보(Hallucination) 없이 정확한 답변을 생성할 수 있도록 관련 컨텍스트(Context)를 실시간으로 검색하여 주입하는 아키텍처를 구축할 때
- **이미지/비디오 유사도 검색**: 텍스트 태그 없이 오직 업로드된 이미지나 비디오 파일의 시각적 특징을 비교하여 유사한 미디어를 찾아내야 할 때
- **이상 탐지(Anomaly Detection)**: 고차원 금융 데이터나 보안 로그의 패턴을 분석하여 평소의 벡터 군집(Cluster) 범위를 벗어나는 이상치(Outlier)를 조기에 식별할 때
- **중복 제거(Deduplication)**: 띄어쓰기나 어미가 조금씩 달라 정밀 매칭으로는 잡히지 않는 거의 유사한 중복 문서 및 가짜 뉴스 콘텐츠를 찾아내어 정제할 때

## 2. 데이터 영속성 및 관리

- **데이터 영속성(Data Persistence)** 의 추상화 수준은 개발 속도, 코드의 유지보수성, 그리고 애플리케이션의 런타임 성능에 막대한 영향을 미친다.
- 애플리케이션의 복잡성, 개발 팀의 기술적 전문성, 그리고 프로젝트의 일정에 따라 최적의 영속성 전략을 선택해야 한다.

### 2.1. 데이터베이스 직접 접근

- 데이터베이스 직접 접근(Direct Database Access)은 가장 낮은 수준의 추상화 단계로, 개발자가 코드 내에 원시 SQL(Raw SQL)이나 데이터베이스 전용 질의 언어를 직접 작성하여 데이터베이스와 소통하는 방식이다.
- 데이터베이스 엔진에 대한 완전한 제어권과 최적화 기회를 제공하지만, 반대로 깊이 있는 데이터베이스 지식이 필요하며 문구 오류나 **SQL 인젝션(SQL Injection)** 같은 보안 취약점에 노출되기 쉽다.

### 2.2. 저장소 패턴(Repository Pattern)

- **저장소 패턴(Repository Pattern)** 은 순수한 비즈니스 로직(도메인 레이어)과 비구조화된 데이터 접근 코드(인프라 레이어) 사이에 명확한 추상화 계층을 제공하는 디자인 패턴이다.
- 도메인 객체에 접근하는 인메모리 컬렉션(Collection)과 유사한 인터페이스 역할을 수행하며, 구체적인 데이터베이스 CRUD 작업의 복잡성을 내부로 숨긴다.
- 저장소 패턴은 다음과 같은 강력한 이점을 제공한다.
  - **관심사의 분리(SoC, Separation of Concerns)**: 데이터 접근 로직을 비즈니스 로직과 철저히 분리하여 코드가 엉키는 것을 방지한다.
  - **테스트 용이성**: 인터페이스를 기반으로 설계되므로, 실제 데이터베이스를 구동하지 않고도 **모의 저장소(Mock Repository)** 객체를 주입하여 비즈니스 로직의 단위 테스트를 쉽게 수행할 수 있다.
  - **코드 구성 최적화**: 파편화되기 쉬운 데이터 접근 로직을 특정 저장소 클래스 한곳에 집중시켜 관리 효율성을 높인다.

### 2.3. 객체 관계 매핑(ORM)

- **하이버네이트(Hibernate)**, **JPA(Java Persistence API)**, 엔티티 프레임워크(Entity Framework) 같은 **객체 관계 매핑(ORM, Object-Relational Mapping)** 프레임워크는 데이터베이스 상호작용에서 가장 높은 수준의 추상화를 제공한다.
- 애플리케이션의 도메인 객체를 데이터베이스 테이블에 메타데이터를 통해 직접 매핑한다. 이를 통해 개발자가 수동으로 SQL을 작성하고 ResultSet을 객체로 변환(Mapping)하는 반복적인 번거로움을 제거함으로써, 객체 지향 프로그래밍 패러다임과 관계형 데이터베이스 사이의 패러다임 불일치(Impedance Mismatch) 간극을 메워준다.

## 3. 데이터베이스 연결과 트랜잭션

- 데이터베이스 연결(Connection)은 애플리케이션이 데이터베이스 엔진과 통신하는 물리적·논리적 방식을 정의하며, 트랜잭션(Transaction)은 단일 작업 단위 안에서 복수의 상태 변경 작업이 격리되어 함께 실행되는 방식을 제어한다.

### 3.1. 데이터베이스 연결

- 엔터프라이즈 애플리케이션 작업을 수행할 때는 적절한 연결 관리 전략을 적용해야 시스템의 성능과 신뢰성을 극대화할 수 있다.
- 매 요청마다 비용이 비싼 네트워크 연결을 새롭게 생성하고 폐기하는 방식 대신, 애플리케이션 구동 시점에 미리 일정 수의 연결을 맺어두고 재사용하는 **커넥션 풀(Connection Pool)** 아키텍처를 도입해야 한다.
- 자바 생태계에서는 압도적인 속도와 안정성을 자랑하는 **히카리CP(HikariCP)** 가 표준 커넥션 풀 라이브러리로 널리 사용된다.

### 3.2. 트랜잭션(Transaction)

- **데이터베이스 트랜잭션(Database Transaction)** 은 하나 이상의 데이터베이스 변경 작업을 하나의 논리적인 원자적 작업으로 묶어서 실행하는 단위다.
- 트랜잭션은 네트워크 장애나 비즈니스 예외 등으로 인해 데이터의 일부만 반영되는 불안정한 상태(Partial State)를 완전히 차단함으로써, 데이터의 강력한 무결성을 유지하는 최후의 보루 역할을 한다.

## 4. 일관성 모델 및 캐시 전략

- 분산 아키텍처 환경에서 시스템의 기초는 높은 성능을 제공하면서도, 도메인 데이터를 안정적이고 정합성 있게 관리하는 능력에 달려 있다.

### 4.1. 일관성 모델(Consistency Model)

- 트랜잭션 개념을 확장한 **일관성 모델(Consistency Model)** 은 시스템의 여러 영역에서 데이터의 정확성을 애플리케이션 레벨이 어떻게 다룰지 정의하는 명세다.
- 단일 트랜잭션이 개별 CRUD 작업의 데이터 무결성을 유지한다면, 일관성 모델은 여러 대의 분산 서버에 데이터가 복제되어 있거나 다중 사용자가 동시에 동일한 정보에 접근할 때 전체적인 데이터 정합성이 어떻게 유지되는지 결정한다.
- 데이터베이스 레이어에서 일관성이란 ACID 속성 중 하나로, 모든 트랜잭션이 데이터베이스를 하나의 유효한 상태(Valid State)에서 또 다른 유효한 상태로 이동시키면서 정의된 모든 비즈니스 규칙, 제약 조건, 스키마 관계를 모순 없이 준수하게 만드는 것을 의미한다.
- 비즈니스 도메인마다 변경된 데이터가 시스템 전체 노드에 얼마나 빨리 노출되어야 하는지에 대한 요구사항이 다르므로 다음과 같은 여러 가지 모델을 선택하여 적용한다.

#### 일관성 모델 유형

- **강력한 일관성(Strong Consistency)**:
  - 시스템의 모든 분산 노드에서 동시에 항상 동일한 데이터를 바라보며, 변경 사항이 발생한 정확한 시퀀스(순서)대로 클라이언트에 노출되도록 보장한다.
  - 예를 들어 사용자 계좌 잔액을 변경하면, 이후에 발생하는 모든 읽기(Read) 작업에는 즉시 그 변경 사항이 반영된다.
  - 시스템은 이를 확실하게 보장하기 위해 데이터 동기화가 완료될 때까지 일시적으로 다른 요청을 차단(Locking)할 수 있으며, 시스템 가용성보다 데이터의 정확성을 절대적으로 우선시한다.
- **최종 일관성(Eventual Consistency)**:
  - 시간이 흐르면 결론적으로 동기화가 해결되는 일시적인 데이터 불일치 상태를 허용함으로써, 애플리케이션이 멈춤 없이 빠르게 실행되는 가용성을 우선한다.
  - 예를 들어 글로벌 쇼핑몰에서 제품 가격을 업데이트했을 때, 전 세계 모든 사용자에게 즉시 동일한 가격이 보이는 것이 아니라 분산 캐시나 레플리카 노드의 전파 속도에 따라 잠시 이전 가격을 볼 수도 있는 상황을 수용한다.
  - 이 모델은 실시간 즉각적인 일관성이 비즈니스에 치명적이지 않은 대규모 트래픽 환경에서 매우 유용하다.
- **인과 일관성(Causal Consistency)**:
  - 인과 관계(Cause and Effect)로 묶여 있는 작업 간의 실행 순서만을 엄격하게 유지하는 모델이다.
  - 예를 들어 협업 도구에서 사용자 A가 작성한 특정 편집 내용에 대해 사용자 B가 댓글을 달았다면, 시스템의 모든 사용자는 반드시 댓글보다 편집 내용을 먼저 보게끔 순서가 보장된다. 반면 사용자 C가 진행한 이와 무관한 독립적 편집 사항은 사용자마다 서로 다른 순서로 나타날 수 있다.
  - 강력한 일관성과 최종 일관성 사이에서 적절한 기술적 타협점을 제공한다.
- **세션 일관성(Session Consistency)**:
  - 개별 사용자 세션(Session) 범위 내에서는 완벽한 일관성을 보장하되, 서로 다른 사용자 간의 일시적인 데이터 차이는 허용하는 방식이다.
  - 예를 들어 내가 장바구니에 상품을 추가하면, 다른 사용자의 화면에서는 재고 동기화 지연으로 살짝 잘못된 이전 수량이 보일지라도, 내 세션 내부에서는 항상 내가 추가한 상품이 정확하게 반영되어 보인다.
  - 개별 사용자 관점에서 논리적으로 모순 없는 사용자 경험(UX)을 제공할 때 유용하다.

#### CAP 정리(CAP Theorem)

- **CAP 정리(CAP Theorem)** 는 분산 데이터 시스템이 가질 수 있는 근본적인 트레이드오프(Trade-off) 구조를 네트워크 수준에서 설명하는 이론이다.
- 데이터가 네트워크를 통해 여러 서버에 분산 및 복제되어 있을 때, 아래 3가지 속성 중 최대 두 가지만 동시에 만족할 수 있으며 세 가지를 모두 완벽하게 보장하는 분산 시스템은 존재하지 않는다.
  - **일관성(Consistency)**: 분산된 모든 노드가 어느 시점에나 동시에 항상 같은 데이터를 보여준다.
  - **가용성(Availability)**: 일부 분산 노드에 장애가 발생하여 멈추더라도, 정상 노드는 계속해서 클라이언트의 요청에 응답할 수 있다.
  - **분할 내성(Partition Tolerance)**: 노드 간의 네트워크 연결이 끊어지는 물리적 단절(Network Partition) 상황이 발생해도 전체 시스템은 가동을 멈추지 않는다.
- 실무 환경에서 네트워크 단절 위험은 상존하므로 분할 내성(P)은 필수 조건이며, 이에 따라 시스템 설계 방향은 크게 다음과 같이 도출된다.
  - **CP 시스템**:
    - 가용성(A)을 일시적으로 희생하더라도 데이터의 일관성(C)과 분할 내성(P)을 최우선으로 확보하는 아키텍처다.
    - 네트워크 파티션 문제가 발생하면 잘못되거나 오래된 데이터를 노출할 위험을 감수하기보다, 시스템이 차라리 클라이언트의 요청을 거절(에러 반환)하는 방식을 취한다.
    - 데이터의 정확성이 생명인 금융 권 계좌 시스템, 코어 재고 관리 시스템, 인증 서비스 등이 여기에 해당된다.
  - **AP 시스템**:
    - 일간성(C)을 일시적으로 타협하는 대신 시스템의 가용성(A)과 분할 내성(P)에 집중하며 최종 일관성을 수용하는 아키텍처다.
    - 콘텐츠 전송 네트워크(CDN), 소셜 미디어 피드 시스템, 연관 추천 엔진 등 시스템이 중단 없이 돌아가는 가용성이 실시간 정합성보다 훨씬 중요한 도메인에 이상적이다.
    - 유저가 최신 게시물을 몇 초 늦게 보더라도 전체 서비스가 먹통이 되는 것보다는 약간 오래된 소셜 미디어 글이라도 정상 노출되는 편이 낫기 때문이다.
  - **CA 시스템**:
    - 데이터의 강력한 일관성(C)과 상시 가용성(A)을 동시에 제공하지만, 분산 환경의 핵심인 네트워크 분할 내성(P) 구조를 가지지 못한다.
    - 따라서 단일 노드 데이터베이스나 완벽하게 신뢰할 수 있는 네트워크 인프라 위에 묶인 로컬 클러스터 환경에서만 성립 가능하다.
    - 네트워크 분산 아키텍처가 없는 단일 인스턴스 형태의 MySQL이나 PostgreSQL 같은 전통적인 데이터베이스 관리 시스템이 기본적으로 이 모드로 작동한다.

### 4.2. 캐시(Cache)를 사용한 성능 최적화

- **캐시(Cache)** 는 애플리케이션의 응답 속도를 획기적으로 낮추고 백엔드 데이터베이스의 IOPS 부하를 줄이기 위해, 자주 조회되는 데이터를 디스크보다 접근 속도가 압도적으로 빠른 고속 메모리 공간에 임시 저장하는 기술이다.
- 캐시는 애플리케이션이 데이터를 요청할 때 애플리케이션 레이어와 데이터베이스 레이어 사이에서 고속 메모리 저장 계층 역할을 수행한다.
- 데이터베이스를 직접 호출하면 아무리 가벼운 쿼리라 할지라도 내부적인 네트워크 핸드셰이크와 디스크 I/O를 거치며 수십에서 수백 밀리초($\text{ms}$)의 비용이 고스란히 소요된다.
- 캐시는 데이터를 고속 **인메모리(In-Memory)** 에 적재하므로, 시스템 상황에 따라 응답 시간을 최소 10배에서 100배 이상 단축시킬 수 있다.
- 하지만 캐시 계층을 도입하면 데이터 원본이 변경되었을 때 캐시를 적절히 만료시키고 최신 상태를 유지하기 위한 추가적인 설계 복잡성이 수반되며, 인프라적으로 잠재적 실패 지점(Single Point of Failure)이 하나 더 추가된다는 트레이드오프가 존재한다.
- 따라서 명확한 캐시 만료(Invalidation) 정책을 정의해야 하며, 캐시 데이터와 진실의 원천(Single Source of Truth)인 메인 데이터베이스 간의 동기화 관계를 정밀하게 관리해야 한다.

#### 일반적인 캐시 전략

- **조회 후 적재(Cache-Aside / Lazy Loading)**:
  - 애플리케이션 코드가 캐시와 데이터베이스 제어를 모두 직접 책임지는 가장 보편적인 방식이다.
  - 읽기 요청이 들어오면 애플리케이션은 먼저 캐시 서버에 데이터가 존재하는지 확인한다.
  - 캐시에 데이터가 없는 **캐시 미스(Cache Miss)** 가 발생하면 데이터베이스에서 원본 데이터를 직접 조회한 후, 향후 조회를 위해 캐시에 데이터를 적재하고 클라이언트에 반환한다.
  - 데이터가 이미 캐싱되어 있는 **캐시 히트(Cache Hit)** 상태라면 데이터베이스를 거치지 않고 캐시에서 즉시 데이터를 읽어온다.
  - 읽기 중심의 트래픽을 처리하고 캐시 대상을 정밀하게 제어하고 싶을 때 적합하다. 단, 데이터의 첫 번째 요청은 캐시 미스로 인해 항상 데이터베이스를 거치므로 지연이 발생한다.
  - 이를 방지하기 위해 서비스 오픈 전 캐시에 데이터를 미리 밀어 넣는 **웜업(Warm-up)** 작업을 수행하기도 한다.
- **동기 기록(Write-Through)**:
  - 애플리케이션이 데이터를 저장하거나 수정할 때, 캐시와 메인 데이터베이스에 동시에 쓰기 작업을 수행하는 전략이다.
  - 캐시와 데이터베이스가 항상 완벽하게 동일한 최신 데이터를 유지하도록 보장하므로 데이터 불일치 위험이 없다.
  - 그러나 모든 데이터 쓰기 작업마다 두 번의 저장소 네트워크 I/O가 발생하므로 쓰기 레이턴시(Latency)가 증가한다.
  - 캐시와 데이터베이스 간의 강력한 일관성이 절대적으로 요구되며, 쓰기보다 읽기 트래픽의 비중이 훨씬 높은 도메인에 적합하다. 자주 읽지 않는 무분별한 데이터까지 캐시에 계속 쌓일 수 있으므로 적절한 TTL 설정이 필수적이다.
- **지연 기록(Write-Back / Write-Behind)**:
  - 애플리케이션이 모든 쓰기 작업을 고속 캐시 서버에만 즉시 기록한 뒤, 백엔드 데이터베이스에는 특정 주기나 대량의 청크 단위로 모아서 비동기(Asynchronous) 업데이트를 수행하는 전략이다.
  - 쓰기 작업이 캐시 메모리 속도 수준으로 처리되므로 쓰기 성능을 극대화할 수 있으며, 데이터베이스 디스크 쓰기 부하를 대폭 경감시킨다.
  - 그러나 캐시에만 데이터가 기록된 상태에서 데이터베이스로 영속화되기 전에 캐시 서버가 다운되거나 장애가 발생하면, 메모리에 머물던 최신 데이터가 유실될 수 있는 치명적인 위험을 안고 있다.
  - 약간의 데이터 유실 위험을 감수하더라도 극단적인 쓰기 성능 대역폭이 필요한 로그 수집, 게임 실시간 지표 분석, 대규모 유저 활동 스트리밍 데이터 처리 등에 적합하다.

#### 캐시 도입 시 고려 사항

- 캐시는 성능 향상을 위한 강력한 도구이지만 도입에 따른 아키텍처 복잡성 증가라는 명확한 대가가 따른다. 캐시 적합성을 판단할 때는 아래의 체크리스트를 참고해야 한다.
  - 이 도메인 데이터의 변경 빈도가 얼마나 낮은가? (자주 변경되는 데이터는 캐시 효율이 떨어진다.)
  - 사용자에게 일시적으로 오래된 레거시 데이터가 노출되었을 때 비즈니스적 손실이나 장애 위험은 어느 정도인가?
  - 현재 데이터베이스의 특정 SQL 조회 비용과 디스크 I/O 병목이 시스템 전체 성능에 얼마나 악영향을 미치고 있는가?

#### 캐시와 일관성

- 데이터를 캐싱한다는 것은 원본 데이터가 변경되었을 때 지난 상태가 될 수 있는 유효기간이 있는 임시 복사본을 유지하는 행위다. 정합성 수준에 따라 다음과 같이 대응한다.
  - **TTL(Time-To-Live) 기반 Cache-Aside**: 캐시 아이템에 적절한 만료 시간(TTL)을 설정하여, 시간이 흐르면 자동으로 메모리에서 파기되도록 유도함으로써 최종 일관성을 자연스럽게 만족시킨다.
  - **Write-Through 캐시**: 데이터 생성 및 수정 시점에 무조건 캐시를 함께 갱신하여, 쓰기 레이턴시 손해를 보더라도 캐시와 원본 데이터 간의 강력한 일관성을 강제한다.
  - **Write-Back 캐시**: 성능을 극대화하는 대가로 비동기 동기화 주기 동안 발생하는 데이터의 일시적인 불일치 상태를 비즈니스적으로 수용한다.

#### 분산 시스템에서의 분산 캐시(Distributed Cache)

- 마이크로서비스나 다중 WAS 환경에서는 다수의 캐시 노드가 인프라에 존재하므로 캐시 일관성 문제가 훨씬 더 복잡해진다.
- 개별 서버의 로컬 메모리 캐시를 사용할 경우 서버 간 데이터 불일치가 발생하므로, 일반적으로 **레디스(Redis)** 나 Memcached 같은 전역적인 공통 **분산 캐시(Distributed Cache)** 인프라를 별도로 구축하여 운영한다.
- 데이터의 양이 늘어나면 캐시 서버 자체를 수평 확장하기 위해 데이터를 노드별로 쪼개어 보관하는 **샤딩(Sharding)** 및 레플리케이션(Replication) 복제 아키텍처 전략을 정밀하게 설계해야 한다.

## 5. 데이터 증가 대책

- 데이터양이 기하급수적으로 증가할 때는 시스템의 성능 지표를 정밀하게 모니터링해야 한다. 특히 SQL 질의 응답 시간이 허용된 임계치를 초과하거나, 데이터베이스 서버의 CPU 사용률이 지속적으로 고점을 찍을 때, 메모리(RAM) 점유율이 꾸준히 우상향할 때, 그리고 느린 반응 속도로 인해 사용자 불만이 증가할 때는 아키텍처 관점의 데이터 증가 대책을 즉시 수립해야 한다.

### 5.1. 확장 전략의 활용

- **수직 확장(Scale-up, Vertical Scaling)**:
  - 기존 데이터베이스 서버의 하드웨어 스펙을 높이기 위해 자원(CPU, RAM, SSD)을 추가하는 방식이다.
  - 별도의 애플리케이션 아키텍처 변경이나 소스 코드 수정 없이 적용할 수 있어 초기 대응으로 가장 선호되지만, 부품 자체의 물리적인 하드웨어 한계에 금방 부딪히며 비용 증가 폭이 비선형적으로 크다는 단점이 있다.
- **수평 확장(Scale-out, Horizontal Scaling)**:
  - 저렴한 사양의 데이터베이스 서버 노드를 여러 대 추가하여 데이터를 분산 저장 및 처리하는 방식이다.
  - 수직 확장이 한계에 도달했거나, 글로벌 서비스 환경에서 지리적인 데이터 분산 처리가 필요할 때 반드시 도입해야 하는 필수 전략이다.
- **읽기 전용 복제본(Read Replica)**:
  - 주 데이터베이스(Primary/Master Node)는 생성·수정·삭제(Write) 작업만 전담하고, 복제된 사본 데이터베이스(Secondary/Replica Node)를 여러 대 두어 조회(Read) 작업을 분산 처리하는 아키텍처다.
  - 일반적인 읽기 중심(Read-Heavy) 웹 애플리케이션에서 데이터베이스 부하를 줄이기 위해 가장 먼저 고려해야 하는 전략이다. 전체 데이터베이스를 샤딩하는 것보다 구현 난이도가 낮으면서도 시스템 조회 성능을 획기적으로 향상할 수 있다.
- **샤딩(Sharding / 파편화)**:
  - 사전에 정의된 **샤드 키(Shard Key)** 를 기반으로 거대한 데이터셋을 서로 다른 독립된 물리 데이터베이스 노드에 수평적으로 분할하여 저장하는 방식이다.
  - NoSQL이나 분산 RDB 환경에서 가장 복잡하고 관리 리소스가 많이 드는 기술이지만, 단일 서버의 파일 시스템 용량을 초과하는 초대용량 데이터를 가진 대규모 글로벌 애플리케이션을 안정적으로 지탱하기 위해 필수적으로 요구된다.

### 5.2. 성장 중 성능 유지 관리

- 실무 환경에서 메인 저장소의 작업 데이터 용량(Working Set)이 메모리 범위 내에서 안정적이고 관리 가능한 수준으로 유지되도록, 오래된 레거시 데이터를 콜드 스토리지로 이관하거나 삭제하는 별도의 **데이터 보존 정책(Data Retention Policy)** 및 아카이빙 파이프라인을 구축해야 한다.
- 카디널리티(Cardinality)를 고려하여 적절한 데이터베이스 **인덱스(Index)** 를 설계해야 하며, 비즈니스 요구사항에 따라 웹 애플리케이션의 질의(Query) 패턴이 진화함에 따라 정기적으로 기존 인덱스의 적절성을 검토하고 사용되지 않는 인덱스는 과감히 제거해야 한다.
- 데이터베이스 엔진의 실행 계획을 주기적으로 분석하고, 시스템을 갉아먹는 **슬로우 쿼리(Slow Query)** 를 탐지하여 최적화(Tuning)하는 등의 정기적인 데이터베이스 DBA 유지보수 프로세스를 실무 일정에 미리 예약 및 자동화해 두어야 한다.
