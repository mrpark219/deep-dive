# 효과적인 데이터 저장

- 데이터 구조를 정의했다면, 그다음 단계로는 그 데이터를 안전하고 효율적으로 담을 수 있는 최적의 저장 방식을 선택해야 한다.

## 1. 데이터베이스 유형과 사용 사례

- 저장 솔루션(Storage Solution)을 선택할 때는 시스템의 요구사항에 맞춰 다음과 같은 핵심 요소를 반드시 고려해야 한다.
  - **데이터 구조의 복잡성**: 데이터 엔티티 간의 연관 관계가 얼마나 복잡하게 얽혀 있는가?
  - **읽기 대 쓰기 패턴(Read/Write Pattern)**: 애플리케이션의 트래픽 중 조회(Read)와 생성·수정(Write) 중 어느 작업이 더 지배적인가?
  - **질의 복잡성(Query Complexity)**: 비즈니스 로직을 구현하기 위해 데이터베이스에 어떤 종류의 복잡한 질문(조인, 집계 등)을 던져야 하는가?
  - **확장성 요구사항(Scalability)**: 시간이 흐름에 따라 트래픽과 데이터양이 얼마나 늘어날 것이며, 이에 대한 스케일 아웃(Scale-out)이 용이한가?
  - **일관성 요구사항(Consistency)**: 애플리케이션 비즈니스 도메인에서 데이터의 실시간 정합성과 일관성이 얼마나 치명적인가?

### 1.1. 관계형 데이터베이스(RDBMS)

- **관계형 데이터베이스(RDBMS, Relational Database Management System)** 는 데이터를 행(Row)과 열(Column)로 이루어진 2차원 테이블 구조로 관리하며, 기본 키(PK)와 외래 키(FK)를 사용하여 테이블 간의 정형적인 관계를 맺는다.
- 대표적인 솔루션으로는 PostgreSQL, MySQL, Oracle, Microsoft SQL Server 등이 있다.

#### 트랜잭션(Transaction)과 ACID

- 관계형 데이터베이스는 **ACID 속성**을 엔진 레벨에서 철저하게 준수함으로써 데이터의 강력한 무결성(Integrity)을 보장한다.
  - **원자성(Atomicity)**:
    - 트랜잭션은 '올 오어 낫싱(All or Nothing)'의 원칙을 따르는 논리적 작업 단위다.
    - 트랜잭션 내부에 포함된 모든 변경 사항이 완벽하게 성공(Commit)하거나, 단 하나라도 실패하면 아예 아무것도 실행되지 않은 상태로 롤백(Rollback)되어야 한다.
  - **일관성(Consistency)**:
    - 데이터베이스는 각 트랜잭션이 수행되기 전과 성공적으로 완료된 후 모두 내부적으로 유효한 상태(Valid State)를 유지해야 한다.
    - 데이터베이스에 설정된 모든 도메인 제약 조건, 외래 키 관계, 고유성(Unique) 규칙 등이 엄격하게 지켜지므로 무효한 데이터가 적재되는 상황을 원천 차단한다.
  - **격리성(Isolation)**:
    - 동시에 실행되는 여러 트랜잭션은 서로의 중간 작업 과정에 간섭하거나 영향을 미치지 않는다.
    - 격리 수준(Isolation Level) 설정에 따라 제어되며 동시 다발적인 트랜잭션 요청 속에서도 **더티 리드(Dirty Read)**, **논리피터블 리드(Non-repeatable Read)**, **팬텀 리드(Phantom Read)** 같은 정합성 오류를 방지한다.
  - **지속성(Durability)**:
    - 트랜잭션이 성공적으로 완료되어 커밋되면, 그 결과는 데이터베이스 스토리지에 영구적으로 기록된다.
    - 커밋 직후에 예기치 못한 시스템 장애, 서버 다운, 또는 전원 차단 사태가 발생하더라도 데이터는 완벽하게 보존된다.

#### 관계형 데이터베이스를 사용하는 대표적인 예시

- 엔티티 간에 1:N, N:M 등 명확하고 정형적인 관계가 맺어져 있을 때
- 비즈니스 로직 상 다중 테이블 간의 복잡한 질의와 **조인(Join)** 연산이 빈번하게 필요할 때
- 금융 결제, 자금 이체, 예약 시스템처럼 데이터의 정합성과 **트랜잭션(Transaction)** 의 일관성이 절대적으로 중요할 때
- 스키마 구조가 확립되어 있어 데이터 구조의 변화가 그리 자주 발생하지 않을 때

### 1.2. 문서 데이터베이스(Document Database)

- **문서 데이터베이스(Document Database)** 는 관계형 데이터베이스의 엄격한 테이블 구조를 탈피하여, 유연한 **JSON** 또는 BSON 형태의 문서(Document) 단위로 데이터를 저장하는 대표적인 NoSQL 저장소다.
- 고정된 스키마가 없으므로 하나의 컬렉션(Collection) 안에서도 각 문서마다 서로 전혀 다른 구조의 필드를 가질 수 있어 유연성이 매우 뛰어나다.
- 대표적인 솔루션으로는 MongoDB, Couchbase, 그리고 파이어베이스(Firebase)의 클라우드 파이어스토어(Cloud Firestore) 등이 있다.

#### 문서 데이터베이스를 사용하는 대표적인 예시

- 각 레코드마다 보관해야 하는 데이터의 항목이나 구조가 제각각 다를 때
- 데이터 트래픽과 용량이 방대하여 노드 추가를 통한 **수평 확장(Scale-out / Sharding)** 이 필수적일 때
- 비즈니스 요구사항과 서비스의 스펙이 극심하게 변경되어 스키마 변경 비용을 최소화해야 할 때
- 구조가 동적으로 변하는 콘텐츠 관리 시스템(CMS)이나 커스터마이징이 잦은 이커머스 상품 카탈로그를 구축할 때

### 1.3. 키-값 저장소(Key-Value Store)

- **키-값 저장소(Key-Value Store)** 는 가장 단순하고 직관적인 형태의 NoSQL 데이터베이스로, 고유한 키(Key)와 이에 대응하는 데이터 값(Value)의 쌍으로 이루어진 데이터 구조를 가진다.
- 키를 통한 일대일(O(1)) 매핑 구조를 취하므로 초고속 읽기/쓰기 성능을 자랑하지만, 관계형 데이터베이스와 같은 복잡한 인덱싱이나 조건 절 질의(Query) 기능은 매우 제한적이다.
- 대표적인 솔루션으로는 인메모리(In-Memory) 아키텍처 기반의 레디스(Redis), AWS의 관리형 NoSQL 서비스인 다이나모DB(DynamoDB) 등이 있다.

#### 키-값 저장소를 사용하는 대표적인 예시

- 밀리초(ms) 또는 마이크로초(µs) 단위의 극단적으로 빠른 읽기/쓰기 처리 성능이 요구될 때
- 검색 조건이 복잡하지 않고, 오직 특정 키를 통해 데이터를 조회하는 단순한 데이터 접근 패턴을 가질 때
- 데이터베이스의 부하를 줄이기 위한 임시 **캐시 레이어(Cache Layer)**, 대규모 유저 접속을 관리하는 **세션 저장소(Session Store)**, 분산 시스템의 실시간 환경 설정(Configuration) 도구를 구축할 때
- 다중 테이블 조인이나 집계 연산 같은 복잡한 질의 기능보다, 압도적인 시스템 처리 속도와 수평적 확장성이 훨씬 더 중요할 때

### 1.4. 그래프 데이터베이스(Graph Database)

- **그래프 데이터베이스(Graph Database)** 는 네트워크 형태로 서로 복잡하게 연결된 대규모 데이터를 효율적으로 저장하고 질의하도록 설계된 NoSQL 저장소다.
- 데이터의 정보를 **노드(Node, 개체)** 와 **에지(Edge, 개체 간의 관계)** 로 표현하며, 각 노드와 에지는 고유한 속성(Property)을 가질 수 있다.
- 테이블과 외래 키를 사용하는 관계형 데이터베이스와 달리, 그래프 데이터베이스는 관계(Edge) 자체를 데이터 모델의 **일급 객체(First-class Citizen)** 로 취급하여 물리적으로 연결해 둔다.
- 덕분에 데이터 구조 내에 존재하는 특정 패턴을 빠르게 찾아내고, 수많은 연결 고리를 막힘없이 추적할 수 있도록 돕는 특화된 그래프 질의 언어(Cypher 등)를 지원한다.
- 대표적인 솔루션으로는 Neo4j, JanusGraph, AWS의 관리형 서비스인 아마존 냅튠(Amazon Neptune) 등이 있다.

#### 그래프 데이터베이스를 사용하는 대표적인 예시

- 비즈니스 도메인의 데이터가 다대다(N:M) 형태로 고도로 얽히고 연결되어 있을 때
- 데이터 속성 자체만큼이나 개체 간의 유기적인 관계를 파악하는 것이 비즈니스 핵심일 때
- 다단계 조인을 넘어선 복잡한 관계 순회(Traversing)나 최단 경로 찾기(Shortest Path) 연산을 실시간으로 수행해야 할 때
- 방대한 인맥 레이어의 **소셜 네트워크 서비스(SNS)**, 고도화된 연관 **추천 엔진(Recommendation Engine)**, 지식 데이터간의 상관관계를 정의하는 **지식 그래프(Knowledge Graph)** 를 구축할 때

### 1.5. 벡터 데이터베이스(Vector Database)

- **벡터 데이터베이스(Vector Database)** 는 AI 모델이 생성한 고차원 수치 벡터(Numerical Vector) 데이터를 안정적으로 적재하고, 이들 간의 **유사도 검색(Similarity Search)** 을 초고속으로 수행하도록 설계된 특수 목적형 데이터베이스 시스템이다.
- 문자나 숫자 같은 이산적인(Discrete) 값을 고정된 필드에 저장하는 전통적인 데이터베이스와 달리, 벡터 데이터베이스는 비정형 데이터의 맥락적·의미론적 의미를 포착한 수학적 표현체인 **임베딩(Embedding)** 데이터를 다룬다.
- 대표적인 솔루션으로는 클라우드 네이티브 서비스인 파인콘(Pinecone), 웨비에이트(Weaviate), 오픈소스 기반의 크로마(Chroma), Qdrant 등이 있다.

#### 벡터 임베딩(Vector Embedding)

- 텍스트, 이미지, 오디오 등의 원천 데이터를 AI 인코더 모델을 거쳐 수백에서 수천 차원의 고차원 공간상에 숫자의 배열로 변환하여 나타내는 방식이다.
- 비즈니스 의미나 맥락이 서로 비슷한 항목일수록 이 고차원 공간 속에서 서로 가까운 거리(밀접한 벡터)를 가지며, 전혀 다른 성격의 항목들은 수학적으로 멀리 떨어지게 배치된다.

#### 유사도 검색(Similarity Search)

- 벡터 데이터베이스의 데이터 접근 패러다임으로, 유저가 던진 질의와 가장 가깝거나 유사한 맥락을 가진 항목들을 찾아내는 핵심 작업이다.
- 전통적인 데이터베이스처럼 쿼리와 정확히 일치하는(Exact Match) 텍스트를 찾는 것이 아니라, "이 질문의 의도와 가장 의미가 비슷한 콘텐츠를 순서대로 보여달라"고 요청하는 방식(코사인 유사도, 유클리드 거리 등 활용)이다.

#### 벡터 데이터베이스를 사용하는 대표적인 예시

- **의미론적 검색(Semantic Search)**: 특정 키워드가 본문에 직접 포함되어 있지 않더라도, 질문의 문맥과 의미를 기반으로 연관 문서를 찾아내야 할 때
- **추천 시스템**: 유저가 소비한 콘텐츠의 벡터 성향을 분석하여, 시각적·의미론적으로 유사한 제품이나 미디어를 맞춤형으로 제안할 때
- **검색 증강 생성(RAG, Retrieval-Augmented Generation)**: **대규모 언어 모델(LLM)** 이 최신 정보나 기업 내부 지식을 바탕으로 거짓 정보(Hallucination) 없이 정확한 답변을 생성할 수 있도록 관련 컨텍스트(Context)를 실시간으로 검색하여 주입하는 아키텍처를 구축할 때
- **이미지/비디오 유사도 검색**: 텍스트 태그 없이 오직 업로드된 이미지나 비디오 파일의 시각적 특징을 비교하여 유사한 미디어를 찾아내야 할 때
- **이상 탐지(Anomaly Detection)**: 고차원 금융 데이터나 보안 로그의 패턴을 분석하여 평소의 벡터 군집(Cluster) 범위를 벗어나는 이상치(Outlier)를 조기에 식별할 때
- **중복 제거(Deduplication)**: 띄어쓰기나 어미가 조금씩 달라 정밀 매칭으로는 잡히지 않는 거의 유사한 중복 문서 및 가짜 뉴스 콘텐츠를 찾아내어 정제할 때
