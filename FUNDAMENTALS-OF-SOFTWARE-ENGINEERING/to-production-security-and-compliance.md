# 보안과 규제 준수

## 1. 보안 필수 사항(Security Essentials)

- 소프트웨어 보안은 프로젝트 막바지에 단 한 번 수행하고 끝내는 체크리스트가 아니라, 시스템의 라이프사이클 전체에 걸쳐 끊임없는 아키텍처적 관심과 정기적인 패치 주의가 필요한 지속적인 프로세스(Continuous Process)다.

### 1.1. HTTPS를 통한 전송 계층 통신 보호

- 프로덕션을 오고 가는 모든 웹 트래픽은 평문 전송을 철저히 금지하고, 반드시 **HTTPS** 프로콜을 강제하여 네트워크 전송 계층에 **TLS(Transport Layer Security)** 암호화 터널을 결합해야 한다.
- 이를 통해 클라이언트와 웹 서버 사이의 데이터 패킷을 강력하게 암호화함으로써, 퍼블릭 와이파이 환경 등에서 상존하는 해커의 중간자 공격(MitM, Man-in-the-Middle)을 차단하고 데이터 변조 및 패킷 탈취 리스크를 무력화해야 한다.

### 1.2. 인증 및 인가 권장 지침

- **인증(Authentication)** 은 시스템에 접근을 요청한 사용자가 "정말 그 본인이 맞는지" 신원을 명확히 검증하는 신뢰 프로세스이며, **인가(Authorization)** 는 신원 인증이 완료된 유저가 시스템의 "어떤 구체적인 리소스와 비즈니스 기능 권한에 접근할 수 있는지" 세부 권한 영역을 통제하는 행위다.
- 보안 구멍(Vulnerability)이 뚫리기 가장 쉬운 영역이므로, 엔지니어가 로우 레벨의 필터(Filter)나 세션 검증 가내수공업 로직을 직접 바닥부터 구현하는 무모한 행위를 지양해야 한다. 대신 수많은 글로벌 보안 오딧(Audit)을 거치며 검증된 스프링 생태계의 표준 프레임워크인 **스프링 시큐리티(Spring Security)** 나 클라우드 표준 자격 증명 솔루션을 도입하여 아키텍처 결함을 차단해야 한다.
- 데이터베이스에 적재되는 사용자의 비밀번호는 단방향 해시 알고리즘을 사용해 암호화해야 한다. 단순 해시 처리는 해커의 사전 대입 테이블인 레인보우 테이블 공격(Rainbow Table Attack)에 쉽게 파쇄되므로, 해싱 전 문자열에 무작위 랜덤 바이트 소금을 치는 **솔트(Salt)** 메커니즘이 내장된 `BCrypt` 또는 `Argon2` 표준 암호화 인코더를 필수로 사용해야 한다.
- 해커의 무차별 대입 공격(Brute-force Attack)을 방어하기 위해, 동일한 계정이나 IP에서 로그인 인증 실패 횟수가 누적될 경우 다음 로그인 시도까지 대기해야 하는 백오프(Back-off) 지연 시간을 **지수적(Exponential)** 으로 가파르게 증가시키거나 계정을 일시 잠금(Lockout) 처리하는 레이트 리미터(Rate Limiter)를 구현해야 한다.
- 인증 완료 후 발급되는 세션 토큰이나 **JWT(JSON Web Token)** 같은 자격 증명 수단은 탈취 및 하이재킹 위험을 방지하기 위해 유효 기간을 극도로 짧게 가져가야 하며, 브라우저 저장 시 자바스크립트 코드로 접근이 불가능하도록 `HttpOnly` 및 `Secure` 쿠키 플래그 설정을 강제하여 **XSS(Cross-Site Scripting)** 공격을 방어해야 한다.
- 비밀번호 인증이라는 단일 장벽의 한계를 넘어서기 위해, 금융 거래나 개인정보 변경 같은 고위험 핵심 비즈니스 도메인 영역에는 유저의 스마트폰 일회용 OTP나 이메일 인증 코드를 추가로 요구하는 다중 요소 2차 인증(**MFA, Multi-Factor Authentication**) 시스템을 다중 레이어로 탑재해야 한다.

## 2. 사용자 데이터 보호 및 마스킹

- 유저의 이름, 전화번호, 이메일 주소, 자택 주소와 같은 **개인식별정보(PII, Personally Identifiable Information)** 데이터는 법적 개인정보 보호법(개인정보보호법, GDPR 등)의 엄격한 규제를 받는다.
- 운영 환경 내부 시스템 화면이나 CS 관리자 도구 백오피스 페이지상에서 이러한 PII 정보가 상담원이나 내부 직원들의 화면에 날것 그대로 전면 노출되는 구조를 원천 차단해야 한다. 이름의 가운데 글자를 별표 처리하거나(`홍*동`), 전화번호 뒷자리를 마스킹 처리하는(`010-****-5678`) 다각적인 **데이터 마스킹(Data Masking)** 변환 필터 레이어를 뷰 엔드포인트 및 API 전송 직전 단계에 자동화 결합해야 한다.

## 3. 암호화(Encryption): 시스템 최후의 방어선

- 설령 해커에게 데이터베이스 서버의 물리 디스크 권한이 통째로 털리는 최악의 인프라 침해 사태가 발생하더라도, 유저의 주민등록번호, 계좌 정보 등 핵심 민감 자산 데이터만큼은 내용을 절대 열어볼 수 없도록 만드는 가상 최후의 방어선이 바로 데이터 평문 저장 금지 원칙이다.
- 자바 생태계에서는 JDK 내부의 표준 고성능 암호화 아키텍처 패키지(`javax.crypto`)를 통해 업계 표준 대칭키 알고리즘인 **AES(Advanced Encryption Standard)** 암호화 모듈을 손쉽게 구현할 수 있도록 지원한다.

## 4. 규제 준수(Compliance) 요구 사항

- **소프트웨어 규제 준수(Software Compliance)** 란 애플리케이션이 서비스되는 국가 및 산업군에서 강제하는 특정 법적 규정, 보안 표준, 거버넌스 법규(예: 국내 개인정보보호법, 정보통신망법, 글로벌 GDPR, 금융권 망분리 규제 등)를 시스템 아키텍처 레벨에서 철저히 준수하는 것을 의미한다. 이를 위반할 경우 막대한 과징금 부과는 물론, 비즈니스 연속성에 치명적인 타격을 입을 수 있다.

### 4.1. 안전한 시스템 구축을 위한 규제 준수 모범 관행

- **설계 단계부터의 개인정보 보호(Privacy by Design)**:
  - 개발이 끝난 후 보안 취약점을 땜질하듯 고치는 것이 아니라, 아키텍처 설계 첫 단계부터 데이터 보호 및 프라이버시 원칙을 시스템 내부에 핵심 메커니즘으로 내재화(Embed)해야 한다.
- **안전한 코딩 표준(Secure Coding Standards) 적용**:
  - OWASP Top 10, KISA 소프트웨어 개발 보안 가이드 등을 기반으로 한 웹 취약점 방어 표준을 일상적인 개발 스프린트에 적용해야 한다.
  - 정적 코드 분석 도구(SonarQube 등)를 도입하여 코드 레벨의 보안 결함을 빌드 타임에 자동으로 필터링하는 파이프라인 구성이 권장된다.
- **종단간 데이터 암호화(End-to-End Encryption) 구현**:
  - 인프라를 통과하는 데이터의 전 생애주기 동안 유실과 탈취를 막기 위해 전송 중(In-Transit) 데이터와 저장 시(At-Rest) 데이터 모두에 엄격한 암호화 알고리즘을 적용해야 한다.
  - **전송 중 데이터**: 외부 인터넷망은 물론, 내부 프라이빗 서브넷 간의 모든 마이크로서비스 및 시스템 상호 통신 파이프라인에도 평문 전송을 금지하고 고도화된 TLS 기반의 **HTTPS** 프로토콜 통신을 강제해야 한다.
  - **저장 시 데이터**: 데이터베이스 파일 시스템이나 클라우드 오브젝트 스토리지에 적재되는 모든 민감 개인식별정보(PII) 및 금융 자산을 암호화하여 영속화해야 한다.
  - **메모리 런타임 보안**: 데이터베이스에 안전하게 암호화되어 있던 데이터를 서버 힙(Heap) 메모리로 읽어 들여 복호화하는 찰나의 순간에 발생할 수 있는 데이터 노출 빈틈(Memory Dump 공격 등)을 경계해야 한다.
  - 자바 생태계에서는 민감 데이터를 처리한 후 가비지 컬렉터(GC)가 수거하기 전까지 문자열 풀(String Pool)에 평문이 잔존할 수 있는 `String` 대신, 사용 직후 값을 임의로 지워버릴 수 있는 `char[]` 배열이나 가변 버퍼를 활용해 메모리 상의 흔적을 수동으로 소거하는 로우 레벨 최적화를 고려해야 한다.
- **데이터 주권 및 권리 요구 대응 구조화**:
  - 사용자의 "잊힐 권리(Right to be Forgotten)"에 따른 회원 탈퇴 시 개인정보 즉시 파기 및 분리 보관 처리 메커니즘을 구축해야 한다. 또한 사용자가 자신의 데이터 제공 내역을 요구할 때 즉각 안전하게 추출해 줄 수 있는 **데이터 다운로드/내보내기(Export)** 아키텍처 파이프라인을 사전에 명확히 설계해 두어야 한다.
- **민감 정보 상호작용의 감사 추적(Audit Trail) 문서화**:
  - 내부 관리자나 시스템 워커가 사용자의 민감한 개인정보를 조회, 수정, 삭제하는 모든 행위(Who, When, What, Why)를 위변조가 불가능한 독립된 보안 감사 로그(Audit Log) 저장소에 실시간으로 기록하고 아카이빙해야 한다.
