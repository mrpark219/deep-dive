# 사람을 위한 코드 작성법

## 1. Writing Code: 코드 쓰기

- 단순히 코드를 타이핑하여 눈앞의 비즈니스 문제를 해결할 수 있다고 해서, 그것이 '코드를 무조건 작성해야 한다'는 당위성을 의미하지는 않는다.
- AI 에이전트와 생성형 코딩 도구가 대중화되면서 엔지니어의 직무는 직접 바닥부터 코딩하는 영역에서 **AI가 생산한 코드를 검토, 정제, 버그 수정 및 교정하는 방향**으로 급격히 진화하고 있다.
- 역설적이게도 AI가 쏟아낸 코드를 올바르게 검증하고 결함을 찾아내려면 **엔지니어 스스로가 훌륭하고 깨끗한 코드를 작성할 수 있는 내공(Clean Code Skill)** 을 갖추고 있어야 한다.
- 코드의 품질을 유지하고 팀의 코드 역량을 상향 평준화하기 위한 실천 방안은 다음과 같다.
  - **테스트 코드 작성**: 테스트는 돈으로 살 수 있는 최고의 살아있는 문서이자 안정 장치다.
  - **코드 리뷰 활성화**: 팀 내에서 오류가 발생하기 쉬운 패턴을 차단하고, 지나치게 기교를 부려 가독성을 해친 코드를 정제하는 강력한 거버넌스 역할을 한다.
- 코드는 궁극적으로 **컴퓨터가 아닌 사람이 읽고 이해하도록 작성**되어야 한다.
- 탁월한 엔지니어는 화려한 테크닉보다 읽는 이의 인지 부하를 줄이는 **간결함, 명확한 의도 전달, 그리고 체계적인 구조화**에 집중하며, 이는 소프트웨어 프로젝트의 장기적인 성공과 직결된다.

## 2. 바퀴 재발명 금지 (Don't Reinvent The Wheel)

- 새로운 코드를 줄줄이 써 내려가기 전에, 풀고자 하는 문제가 이미 생태계 내에서 해결된 것은 아닌지 반드시 검토해야 한다. 5분의 영리한 검색이 수 시간, 수일의 무의미한 삽질을 막아준다.
- 자바를 비롯한 최신 프로그래밍 언어와 프레임워크는 끊임없이 진화하며 유용한 빌트인(Built-in) 기능을 추가하고 있으므로, 기술 트렌드와 신규 스펙을 주기적으로 학습하는 것은 엔지니어의 의무다.

### 2.1. 의사결정 트리 (Decision Tree)

- 새로운 로직을 구현하기 전, 아래의 질문을 차례대로 던지며 검증 단계를 거쳐야 한다.
  - 현재 코드 어딘가에 이 작업이 이미 구현되었는가?
  - JDK가 이미 이 작업을 수행하는가?
  - 이 문제를 해결하는 스프링 프레임워크나 스프링 부트 프로젝트가 있는가?
  - 구글 구아바에 해결책이 존재하는가?
  - 아파치 커너즈에 해결책이 존재하는가?
  - 프로젝트에서 이미 사용하는 라이브러리 중 이 문제를 해결한느 것이 있는가?
  - 이 문제를 해결하는 오픈소스 라이브러리가 있는가?

## 3. 좋은 코드란 무엇인가

- 좋은 코드와 나쁜 코드를 구별하는 기준은 주관적일 수 있지만, 엔지니어로서 경력이 쌓이면 코드에서 직관적으로 번지는 **'코드 냄새(Code Smell)'** 를 감지할 수 있게 된다.
- 주관성을 배제하고 객관적인 소프트웨어 품질을 유지하기 위해 다양한 **정적 분석 도구 및 지표(Metrics)** 를 활용할 수 있다.

### 3.1. 소프트웨어 품질 지표와 도구

- **순환 복잡도(Cyclomatic Complexity)**: 소스 코드 내부에서 제어 흐름에 따라 발생할 수 있는 독립적인 실행 경로의 수를 측정한다. 분기점(if, loop 등)이 많아 경로가 늘어날수록 복잡도가 증가하며, 이는 강력한 리팩터링 대상임을 시사한다.
- **정적 분석 도구(Static Analysis Tools)**: PMD, 소나큐브(SonarQube), Checkstyle 등을 빌드 파이프라인에 통합하면 특정 유형의 잠재적 버그와 안 좋은 코딩 관행(Anti-pattern)이 코드베이스에 스며드는 것을 원천 차단할 수 있다.
- **호손 효과(Hawthorne Effect)의 긍정적 활용**: 사람은 관찰받고 있다고 느낄 때 행동을 수정하는 경향이 있다. 팀 내 테스트 커버리지(Code Coverage) 통계를 대시보드 등에 투명하게 게시하면 자연스럽게 테스트 코드를 더 촘촘히 작성하는 문화를 유도할 수 있다.

### 3.2. 지표 오남용 경계

- 무언가를 정량적으로 측정할 수 있다고 해서 그것이 무조건 의미 있는 통찰을 제공하는 것은 아니다. 모든 지표는 반드시 **비즈니스 맥락과 아키텍처 구조를 고려하여 해석**해야 한다.
- 예를 들어, 특정 클래스에 의존하는 결합도(Coupling) 지표가 지나치게 높다면 일반적으로 리팩터링 대상이다.
  - 그러나 해당 객체가 하위 시스템의 복잡성을 감추고 단일 창구를 제공하는 **퍼사드 패턴(Facade Pattern)** 으로 설계된 경우라면, 높은 결합도는 결함이 아니라 의도된 설계 구조다.
- 따라서 지표의 단편적인 숫자에 매몰되지 말고, **프로젝트의 장기적 목표와 연결하여 전체적인 지표의 추세(Trend)** 가 양호한 방향으로 발전하고 있는지 관찰하는 눈을 길러야 한다.

## 4. 간결함의 가치

- 대부분의 경우 코드의 물리적인 규모는 작을수록 좋다. 시스템을 구성하는 코드양이 적을수록 엔지니어가 머릿속에 담아야 할 인지 부하도 정비례하여 줄어든다.
- 대규모 모놀리식 레거시 시스템은 수백 페이지에 달하는 온보딩 지침서와 주/월 단위로 측정되는 복잡한 빌드·배포 절차를 가지곤 한다. 이로 인해 신규 합류한 개발자가 코드를 파악하고 첫 기여를 하기까지 수 주에서 수 개월이 소요되기도 한다.
- 프로젝트가 한 사람이 모든 코드를 이해할 수 없는 임계점에 도달하면서, 이를 해결하기 위해 시스템을 작고 명확한 단위로 쪼개는 **마이크로서비스 아키텍처(MSA)** 와 **FaaS(Function-as-a-Service)** 패턴이 부상하게 되었다. 코드 규모를 작게 유지할수록 생산성 확보와 시스템 파악이 압도적으로 빨라진다.

### 4.1. 컴퓨터 과학의 제0법칙: 고응집과 저결합

- 소프트웨어 엔지니어링에서 복잡한 코드를 통제하기 위해 사용하는 수많은 기법은 결국 **높은 응집도(High Cohesion)와 낮은 결합도(Low Coupling)** 라는 컴퓨터 과학의 대원칙으로 귀결된다.
- **응집도(Cohesion)**: 모듈 내의 구성요소들이 얼마나 밀접하게 관련되어 있는지를 나타내는 척도다. 높은 응집도는 **본질적으로 유사한 책임을 가진 기능들이 한곳에 긴밀하게 모여 있음**을 뜻한다.
- **결합도(Coupling)**: 서로 다른 모듈이나 루틴이 서로에게 의존하는 정도를 말한다. 결합도가 높은 코드는 한 곳을 수정했을 때 전혀 연관 없어 보이는 다른 시스템 영역에 예기치 못한 사이드 이펙트(Side Effect)를 유발하므로 유지보수를 극도로 어렵게 만든다.
- 높은 응집도와 낮은 결합도를 유지하면 코드의 가독성이 비약적으로 향상되며, 시스템을 안전하게 변경하고 발전시킬 수 있다. 거시적 관점에서 **마이크로서비스 아키텍처는 이 법칙을 서비스 수준으로 확장하여 구현한 결과물**이다.

### 4.2. 상용 코드(Boilerplate Code) 주의

- IDE나 프레임워크가 자동으로 생성해 주는 상투적인 반복 코드(Boilerplate)일지라도 무분별한 양산은 경계해야 한다. 직접 타이핑하지 않았더라도 그 코드는 결국 프로젝트의 수명 주기 내내 팀이 안고 가야 할 부채가 되기 때문이다.
- 클래스의 길이는 모니터 화면에서 몇 페이지 이내로 짧게 유지되어야 한다. 끝없는 스크롤이 필요하다면 해당 클래스가 너무 많은 책임을 지고 있다는 위험 신호다.

### 4.3. 상속보다는 구성 (Composition over Inheritance)

- 다형성을 지원하는 객체지향 언어에서 상속(Inheritance)은 강력한 도구이며 도메인 세계에 명확한 'is-a' 관계가 존재하지만, 실무에서는 자주 남용된다.
- 많은 개발자가 단지 코드를 재사용하겠다는 목적으로 상속을 남용하곤 한다. 그러나 **재사용은 올바른 객체 설계의 부수적인 이점일 뿐, 상속의 본질적인 목적이 아니다**.
- 예를 들어 `차량(Vehicle)` 클래스가 있고 이를 상속받는 `트럭`과 `전기차`가 있다고 가정하자. 만약 상위 클래스가 내연기관 중심의 속성을 가지고 있다면, 엔진이 없는 전기차는 상속 관계로 인해 모순적인 구조를 갖게 된다.
- 상속은 컴파일 시점에 관계가 결합되어 변경에 취약하지만, **구성(Composition)** 은 런타임에 유연하게 관계를 바꿀 수 있으므로 설계의 유연성을 위해 상속보다 구성을 우선시하는 아키텍처적 접근이 필요하다.

### 4.4. 짧은 메서드 지향

- 가능한 한 코드 줄 수가 **한 자리(10줄 미만)** 인 짧은 메서드를 작성하는 것을 지향해야 한다.
- 개별 메서드는 **단 하나의 책임만 수행(Single Responsibility Principle)** 해야 하며, 그 일을 완벽히 처리한 뒤 다른 메서드들과 유기적으로 협력하여 거시적인 비즈니스 목표를 달성해야 한다.
- 메서드의 이름에 `and`, `or` 같은 **접속사가 포함되어 있다면 해당 메서드가 너무 많은 일을 분기하거나 병렬로 처리하고 있다는 명백한 신호**다.
- 메서드 명명 시 화려한 기교를 부리지 말고 의도를 명확하고 간결하게 서술해야 한다. 만약 이름을 짓기가 너무 어렵다면, 그 메서드가 과도한 책임을 떠안고 있는 것은 아닌지 의심하고 기능을 잘게 쪼개야 한다. 코드양이 아주 적더라도 조직적인 중복을 제거하고 극도의 단순함을 유지하는 것이 휴먼 에러를 방지하는 지름길이다.

## 5. 독자를 위한 코드 작성

- 코드에 항상 황금률을 적용해야 한다. 우리가 작성한 코드를 넘겨받아 유지보수할 미래의 개발자(혹은 몇 달 뒤의 나)의 삶을 개선하는 방향으로 코드를 작성해야 한다.
- 코드를 작성할 때는 **의도(What)와 구현(How)을 철저히 분리**해야 한다.
- 동료가 메서드 본문의 세부 구현을 일일이 조사하지 않더라도, **메서드 이름만 읽으면 이 코드가 무엇을 하려는지(의도) 바로 파악할 수 있어야 한다**. 이 원칙을 극한으로 추구하다 보면 자연스럽게 단 몇 줄, 심지어 단 한 줄짜리 명확한 함수들이 도출된다.
- **신문 기사 스타일의 코드 배치**: 잘 쓰인 신문 기사는 헤드라인으로 시선을 끌고, 서두 문단에서 핵심 요약을 제공한 뒤, 뒤로 갈수록 세부적인 배경 설명이 이어진다. 코드 역시 이 구조를 따라야 한다. 클래스 이름은 기사의 헤드라인이며, 상단의 고수준 메서드 서명을 훑는 것만으로도 시스템의 요지를 파악할 수 있어야 한다. 세부 구현을 알고 싶을 때만 하단의 저수준 함수 호출부로 탐색해 내려가도록 유도해야 한다.

## 6. 코드 주석의 문제

- 모든 언어는 개발자 간의 소통을 위한 주석(Comment) 기능을 제공하지만, **코드에 주석이 많다는 것은 역설적으로 그 코드가 나쁜 코드(Code Smell)임을 증명하는 신호**다.
- 주석은 코드의 비즈니스 로직을 말로 한 번 더 반복하기 때문에 **DRY(Don't Repeat Yourself, 중복 배제) 원칙을 정면으로 위반**한다. 코드가 수정될 때 주석이 함께 최신화되지 않으면, 주석은 거짓말을 하는 위험한 부채로 변질된다.
- 선택한 언어의 미묘하고 트릭 가득한 마법(Magic) 같은 기능을 남용하여 주석으로 변명하기보다, 명확한 메서드명과 변수명을 통해 코드 자체로 설명하는 것이 훨씬 낫다.
- 만약 도저히 코드를 더 단순화할 수 없어 주석을 남겨야만 한다면, '무엇을(What)' 했는지가 아니라 **'왜(Why)' 이런 제약 조건이나 비즈니스 배경 때문에 이렇게 구현했는지**에 초점을 맞추어 작성해야 한다.

## 7. 테스트를 문서로 사용

- 주석은 코드를 문서화하기에 적절한 수단이 아니므로, 대신 **테스트 코드를 작성**해야 한다.
- README나 설계 명세서는 작성과 동시에 코드와 멀어지며 낡아가지만, **테스트 코드는 애플리케이션과 함께 진화하는 실행 가능한 명세서(Living Documentation)** 다.
- 코드를 작성할 때 테스트를 유기적으로 동반하면 과감한 리팩터링이 가능해지며, 코드 품질에 대한 확신이 높아진다. 테스트는 후임 개발자에게 가장 안전하고 정확한 길잡이가 된다.
- 마이크로서비스 환경에서는 **소비자 주도 계약(CDC, Consumer-Driven Contract)** 테스트 패턴을 활용하여 서비스 간의 명세를 전달할 수 있다. 하류(Downstream) 시스템의 요구사항을 담은 계약을 위반하지 않는 한, 상류(Upstream) 시스템을 실수로 망가뜨릴까 봐 두려워할 필요 없이 코드를 주도적으로 진화시킬 수 있다. 자바 생태계에서는 대표적으로 **스프링 클라우드 컨트랙트(Spring Cloud Contract)** 등의 도구를 활용한다.

## 8. 멋부린 코드 금지

- 우리가 다루는 비즈니스 도메인은 그 자체로 이미 복잡하다.
- **본질적 복잡성(Essential Complexity)**: 비즈니스 규칙의 미묘한 차이, 커뮤니케이션 비용 등 소프트웨어 자체에 내재된 복잡성으로, 공학적으로 제거할 수 없는 영역이다. 엔지니어는 이 본질적 복잡성을 통제하는 대가로 가치를 인정받는다.
- **부수적 복잡성(Accidental Complexity)**: 과도한 오버 엔지니어링, 불필요한 도구 도입, 요란하고 기교 섞인 기술 사용 등 개발자가 필요 이상으로 일을 어렵게 만들어 스스로 심화시키는 복잡성이다.
- 해결책에 확신이 서지 않을 때는 무조건 **단순함** 을 유지해야 한다. 내가 명확히 이해하는 기교 가득한 문법일지라도 팀 전체가 공유하지 못한다면 에러를 유발하는 시한폭탄이 된다. 이러한 제약을 팀 내 **코딩 표준(Convention)** 에 엄격히 반영하는 것을 주저하지 말아야 한다.

## 9. 코드 리뷰 피라미드와 올바른 접근법

- 코드 리뷰는 구체적인 구현 검증을 넘어 팀의 지식을 사회화하고, 함께 학습하며, 경험을 공유하는 가장 훌륭한 수단이다.
- 효율적인 코드 리뷰를 위해서는 **코드 리뷰 피라미드(Code Review Pyramid)** 구조를 이해하고 노력을 적절한 레이어에 분배해야 한다.
  - **자동화 레이어**: 서식, 린팅, 정적 스타일 문제 등 피라미드 하단의 요소들은 인간의 에너지를 쏟지 말고 소나큐브, 린터 등 **자동화 툴에 전적으로 위임**한다.
  - **인간의 리뷰 가치**: 엔지니어의 시간과 노력을 쏟아야 할 곳은 피라미드 상단이다. '모델이 비즈니스 맥락에 맞게 올바르게 설계되었는가?', '잘못된 추상화를 선택하지 않았는가?', '오류가 발생하기 쉬운 형식을 사용했는가?', '로그와 지표가 적절히 심어졌는가?'와 같은 공학적 설계 검증에 집중해야 한다.
- **커뮤니케이션 태도**: 코드 리뷰 시 무엇보다 **친절함과 공감**이 선행되어야 한다. 텍스트 기반 댓글에는 말의 어조나 몸짓이 결여되어 있어 비꼬는 의도가 없더라도 상대방에게 공격적으로 받아들여질 수 있다. 목표는 기술적 우위를 자랑하는 것이 아니라 **함께 코드를 개선하는 것**임을 잊지 말아야 한다.

## 10. 형식적인 피드백 탈피와 신뢰 구축

- 단순히 풀 리퀘스트(PR)에 승인 도장(LGTM: Looks Good To Me)만 찍거나, 텍스트 댓글로 소모적인 논쟁만 벌이는 형식적인 코드 리뷰는 팀의 결속력을 해친다.
- 코드 리뷰의 진정한 목적은 관용구 오용이나 가치 없는 안티패턴을 피하고, 팀원들이 자유롭게 질문할 수 있는 **건설적인 피드백 환경과 집단적 코드 소유권(Collective Code Ownership)** 을 확립하는 것이다.
- 온라인 서면 검토 중 의견 대립이 심화되거나 의도가 꼬일 위험이 있다면, 지체 없이 **개발자와 직접 만나거나 화상 통화로 대면 소통**을 진행하여 오해를 풀고 신뢰를 구축해야 한다. 완벽한 사람은 없으며, 리뷰는 누군가를 창피 주는 자리가 아니라 배울 기회이자 가르칠 기회다.

## 11. 성장을 가속화하는 새로운 언어 학습

- 코드를 작성하는 역량을 비약적으로 높이고 싶다면, 역설적으로 **새로운 프로그래밍 패러다임과 언어를 적극적으로 학습**해야 한다.
- 자바/스프링 진영에 익숙한 엔지니어일지라도 완전히 다른 패러다임을 가진 언어(예: 함수형 언어, 시스템 언어 등)를 공부하면, 문제를 바라보는 시각이 넓어지고 기존 언어를 활용하는 기법 역시 고도화된다.
- 프로그래밍 언어는 비즈니스 문제를 해결하기 위한 도구일 뿐이므로, 특정 도구에 편협하게 갇히는 것을 경계해야 한다. 관련 기술 블로그, 팟캐스트, 최신 컨퍼런스 영상(콘텐츠)을 릴리즈마다 챙겨보고, 기술 커뮤니티나 사용자 모임에 참여하여 다른 엔지니어들과 선택지를 탐구하고 시야를 넓히는 데 시간을 투자하는 것이 기본 소양이다.
