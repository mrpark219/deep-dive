# 데이터 질의와 성능 관리

## 1. 효율적인 질의문 작성법

- 개발 초기에는 작성한 SQL 질의문(Query)을 통해 원하는 결과 집합을 올바르게 뽑아내는 것만으로도 만족스러울 수 있다.
- 하지만 기능을 완성한 후에는 반드시 "이 질의문의 성능과 처리 속도를 어떻게 더 개선할 수 있을까?"라는 엔지니어링 관점의 질문을 던져야 한다.
- 효율적인 질의문을 작성하는 능력은 백엔드 애플리케이션의 런타임 성능에 가장 직관적이고 지대한 영향을 미치는 필수적인 기본 기술이다.
- 최적화되지 않은 대량의 레거시 질의문은 데이터베이스 서버의 과도한 리소스(CPU, I/O)를 소비하고 시스템 전반에 치명적인 병목 현상을 일으키며, 결과적으로 느리고 실망스러운 사용자 경험(UX)으로 이어진다.

### 1.1. 기본 질의 최적화

- 가장 기본적이면서도 효과적인 최적화 방법은 테이블의 모든 필드를 무분별하게 긁어오는 `SELECT *` 구문을 지양하고, 비즈니스 로직에 실제 필요한 특정 컬럼(Field)들만 명시하여 조회하는 것이다.
- 필요한 열만 선택적으로 프로젝션(Projection)하고 대량 데이터의 경우 `LIMIT` 등을 통해 결과 레코드 수를 엄격히 제한하면, 데이터베이스 엔진의 I/O 부담과 네트워크를 통해 애플리케이션 서버로 전송되는 데이터 페이로드(Payload) 양을 획기적으로 줄일 수 있다.

### 1.2. 사전 준비 구문(Prepared Statement)

```java
String sql = "SELECT id, name, email FROM users WHERE email = ?";

PreparedStatement pstmt = conn.prepareStatement(sql);
pstmt.setString(1, userEmail);

ResultSet rs = pstmt.executeQuery();
```

- **사전 준비 구문(Prepared Statement)** 은 애플리케이션 구동 시점에 사전에 컴파일된 SQL 구문으로, 실행 시점에 동적인 데이터 매개변수(Parameter) 값만 안전하게 바인딩하여 실행하는 방식이다. 이를 통해 시스템 성능을 크게 높이고 **SQL 인젝션(SQL Injection)** 공격을 원천 차단한다.
- 사전 준비 구문을 사용하면 데이터베이스 엔진은 최초 호출 시 질의문을 미리 구문 분석(Parsing)하고 최적화하여 실행 계획을 수립한 뒤, 나중에 매개변수만 바뀌어 들어오는 동일한 구조의 호출에 대해 해당 **실행 계획(Execution Plan)** 을 그대로 재사용한다.
- 이러한 질의 계획 캐시(Query Plan Cache) 메커니즘을 활용하면, 반복적으로 자주 실행되는 핵심 질의의 실행 시간을 최소 20%에서 50%까지 크게 단축시킬 수 있다.
- 성능적 이점 외에도, 사전 준비 구문은 유저 입력값을 플레이스홀더(`?`)에 매핑하는 과정에서 매개변수 이스케이프(Escape) 처리를 데이터베이스 드라이버 레벨에서 자동으로 수행하므로, 특수문자 입력에 따른 SQL 인젝션 공격으로부터 애플리케이션을 안전하게 보호한다.

### 1.3. 색인(Index) 관리

- 데이터베이스 **색인(Index)** 은 특정 컬럼의 값과 해당 레코드가 디스크상에 실제로 저장된 물리적 위치를 직접 연결하는 고속 별도 조회 테이블(주로 B-Tree 구조)을 생성하여, 느린 전체 테이블 스캔(Full Table Scan)을 즉각적인 포인트 조회(Index Seek)로 전환하는 기술이다.
- 적절한 색인이 걸려있지 않다면, 데이터베이스 엔진은 단 하나의 레코드를 찾기 위해 수백만 건의 전체 테이블 데이터를 처음부터 끝까지 차례대로 읽어 내려가는 비효율을 감수해야 한다.
- 하지만 색인 구축은 공짜가 아니다. 색인 테이블 유지를 위한 별도의 물리적인 디스크 저장 공간이 추가로 필요하며, 데이터가 생성·수정·삭제(`INSERT`, `UPDATE`, `DELETE`)될 때마다 데이터베이스가 연관된 모든 색인 트리까지 함께 실시간으로 갱신해야 하므로 전반적인 쓰기(Write) 작업 성능이 저하되는 트레이드오프가 존재한다.

#### 색인(Index) 설계 시 필수 고려 대상 목록

- `WHERE` 조건절에서 검색 및 필터링 조건으로 매우 자주 사용되는 열을 우선적으로 색인해야 한다.
- 테이블 간의 `JOIN` 결합 조건에 활용되는 외래 키(FK) 등의 열을 반드시 색인해야 한다.
- `ORDER BY`를 통한 정렬이나 `GROUP BY`를 통한 그룹화 절에 빈번하게 사용되는 열을 색인해야 정렬 연산(Sort) 부하를 방지할 수 있다.
- 다수의 열이 하나의 질의문 조건에 연쇄적으로 걸러지는 복잡한 패턴의 경우, 열의 순서를 고려한 **복합 색인(Composite Index)** 구성을 적극 검토해야 한다.
- 반면 성별이나 성공 여부처럼 고유한 값의 종류가 몇 개 되지 않는, 즉 **카디널리티(Cardinality / 기수)** 가 극도로 낮은 열은 색인을 생성해도 성능 향상 없이 쓰기 비용만 가중되므로 색인 대상에서 제외해야 한다.

### 1.4. 대용량 결과 처리

- 개발 단계에서는 로컬 환경의 적은 데이터만 다루다 보니, 실제 프로덕션 운영 중에 발생할 수 있는 거대한 양의 데이터를 간과하기 쉽다.
- 시스템의 안정성을 확보하는 가장 확실한 방법은 단일 SQL 질의에서 반환하는 결과 레코드 수를 엄격하게 제한하는 것이다.
- 데이터베이스 조회에 **페이지네이션(Pagination)** 아키텍처를 도입하면, 수백만 건의 데이터를 관리 가능한 작은 덩어리(Chunk)로 나누어 효율적으로 탐색할 수 있다.
- 페이징을 구현하는 방식에는 크게 두 가지 메커니즘이 존재하며, 각 방식의 특징은 다음과 같다.
  - **오프셋 기반 페이지네이션(Offset-based Pagination)**:
    - 조회할 페이지의 시작 행 번호(`OFFSET`)를 지정하는 전통적인 기법이다. 그러나 데이터베이스 엔진이 시작 지점 이전의 모든 레코드를 메모리에 읽어 들인 후 버리는 방식으로 동작하기 때문에, 페이지 번호가 뒤로 갈수록(Deep Page) 탐색 성능이 급격히 저하된다.
  - **커서 기반 페이지네이션(Cursor-based / Keyset-based Pagination)**:
    - 마지막으로 확인한 레코드의 고유 키(ID 등)를 기준점(Cursor)으로 삼아, 그 다음 구간의 데이터만 조건절(`WHERE`)로 정확히 타겟팅하여 가져오는 최적화 방식이다. 이 기법은 전체 데이터를 순차 탐색하지 않으므로 데이터 규모나 페이지 깊이에 상관없이 항상 일정한 고속 성능을 보장한다.
- 대용량 데이터셋에서 전체 행 수(`COUNT(*)`)를 집계하는 작업은 디스크 I/O 부하가 매우 크므로, 데이터가 자주 변경되지 않는 도메인이라면 통계 지표나 총 페이지 수를 구하기 위한 전체 행 수를 별도의 고속 캐시 레이어에 보관하여 재사용해야 한다.
- 만약 데이터의 규모가 압도적으로 크다면 비싼 카운트 쿼리를 실시간으로 실행하기보다, 데이터베이스의 통계 메타데이터를 활용하여 행 수를 대략적으로 추정(Estimation)하는 방식을 도입해야 한다.
- 사용자 인터페이스(UI) 측면에서도 전체 페이지 번호를 일일이 노출하며 잦은 카운트 쿼리를 유발하는 구조를 지양하고, 다음 페이지의 존재 여부(`Has Next`)만 1건 더 조회하여 처리하는 **무한 스크롤(Infinite Scroll)** 이나 '더 보기' 버튼 형태의 UX 패턴을 적극적으로 도입해야 한다.

## 2. 도구 및 모범 사례

### 2.1. 질의 실행 계획(Execution Plan)의 이해

- **데이터베이스 옵티마이저(Database Optimizer)** 혹은 플래너(Planner)는 선언적인 SQL 질의문을 데이터베이스가 내부적으로 처리할 수 있는 형태인 **실행 계획(Execution Plan)** 이라는 물리적인 프로그램으로 변환하는 데이터베이스 관리 시스템(DBMS)의 핵심 구성 요소다.

#### 데이터베이스 옵티마이저

- 옵티마이저는 전달받은 SQL 구문을 정밀하게 분석하고, 사용 가능한 색인(Index)의 유무, 테이블 통계 정보, 데이터 분포도 등 다양한 실행 경로를 종합적으로 고려하여 가장 효율적인 실행 계획을 생성한다.
- 옵티마이저의 작동 패러다임은 크게 다음 두 가지 유형으로 나뉜다.
  - **규칙 기반 옵티마이저(RBO, Rule-Based Optimizer)**: 사전에 정의된 엄격한 우선순위 규칙(예: "사용 가능한 색인이 있다면 조건과 상관없이 항상 우선 사용한다")만을 기계적으로 따른다.
  - **비용 기반 옵티마이저(CBO, Cost-Based Optimizer)**: 쿼리를 수행할 수 있는 다양한 실행 계획 후보군을 생성한 뒤, 테이블 통계와 시스템 I/O 자원을 바탕으로 각 계획의 계산 비용을 산출하여 가장 예상 비용이 낮은 최적의 경로를 선택한다.
- 현대의 대다수 엔터프라이즈 데이터베이스 관리 시스템은 고도화된 **비용 기반 옵티마이저(CBO)** 를 표준으로 채택하여 작동한다.

#### 질의 실행 계획 확인 및 활용하기

```sql
EXPLAIN SELECT u.username, p.bio
FROM users u JOIN profiles p ON u.id = p.user_id
WHERE u.active = true;
```

- 대부분의 데이터베이스 엔진은 작성한 SQL 질의가 내부적으로 어떻게 처리되는지 시각화해 주는 `EXPLAIN` 명령어를 기본적으로 제공한다.
- 특히 **PostgreSQL**과 **MySQL** 등의 데이터베이스는 쿼리를 실제로 구동하여 옵티마이저의 비용 예측값과 실제 런타임 수행 결과를 대조해 볼 수 있도록 추가적인 로깅 정보를 제공하는 `EXPLAIN ANALYZE` 구문도 함께 지원한다.
- 실행 계획 분석 리포트는 백엔드 엔지니어에게 다음과 같은 유용한 인프라 정보를 제공한다.
  - 다중 테이블 관계 속에서 어떤 테이블에 어떤 순서로 접근(Access)할 것인가
  - 각 테이블을 엮을 때 어떤 **조인 메커니즘(Join Mechanism)** 이 채택될 것인가(중첩 루프 조인, 조인할 대상을 정렬하여 묶는 병합 조인, 해시 테이블을 빌드하여 매칭하는 해시 조인 등)
  - 조건 절과 조인 절을 처리하기 위해 각 테이블에 생성된 어떤 색인(Index)이 실제로 사용될 것인가
  - 물리 디스크나 메모리 버퍼에서 데이터를 읽을 때 어떤 방식을 취할 것인가(전체 데이터를 처음부터 읽는 순차 스캔, 색인 트리를 타는 색인 스캔 등)
  - 시간 및 컴퓨팅 자원 측면에서 옵티마이저가 산출한 예상 비용(Cost)과 실제 소요된 물리 비용
  - `ANALYZE` 명령어 사용 시, 시스템이 예측한 결과 행 수(Rows)와 실제 반환된 행 수 간의 상호 비교 데이터
- 실행 계획을 검토할 때는 시스템 아키텍처에 병목을 일으키는 다음과 같은 주요 위험 신호(Smell)를 포착해야 한다.
  - 적절한 색인이 존재하거나 걸려 있어야 하는 조건임에도 불구하고, 이를 타지 않고 거대한 테이블을 처음부터 끝까지 읽는 **전체 테이블 스캔(Full Table Scan / Sequential Scan)** 이 발생하는 경우
  - 처리해야 하는 데이터셋의 규모와 분포도에 비해 명백히 비효율적인 조인 방식이 채택되어 리소스를 낭비하고 있는 경우
  - 쿼리 조건 절의 결함으로 인해 인덱스가 누락되거나 생성된 색인을 옵티마이저가 전혀 활용하지 못하는 경우
  - 특정 쿼리 블록에서 유독 높은 예상 비용 수치나 과도한 행 수가 집계되는 경우
  - 옵티마이저가 통계 정보 오류로 계산해 낸 결과 예측값과 실제 런타임 결과값 사이에 심각한 불일치가 관찰되는 경우 (이 경우 데이터베이스 통계 정보를 갱신하는 분석 쿼리를 수동으로 실행해 주어야 한다.)

### 2.2. 데이터베이스 모니터링 및 분석

- 인프라 레벨뿐만 아니라 백엔드 애플리케이션 관점에서의 데이터베이스 모니터링 체계를 반드시 통합해야 한다.
- 이때 시스템의 안정적인 운영을 위해 요구되는 것이 바로 애플리케이션 수준의 **관측 가능성(Observability)** 이다.
- 애플리케이션 관측 가능성은 데이터베이스 상호작용 파이프라인이 전체 애플리케이션의 상태와 런타임 성능에 어떤 영향을 미치는지 유기적으로 파악할 수 있는 강력한 통찰을 제공한다.
- 엔지니어링 관점에서 관측 가능성이란 시스템이 외부로 생성해 내는 출력(Output) 데이터에 기반하여, 복잡한 시스템 내부의 런타임 상태를 얼마나 정확하게 이해하고 추론할 수 있는지를 뜻한다.
- 자바 생태계의 **스프링 부트(Spring Boot)** 프레임워크는 자체적인 **액추에이터(Actuator)** 기능과 표준 지표 수집 라이브러리인 **마이크로미터(Micrometer)** 를 통해 높은 수준의 관측 가능성 기술 스택을 기본 제공한다.
- 관측 가능성을 구성하는 3대 핵심 개념은 다음과 같다.
  - **로그(Log)**:
    - 시스템 내부에서 일어나는 주요 비즈니스 이벤트와 예외 오류를 타임스탬프 순으로 기록하는 텍스트 데이터다.
    - 사후 문제를 정밀하게 진단할 때 유용하며, 특정 시점의 지표나 분산 요청 추적 ID와 결합하여 장애의 근본 원인을 식별하는 기반이 된다.
  - **지표(Metric)**:
    - 시스템 성능과 자원 상태를 시계열 형태의 정량적 데이터로 수집하는 수치 지표다.
    - 데이터베이스 상호작용 관점에서는 커넥션 풀 사용량(활성 커넥션 수, 유휴 커넥션 수, 연결을 얻기 위해 큐에서 대기 중인 스레드 수), SQL 질의 실행 레이턴시, 트랜잭션 유지 시간, 그리고 데이터베이스 호출 오류율 등이 포함된다.
  - **추적(Trace)**:
    - 사용자의 HTTP 요청 하나가 시작되어 애플리케이션의 여러 내부 컴포넌트, 혹은 마이크로서비스 아키텍처(MSA)의 복잡한 분산 네트워크 전체로 전파되는 유기적인 실행 경로를 따라가는 기술이다.
    - 개별 데이터베이스 호출 및 네트워크 I/O 구간에 소요된 병목 시간을 마이크로초 단위로 시각화해 주므로, 전체 시스템에서 시간 지연(Latency)이 발생하는 정확한 위치를 찾아내는 데 핵심적인 역할을 한다.

### 2.3. 복잡성과 성능의 균형 맞추기

- 데이터베이스 작업 및 영속성 레이어를 설계할 때는 애플리케이션 코드의 가독성·유지보수성과 런타임 질의 성능(Performance) 사이의 적절한 트레이드오프(Trade-off) 균형을 맞춰야 한다.
- 아키텍처적 결정을 내릴 때는 다음과 같은 다각적인 요소를 종합적으로 고려해야 한다.
  - 처리 및 적재해야 하는 데이터의 전반적인 규모
  - 비즈니스 도메인이 요구하는 실시간 성능 및 지연 시간(Latency) 임계치
  - 데이터 접근 코드를 지속적으로 유지보수하고 수정하는 데 드는 비용
  - 기술 스택에 대한 개발 팀원들의 전반적인 SQL 및 ORM 숙련도
  - 비즈니스 변경에 따른 향후 도메인 모델 스키마의 유연성 요구 정도
- 가장 이상적인 모범 사례는 생산성과 가독성을 위해 대부분의 일반적인 비즈니스 CRUD 작업에는 **ORM(JPA/Hibernate 등)** 을 표준으로 사용하고, 대규모 통계 집계나 고도화된 최적화가 필수적인 성능 크리티컬(Performance-Critical) 경로에는 원시 SQL이나 **Querydsl**, **MyBatis** 등의 SQL 매퍼 도구를 직접 혼용하여 사용하는 것이다.
