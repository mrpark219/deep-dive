# 코드 문해력 = 실무 기본 능력

## 1. Code Reading: 코드 읽기

- 대부분 코딩을 처음 배울 때는 코드를 직접 작성하는 연습에 집중하지만, 실제 현업 소프트웨어 엔지니어는 **시간의 대부분을 코드를 읽는 데 소모**한다.
- AI 에이전트나 AI가 생성한 코드를 검증 없이 그대로 붙여 넣는 '바이브 코딩(Vibe Coding)'이 등장하면서, 역설적으로 생성된 코드의 결함을 찾아내고 분석하는 **코드 문해력은 엔지니어에게 더욱 중요해질 것**이다.
- 비즈니스 환경에서는 원치 않더라도 타인이 작성한 레거시 코드를 기반으로 작업해야 하는 상황이 필연적으로 발생한다.

## 2. 기존 코드 작업의 어려움

- 기존 코드와 관련된 문제를 해결해야 할 때, 엔지니어가 극복해야 할 실질적인 장애물은 다음 **네 가지**다.
- **비즈니스 도메인의 복잡성 이해**: 해결하려는 사업적 문제를 정확히 이해해야 한다. 소프트웨어가 비즈니스 전반을 지배하면서 엔지니어가 다루어야 할 도메인의 복잡도와 정밀함은 과거와 비교할 수 없을 정도로 고도화되었다.
- **전임 개발자의 주관적 맥락 파악**: 이전에 코드를 설계한 개발자의 관점에서 문제를 바라봐야 한다. 개발자마다 선호하는 아키텍처 스타일(예: 함수형 프로그래밍), 커스텀 패턴, 혹은 생소한 라이브러리를 활용하는 성향이 다르므로 전임자의 의도를 파악하는 것이 가장 까다롭다.
- **부적절한 추상화의 해결**: 기존 코드가 적절한 추상화 수준을 갖추지 못했을 가능성이 크다. 도메인의 미묘한 차이를 포착하지 못하고 모델을 지나치게 일반화했거나, 특정 패턴을 억지로 짜 맞춘 경우 코드를 이해하기 쉽게 만들기 위해 대폭적인 **리팩터링(Refactoring)** 이 요구된다.
- **기술 부채의 연대 측정**: 고고학자처럼 세월에 따라 쌓인 기술 부채의 퇴적층을 하나씩 벗겨내야 한다. 소프트웨어 패러다임과 베스트 프랙티스는 계속 변하므로, 현재의 기준이 아닌 **코드가 작성된 당시의 언어 스펙과 프레임워크 버전의 렌즈를 통해 코드를 분석**해야 한다.
- 기존 코드를 다루는 작업은 이와 같은 기술적 난이도 외에도, 개발자 내면의 인지 편향을 자극하여 심리적 거부감을 유발하기도 한다.

## 3. 인지 편향(Cognitive Bias)의 극복

- 타인의 코드를 분석하고 개선할 때 성장을 방해하는 대표적인 인지 편향은 다음과 같다.
- **이케아 효과(IKEA Effect)**: 사람은 자신이 직접 만든 결과물에 주관적으로 더 높은 가치를 부여하는 성향이 있다. 이로 인해 많은 개발자가 '내가 짠 코드와 작업 방식이 가장 올바르다'는 확고한 독단에 빠지기 쉽다.
- **단순 노출 효과(Mere Exposure Effect)**: 인간은 기본적으로 자신에게 이미 익숙하고 자주 접한 환경에 본능적으로 끌린다. 그 결과 특정 프로그래밍 언어, 프레임워크, 라이브러리에 대해 편협한 시각을 갖게 되며 새로운 대안을 배척하는 오류를 범한다.
- 성공적인 엔지니어로 성장하고 동료와 생산적으로 협업하기 위해서는, 코드를 읽고 평가할 때 내면에서 작동하는 이러한 **인지 편향을 끊임없이 경계하고 객관성을 유지하려는 노력**이 필수적이다.

## 4. 낯선 코드 접근법

- 기존 프로젝트에 빠르게 적응하기 위해서는 팀원과 소통하는 것부터 시작하는 것이 좋다. 기본적인 프로젝트 개요 파악은 온보딩(Onboarding) 과정의 필수 요소여야 한다.
- 시간을 들여 문서를 살펴봐야 한다. 대부분의 프로젝트에는 상황을 파악하는 데 유용한 `README.md` 파일이 존재하며, 일부 프로젝트에는 간결한 개요를 제공하기 위해 만들어진 위키(Wiki)나 별도의 웹사이트가 있다.
- 프로젝트 내에 **아키텍처 결정 기록(ADR, Architecture Decision Record)** 이 존재하는지 확인해야 한다. ADR은 최근 결함이나 장애를 서둘러 처리하느라 종종 사라져 버리는 **과거의 값진 맥락과 아주 중요한 아키텍처 수립 이유**를 명확히 알려준다.
- 만약 읽고 있는 문서가 최신화되지 않고 오래되었다면, 분석한 내용을 바탕으로 문서를 직접 현행화해 보는 것이 좋다. 이는 다음 개발자를 배려하는 황금률을 실천하는 방식이다.
- 신규 온보딩 시 분석하고 문서화해야 할 핵심 질문은 다음과 같다.
  - 해당 서비스가 비즈니스적으로 처리하는 핵심 역할은 무엇인가?
  - 시스템의 내부 구조와 핵심 로직은 어떻게 동작하는가?
  - 데이터베이스, 외부 API 등 무엇에 의존하고 있는가?
  - 로컬 및 개발 환경에서 애플리케이션을 어떻게 실행시키는가?
