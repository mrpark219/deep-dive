# MSA의 이해

## 1. 리액티브 선언: 현대 애플리케이션이 갖춰야 할 바람직한 속성들

- **소프트웨어 아키텍처**는 소프트웨어의 핵심 요소와 그 요소들 간의 관계를 정의한 것이다.
- 아키텍처 설계는 단순한 기능 구현을 넘어, 성능이나 확장성 같은 **비기능 요건을 만족하는 최적의 해결책을 찾는 과정**이다.
- **리액티브 선언문(The Reactive Manifesto)** 은 급변하는 환경에 적응하기 위해 시스템이 갖춰야 할 **4가지 핵심 특성**을 강조한다.
  - **응답성(Responsive)**: 사용자에게 항상 빠르고 적절하게 신뢰성 있는 응답을 제공하는 것을 의미한다.
  - **탄력성(Resilient)**: 장애가 발생해도 시스템 전체가 마비되지 않고, **고장 난 부분을 격리해 빠르게 복구하는 능력**이다.
  - **유연성(Elastic)**: 트래픽 변화에 맞춰 자원을 동적으로 조절(Scale-out/in)하여 **항상 일정한 응답성을 유지하는 능력**이다.
  - **메시지 기반(Message Driven)**: 비동기 메시지 전달로 구성 요소 간 위치 투명성을 확보하며, **느슨한 결합(Loose Coupling)** 과 **논블로킹(Non-blocking)** 통신을 지향한다.
- 결과적으로 현대 아키텍처는 시스템이 변화와 확장에 즉각 대응할 수 있는 **유연성을 갖추는 것**을 최우선 목표로 한다.

## 2. 강한 결합에서 느슨한 결합으로의 아키텍처 변화

- **과거의 폐쇄적 아키텍처 (강한 결합)**
  - 예전에는 아키텍처 구성 요소를 **특정 기업이나 벤더(Vendor)의 제품에 전적으로 의존**하여 구축하는 경우가 많았다.
  - 품질은 벤더가 보장하지만, **특정 기술에 락인(Lock-in)** 되어 시스템 변경과 확장이 어려운 치명적인 단점이 있었다.
  - 벤더 제품을 교체하려면 의존하는 애플리케이션 대부분을 수정해야 하는 **강한 결합(Tight Coupling)** 구조였다.
- **최근의 개방형 아키텍처 (느슨한 결합)**
  - 최근 오픈소스 기술의 품질과 기능이 크게 향상되면서, 서로 다른 제품 간에도 **충분한 호환성을 제공**한다.
  - 아키텍처 설계 역시 특정 솔루션 종속에서 벗어나, 필요한 영역에 가장 적합한 기술을 자유롭게 조합하는 **개방적인 방식**으로 변화했다.
  - 이를 통해 시스템은 특정 기술이나 벤더로부터 독립적인 **느슨한 결합**을 유지하며, 최신 트렌드와 변화에 **빠르고 유연하게 대응**할 수 있다.

## 3. 마이크로서비스의 외부 아키텍처와 내부 아키텍처

- MSA 시스템의 전체 구조는 크게 **인프라(Infrastructure)**, **플랫폼(Platform)**, **애플리케이션(Application)** 영역으로 나뉜다.
  - 최하단에는 시스템 기반이 되는 물리적 또는 가상화된 **하드웨어 인프라**가 존재한다.
  - 인프라 위에는 애플리케이션을 운영하고 구동하기 위한 **플랫폼**이 올라간다.
  - 마지막으로 플랫폼 위에서 실제 비즈니스를 처리하는 **애플리케이션(서비스)** 이 구동되는 계층 구조를 가진다.

### 3.1. MSA 외부 아키텍처 (Outer Architecture)

- 인프라, 플랫폼, 애플리케이션 영역에 존재하는 **다양한 구성 요소와 그들 간의 유기적 관계를 정의**한다.
- 즉, 수많은 마이크로서비스가 안전하고 효율적으로 운영되는 **전체 환경을 설계**하는 과정이다.
- 여기에는 기본 환경뿐만 아니라 마이크로서비스를 통합 라우팅, 모니터링, 관리하기 위한 **공통 애플리케이션(API 게이트웨이, 서비스 디스커버리 등)** 환경까지 모두 포함된다.

### 3.2. MSA 내부 아키텍처 (Inner Architecture)

- 외부 운영 환경에서 한 단계 더 들어가, 실제 핵심 비즈니스가 실행되는 **각 마이크로서비스 자체의 내부 구조를 정의**한다.
- 개별 마이크로서비스가 제공하는 **API 설계, 내부 비즈니스 로직, 이벤트 처리 및 데이터 저장** 등을 응집력 있고 독립적으로 구조화하는 구체적인 방법을 다룬다.

## 4. MSA 구성 요소 및 MSA 패턴

- MSA 구성 요소와 패턴은 역할에 따라 크게 **세 가지**로 나뉜다.
  - **인프라 구성 요소**: 마이크로서비스를 물리적 또는 가상으로 지탱하는 **하부 인프라 구축**에 필요한 요소이다.
  - **플랫폼 패턴**: 인프라 위에서 수많은 마이크로서비스의 라우팅, 모니터링 등 **운영과 관리를 지원**하는 플랫폼 차원의 패턴이다.
  - **애플리케이션 패턴**: 실제 비즈니스 로직을 처리하는 **마이크로서비스 애플리케이션 자체**를 구성하는 데 필요한 패턴이다.

### 4.1. 인프라 구성 요소

#### 1. 퍼블릭 클라우드, 프라이빗 클라우드, 베어메탈(Bare Metal)

- 마이크로서비스는 특정 인프라에 종속되지 않고 **어떤 환경에서도 유연하게 구동**되어야 한다.
- 과거에는 인프라 선택이 소프트웨어 구성에 큰 제약을 주었지만, 현재는 다양한 클라우드 환경과 **유연하게 결합**할 수 있다.
- 단, 가상화 없이 물리 서버만 사용하는 **베어메탈** 환경에서는 OS와 서버 환경을 매번 직접 설정해야 하는 **오버헤드가 발생**한다.

#### 2. 가상 머신(VM)과 컨테이너(Container)

- 가상 인프라를 구축할 때는 **가상 머신**과 **컨테이너** 중 하나를 선택해야 한다.
- **가상 머신(VM)**: 하이퍼바이저(Hypervisor)를 이용해 하나의 물리 서버에서 **여러 개의 독립적인 운영체제(Guest OS)** 를 구동하는 기술이다.
- **컨테이너(Container)**: 하이퍼바이저 없이 호스트 OS 레벨에서 **가상의 격리된 실행 공간**을 만드는 기술이다.
- 두 기술의 핵심적인 차이는 **게스트 OS의 유무**이다.
- VM은 무거운 게스트 OS를 포함하므로 유지보수 오버헤드가 발생하지만, 컨테이너는 **가볍고 빠르게 동작**한다.
- 따라서 빈번하게 배포되는 마이크로서비스의 특성상, 무거운 VM보다는 **도커(Docker)와 같은 컨테이너 환경을 기본 인프라로 채택**하는 것이 훨씬 효율적이다.

#### 3. 컨테이너 오케스트레이션(Container Orchestration)

- 수많은 컨테이너를 효율적으로 관리하려면 **자동화 기술이 필수적**이다.
- 시스템 규모가 커질수록 자동 배치, 복제, 장애 복구, 오토 스케일링(Auto-scaling) 등의 복잡한 관리가 필요하며, 이를 **컨테이너 오케스트레이션**이라고 한다.
- 대표적인 도구로는 **도커 스웜(Docker Swarm)**, **아파치 메소스(Apache Mesos)**, **쿠버네티스(Kubernetes)** 가 있다.
- 특히 **쿠버네티스**는 강력한 자동화 및 확장 기능을 지원하여 현재 **업계 표준(De facto)** 으로 자리 잡았다.

#### 4. 다양한 클라우드 인프라 서비스

- AWS, 애저(Azure), GCP 등 클라우드 서비스 제공자는 개발자의 인프라 구축 부담을 줄이기 위해 **다양한 매니지드 서비스(Managed Service)** 를 제공한다.
- 대표적으로 가상 네트워크(VPC), 가상 스토리지(S3), 관계형 데이터베이스(RDS) 등을 **관리형 서비스 형태**로 쉽게 활용할 수 있다.

### 4.2. 마이크로서비스 운영과 관리를 위한 플랫폼 패턴

#### 1. 개발 지원 환경: 데브옵스(DevOps) 인프라 구성

- 수많은 마이크로서비스를 빠르게 빌드, 테스트, 배포할 수 있도록 돕는 **데브옵스(DevOps)** 환경이 필수적이다.
- **데브옵스(DevOps)**: 개발과 운영을 병행하는 조직 문화이며, 기술적으로는 고품질 소프트웨어를 빠르게 개발하도록 지원하는 자동화된 빌드, 테스트, 배포 환경을 의미한다.
- 수동 빌드 및 배포는 시간이 오래 걸려 비즈니스 민첩성을 떨어뜨리므로 자동화가 절대적으로 필요하다.
- **CI/CD(지속적 통합 및 배포)**: 자동화된 빌드와 배포 작업을 통칭하는 용어다.
  - **CI(Continuous Integration)**: 소스 코드를 자동으로 통합하고 테스트하여 결과를 기록하는 과정이다.
  - **CD(Continuous Delivery/Deployment)**: CI를 거친 실행 파일을 실제 운영 환경(Staging/Production)까지 자동으로 배포하는 방식이다.

#### 2. 빌드/배포 파이프라인 설계

- **빌드/배포 파이프라인**: 코드가 빌드되고 배포되는 동안 수행할 일련의 프로세스(테스트 등)를 하나로 연계하여 시각화된 자동화 절차로 구축하는 것이다.
- 일반적인 파이프라인은 빌드 ➡️ 단위 테스트 ➡️ 정적 분석 ➡️ 배포 과정을 거치며, 필요에 따라 UI 테스트, 통합 테스트, 배포 승인 프로세스 등을 추가할 수 있다.
- **IaC(Infrastructure as Code)**: 인프라 구성을 프로그래밍하듯 코드로 처리하는 기술이다. 소수 인원으로도 수많은 컨테이너 배포를 자동화할 수 있으며, 코드 형태라 공유와 재사용이 쉽다.
- 마이크로서비스는 각자 별도의 코드 저장소(Repository)를 가지고 독립적으로 빌드 및 배포되어야 하므로, **빌드/배포 파이프라인 역시 마이크로서비스별로 완벽히 분리하여 독립적으로 설계**해야 한다.

#### 3. 마이크로서비스 생태계와 운영 관리 요소의 탄생

- AWS 클라우드 환경, **도커(Docker)**, **넷플릭스(Netflix) OSS**, **스프링 프레임워크(Spring Framework)**, 구글 **쿠버네티스(Kubernetes)** 같은 기술들이 마이크로서비스 생태계의 비약적인 발전을 이끌었다.
- 실무 적용 과정에서 트래픽 제어, 모니터링 등 여러 문제 영역이 깊이 논의되었고 다양한 해결책이 제시되었다.

#### 4. 경험으로 획득한 지혜: 마이크로서비스 관리/운영 패턴

- 넷플릭스가 자사의 운영 경험을 바탕으로 OSS(Open Source Software)를 공개하면서, **API 게이트웨이, 서비스 디스커버리, 모니터링, 분산 추적(Tracing)** 기술이 다수의 마이크로서비스 관리를 위한 플랫폼 패턴으로 정착했다.
- 이 패턴들을 적용한 다양한 도구와 발전된 오픈소스들이 탄생했다.

#### 5. 스프링 클라우드(Spring Cloud) = 스프링 부트 + 넷플릭스 OSS

- **스프링 클라우드(Spring Cloud)**: 넷플릭스 OSS의 핵심 개념을 자바 생태계 표준인 **스프링 부트(Spring Boot)** 기반으로 쉽게 통합한 기술 모음이다.
- **스프링 클라우드 기반의 서비스 연계 핵심 흐름**:
  1. **Config 서비스 연계**: 인프라 종속을 막기 위해 외부(Git 등)에 저장된 설정 정보를 중앙 Config 서비스에서 주입받아 클라우드 인스턴스로 로딩한다.
  2. **서비스 레지스트리 등록**: 로딩과 동시에 **유레카(Eureka)** 같은 서비스 레지스트리에 자신의 서비스명과 물리적 IP 주소를 매핑하여 스스로 등록한다.
  3. **API 게이트웨이 라우팅**: 클라이언트는 개별 주소를 알 필요 없이 API 게이트웨이를 통해서만 접근하며, 게이트웨이가 부하 분산(Load Balancing)과 라우팅을 수행한다.
  4. **주소 검색**: API 게이트웨이는 요청 전달 전 서비스 레지스트리를 검색해 호출할 대상의 동적 IP 위치를 가져온다.
  5. **인증/인가**: 동시에 API 게이트웨이는 권한(Auth) 서비스와 연계하여 요청의 권한 및 인증 처리를 중앙 집중적으로 수행한다.
  6. **모니터링과 추적**: 이 복잡한 호출 흐름은 모니터링 서비스와 추적(Tracing) 서비스에 의해 실시간으로 기록되고 시각화된다.

#### 6. 서비스 레지스트리(Service Registry)와 서비스 디스커버리(Service Discovery) 패턴

- **도입 배경**: 클라우드에서는 컨테이너의 잦은 생성과 소멸로 백엔드 서비스의 **IP 주소가 동적으로 계속 변경**되므로, 클라이언트가 IP 정보를 매번 하드코딩할 수 없다.
- 클라이언트나 API 게이트웨이가 여러 마이크로서비스를 호출할 때 최적의 경로를 찾아주는 라우팅 및 로드 밸런싱 기능이 필수적인데, 이를 해결하는 것이 **서비스 디스커버리 패턴**이다.
- **동작 원리 (유레카 기반)**:
  - **등록**: 각 마이크로서비스 인스턴스가 기동될 때 자신의 이름과 IP 주소를 서비스 레지스트리에 등록한다(종료 시 자동 삭제).
  - **탐색**: 특정 서비스를 호출하면 라우터가 서비스 레지스트리를 검색해 매핑된 실제 IP를 찾아 요청을 전달한다.
- **서비스 레지스트리**는 시스템 내 모든 마이크로서비스의 주소를 알고 있는 거대한 중앙 서비스 매핑 저장소 역할을 한다.
- 업무 로직뿐만 아니라 Config, 모니터링, 추적 서비스 등 운영 관리용 기반 인프라 서비스의 주소도 모두 이곳에 보관된다.
- **참고**: 쿠버네티스 환경에서는 유레카 대신 내장 기능인 **Kube-DNS**와 **Service 객체**가 이 역할을 훌륭하게 대체한다.

#### 7. 서비스 단일 진입을 위한 API 게이트웨이 패턴(API Gateway Pattern)

- **도입 배경**: 클라이언트가 다수의 백엔드 서비스를 직접 호출하면 스파게티 네트워크처럼 호출 관계가 복잡해지므로, 이를 통제하기 위해 API 게이트웨이를 둔다.
- **단일 진입점(Single Entry Point)**: 클라이언트 접근 창구를 하나로 통일하여 효율적으로 관리한다.
- **API 게이트웨이의 핵심 역할**:
  - 클라이언트 환경(웹, 모바일 등)에 맞춘 다양한 API 조합을 제공한다.
  - **인증 및 인가(보안) 기능을 중앙 집중식으로 한 번에 처리**할 수 있다.
  - 장애가 발생한 서비스 대신 정상적인 다른 인스턴스로 요청 경로를 우회시킬 수 있다.
- **애플리케이션 레벨(L7) 라우팅**: 하드웨어(L4) 장비보다 더욱 정교하게 애플리케이션 레벨에서 흐름 제어 및 라우팅을 수행한다.
- **필터링(Filtering)**: 라우팅 전후에 필터를 두어 선행 처리, 후행 처리, 에러 처리 등을 쉽게 구현한다.
- **API 게이트웨이가 제공하는 주요 기능**:
  - 서비스 레지스트리와 연계한 **동적 라우팅 및 로드 밸런싱**
  - 권한 서비스와 연계한 **보안(인증/인가)**
  - 로그 집계 서비스와 연계한 **로깅**(API 소비자 정보, 요청/응답 데이터 등)
  - **메트릭 수집**(에러율, 응답 지연 시간 등)
  - 트레이싱 서비스와 연계한 **서비스 분산 추적**(트래킹 ID 기록)
  - 모니터링 서비스와 연계한 **장애 격리**(서킷 브레이커 패턴)
- 쿠버네티스 환경에서는 이 역할을 **Service**와 **인그레스(Ingress)** 리소스를 통해 제공한다.

#### 8. BFF 패턴(Backend For Frontend)

- **도입 배경**: 다양한 모바일 및 웹 기기를 지원하기 위해, 클라이언트 환경별로 특화된 API 조합과 처리가 필요해졌다.
- **BFF의 개념**: 거대한 단일 게이트웨이 대신, **프론트엔드 유형(웹, 모바일 등)에 맞춰 게이트웨이를 각각 분리하는 패턴**이다.
- **장점**:
  - 클라이언트 종류에 따라 응답 포맷이나 데이터를 가장 적합하게 최적화한다.
  - 화면에 필요한 API만 맞춤형으로 가볍게 조합하여 별도로 제공할 수 있다.
- **확장 구조**: 클라이언트 맞춤 처리를 하는 여러 BFF를 앞단에 두고, 그 뒤에 **통합 API 게이트웨이**를 배치하여 공통 보안 처리 등을 한 번에 통제하도록 구성할 수도 있다.

#### 9. 외부 구성 저장소 패턴(External Configuration Store Pattern)

- **도입 배경**: DB 연결 정보나 파일 스토리지 경로를 애플리케이션 코드에 직접 하드코딩하면, 설정이 바뀔 때마다 코드를 재빌드하고 배포해야 하므로 서비스가 중단되고, 설정 불일치 장애가 발생할 수 있다.
- **외부 구성 저장소 패턴**: 환경 설정 정보를 코드 내부가 아닌, **외부 백업 저장소(Config Server)에 모아 공동으로 관리**하는 방법이다.
- **12-Factor App 원칙**: 배포 환경이 계속 변하므로, **코드에서 사용하는 환경 설정 정보는 코드와 완전히 분리하여 관리해야 한다**는 핵심 원칙을 따른다.
- **분리해야 할 핵심 정보**: DB 연결 정보, 배포 환경별 호스트명, 백엔드 서버 연결 리소스 정보 등이다.
- **구현 기술**:
  - **스프링 클라우드 컨피그(Spring Cloud Config)**: 환경 정보를 외부(Git 등)로 분리하고, 애플리케이션 실행 시점에 동적으로 주입한다.
  - **쿠버네티스(Kubernetes)**: 외부 구성 저장소 역할을 **컨피그맵(ConfigMap)** 과 **시크릿(Secret)** 리소스로 훌륭하게 제공한다.

#### 10. 인증/인가 패턴(Authentication/Authorization Pattern)

- **중앙 집중식 세션 관리(Centralized Session Management)**
  - 주로 모노리스 방식에서 사용하며, 서버 메모리에 세션을 저장해 권한을 판단한다.
  - MSA 환경에서는 수평 확장(Scale-out) 시 서버 간 세션이 공유되지 않아 데이터 유실 문제가 발생한다.
  - 이를 해결하기 위해 **레디스(Redis)** 나 **멤캐시드(Memcached)** 같은 외부 인메모리 저장소를 도입하여 모든 분산 서비스가 동일한 세션을 공유하게 만든다.
- **클라이언트 토큰(Client Token)**
  - 상태 정보를 서버가 아닌 **사용자의 클라이언트(브라우저 등)에 저장**하는 무상태(Stateless) 방식이다.
  - 토큰 자체가 사용자의 신원과 권한 정보를 포함하므로, 매 요청마다 헤더에 토큰을 전송하여 각 서비스에서 즉각적으로 인가 처리를 할 수 있다.
  - 위변조를 방지하고 여러 언어에서 지원하는 범용 표준인 **JWT(JSON Web Token)** 를 대표적으로 사용한다.
- **API 게이트웨이를 사용한 클라이언트 토큰(Token with API Gateway)**
  - MSA 환경의 모든 트래픽을 통제하는 **API 게이트웨이가 외부 요청의 단일 진입점으로 추가된 구조**다.
  - 각 서비스가 인증 로직을 중복으로 구현하지 않도록 **인증 전담 서비스(Auth Service)** 를 분리하여 처리를 완전히 위임한다.
  - 클라이언트 요청 시 API 게이트웨이가 인증 서비스와 연동해 토큰 유효성을 검사하고 일괄 처리한 뒤, 안전한 요청만 내부로 라우팅한다.

#### 11. 장애 및 실패 처리를 위한 서킷 브레이커 패턴(Circuit Breaker Pattern)

- 분산 시스템에서는 한 서비스의 장애가 그 서비스를 호출하는 다른 정상 서비스까지 연쇄적으로 마비시키는 심각한 영향을 줄 수 있다.
- 장애가 발생한 서비스를 격리하여 시스템 전체로 장애가 번지는 것을 막는 강력한 방어 수단이 바로 **서킷 브레이커 패턴**이다.
- **장애 전이(Cascading Failure)의 문제점**
  - 특정 서비스에 문제가 생기면 즉각적인 에러 대신 스레드가 한참 동안 대기(Timeout) 상태에 빠지는 현상이 발생한다.
  - 사용자는 장애를 인지하지 못하고 지속해서 재요청을 하게 된다.
  - 대기하는 스레드가 쌓여 정상 서비스의 스레드 풀까지 고갈되며, 결국 **장애가 시스템 전체로 완전히 전이(Cascading Failure)** 되는 최악의 상태에 이른다.
  - 따라서 병목을 막고 자연스럽게 다른 정상 흐름으로 유도하도록 요청 흐름을 선제적으로 차단해야 한다.
- **서킷 브레이커의 작동 원리와 폴백(Fallback)**
  - 집의 누전 차단기와 똑같은 원리로 동작한다.
  - 특정 서비스 호출이 설정된 **임계값(Threshold)을 초과해 연속으로 실패**하면, 회로 차단기가 작동(Open)하여 이후 호출 시도를 대기 없이 **즉시 실패(Fast Fail)** 처리한다.
  - 차단 시 에러를 던지는 대신 우회 처리를 하는 **폴백(Fallback)** 메서드를 지정해 두면, 기본 추천 상품이나 캐시된 데이터를 반환하여 사용자가 장애를 느끼지 않게 대처할 수 있다.

#### 12. 모니터링과 분산 추적 패턴(Monitoring and Distributed Tracing Pattern)

- 서킷 브레이커를 원활히 작동시키려면 각 서비스의 상태를 실시간으로 감지하고 복잡한 호출 관계를 파악하는 **모니터링과 분산 추적 패턴이 필수적**이다.
- **모니터링**: **Resilience4j** 같은 라이브러리를 사용하여 서비스 성능과 에러율을 실시간으로 감시하고, 문제 발생 시 즉각적인 방어 조치를 취한다.
- **분산 추적**: 여러 서비스에 걸친 트랜잭션 흐름을 고유한 트래킹 ID로 묶어 파악하는 기능이다. **집킨(Zipkin)** 같은 오픈소스를 활용하면 호출 경로, 지연(Latency) 구간, 특정 장애 포인트를 쉽게 시각화하여 파악할 수 있다.

#### 13. 중앙화된 로그 집계 패턴(Centralized Log Aggregation Pattern)

- 수많은 서비스 인스턴스가 남기는 방대한 로그를 개별 서버에 직접 접속해 확인하는 것은 물리적으로 불가능하다.
- **12-Factor App의 로그 원칙**
  - 로그는 고정된 파일이 아니라 쉴 새 없이 흐르는 **이벤트 스트림(Event Stream)** 으로 취급해야 한다.
  - 로그를 보관하고 라우팅하는 메커니즘은 특정 인프라에 종속되므로, **애플리케이션은 로그 저장 과정에 절대 관여하지 않아야 한다**.
- **중앙화된 로그 집계**: 애플리케이션은 단순히 표준 출력(stdout)으로 로그를 뱉어내기만 하고, 외부 도구가 로그를 긁어모아 한곳에 저장하고 분석하는 패턴이 필요하다.
- **ELK 스택(Elastic Stack)**: 중앙집중형 로그 관리에 업계 표준처럼 쓰이는 오픈소스 솔루션 조합이다.
  - **E(Elasticsearch)**: 수집된 방대한 데이터를 아주 빠르게 검색하고 분석하는 분산형 검색 엔진이다.
  - **L(Logstash)**: 다양한 마이크로서비스에서 쏟아지는 로그를 수집하고 일정한 포맷으로 변환(필터링)하여 보관소로 전송하는 데이터 처리 파이프라인이다.
  - **K(Kibana)**: 저장된 데이터를 바탕으로 직관적인 대시보드와 UI 화면을 지원하는 시각화 도구다.
- ELK 스택을 도입하면 수많은 서비스의 로그를 **중앙에서 완벽히 통합 관리**할 수 있으며, 치명적인 에러 발생 시 즉시 알람을 보내도록 고도화할 수 있다.

#### 14. 서비스 메시 패턴(Service Mesh Pattern)

- **초창기 MSA 기술의 한계**
  - API 게이트웨이, 서비스 레지스트리 등 운영 관리용 기반 인프라를 별도로 구축해야 하는 번거로움이 있었다.
  - 운영 관리용 라이브러리를 **비즈니스 로직과 함께 코드에 탑재**해야 하므로 로직의 순수성이 떨어졌다.
  - 자바 이외의 다른 언어로 개발된 서비스에는 동일한 관리 기능을 적용하기가 매우 힘들었다.
- **서비스 메시(Service Mesh)**
  - 최근에는 서비스 탐색, 서킷 브레이커, 분산 추적 등의 인프라 기능을 **비즈니스 로직과 완전히 분리하여 네트워크 인프라 계층에서 직접 수행**하는 서비스 메시 패턴이 선호된다.
  - 마이크로서비스 간의 모든 통신을 가로채서 처리하며 앞서 언급한 운영 관리 패턴들을 포괄적으로 지원한다.
- **사이드카(Sidecar) 패턴과 이스티오(Istio)**
  - **이스티오(Istio)**: 구글이 주도하는 대표적인 서비스 메시 솔루션이다.
  - **사이드카 패턴**: 애플리케이션 컨테이너와 완전히 격리된 **별도의 프록시 컨테이너**를 동일한 파드(Pod) 내에 오토바이 사이드카처럼 함께 배포하는 방식이다.
  - 이 사이드카 프록시가 실제 애플리케이션 대신 모든 트래픽을 처리하며 라우팅, 보안, 로깅, 모니터링 등을 투명하게 제공한다.
  - 개발자는 비즈니스 로직에만 집중하고 인프라 관리는 **네트워크 계층(Sidecar)** 에 위임하여, 진정한 관심사 분리를 실현한다.
- **쿠버네티스 연동**: 서비스 메시 기능은 주로 **쿠버네티스(Kubernetes)** 환경에 탑재되어 복잡한 마이크로서비스 간 통신을 중앙에서 통제하고 시각화하는 데 사용된다.

#### 4.3. 애플리케이션 패턴

#### 1. UI 컴포넌트 패턴 또는 마이크로 프론트엔드(Micro Frontend)

- 프론트엔드 역시 백엔드 마이크로서비스처럼 **기능(업무)별로 분리하여 독립적으로 개발 및 배포**하는 아키텍처 패턴이다.
- 프레임 역할을 하는 **부모 창(Shell)** 을 통해 쪼개진 각 프론트엔드 조각들을 하나로 매끄럽게 조합하여 동작하게 만든다.
- 부모 서비스는 전체적인 레이아웃의 틀만 제공하고, 실제 각 기능의 화면 표현과 비즈니스 로직은 **개별 마이크로 프론트엔드 조각이 독립적으로 구현**한다.

#### 2. 마이크로서비스 통신 패턴

- **동기 통신 방식(Synchronous Communication)**
  - **작동 원리**: 클라이언트가 서버 측 마이크로서비스의 **REST API를 호출**할 때 주로 사용하는 기본적인 통신 방법이다. 요청을 보내면 즉각적인 응답이 올 때까지 스레드가 **대기(Blocking)** 한다.
  - **아키텍처 구성**: 다양한 클라이언트 채널 연계나 라우팅, 로드 밸런싱을 원활하게 처리하기 위해 중간에 **API 게이트웨이**를 두어 통신을 제어할 수 있다.
  - **치명적 단점 (장애 전이)**: 호출받은 백엔드 서비스에 장애가 발생하면, 호출한 서비스는 응답을 무한정 기다리거나 계속 재요청해야 한다. 이로 인해 장애가 다른 서비스로 연쇄적으로 전파되는 **장애 전이(Cascading Failure)** 가 발생할 수 있다. 이는 마이크로서비스 간의 결합도가 매우 높다는 것을 의미한다.
- **비동기 통신 방식(Asynchronous Communication)**
  - **작동 원리**: 동기 호출처럼 즉각적인 결과 응답을 기다리지 않는다. **메시지를 보낸 후 응답을 기다리지 않고 곧바로 자신의 다음 비즈니스 로직을 처리(Non-blocking)** 한다.
  - **완결성 보장과 메시지 브로커**: 결과 응답을 바로 받지 않으므로 동기식처럼 데이터 완결성을 즉시 보장할 수는 없다. 이를 보완하기 위해 주로 **아파치 카프카(Apache Kafka)** 나 **래빗엠큐(RabbitMQ)** 같은 **메시지 브로커(Message Broker)** 를 도입한다.
  - **느슨한 결합(Decoupling)**: 메시지를 보내는 **생산자(Producer)** 와 가져다 처리하는 **소비자(Consumer)** 가 직접 연결되지 않고 오직 메시지 브로커를 통해서만 간접적으로 연결된다. 동시에 동작하지 않아도 메시지 큐를 통해 통신이 성립된다.
  - **확장성과 탄력성**: 트래픽이 몰리면 메시지 브로커에 부하가 생길 수 있으나, 브로커 시스템은 손쉽게 수평 확장(Scale-out)이 가능하다. 변화가 잦은 **클라우드 환경에서 탄력성을 극대화**할 수 있다.
  - **병렬 처리**: 하나의 메시지를 여러 서비스가 동시에 구독하여 각각의 로직을 수행하는 강력한 병렬 처리가 가능하다.

#### 3. 저장소 분리 패턴(Database per Service Pattern)

- 기존 모노리스 시스템은 주로 단일 통합 데이터베이스를 사용한다. 애플리케이션 모듈을 기능별로 분리해도 데이터베이스는 격리하지 않고 공유하므로, 강하게 결합된 구조다.
- **저장소 분리 패턴의 핵심 원칙**: 각 마이크로서비스는 자신만의 비즈니스를 처리하기 위한 데이터를 **독립적으로 직접 소유**해야 한다.
- **API를 통한 접근 통제**: 자신이 소유한 물리적 데이터베이스를 외부에 직접 노출하지 않으며, 다른 서비스의 데이터가 필요할 때는 오직 **해당 서비스가 공개한 API를 통해서만 접근**해야 한다.
- **폴리글랏(Polyglot) 저장소 구현**: 데이터 저장소가 서비스별로 완벽히 격리되므로, RDBMS나 NoSQL 등 **각 비즈니스 성격에 가장 적합한 데이터베이스 기술을 자율적으로 선택**할 수 있다.
- **발생하는 과제 (데이터 일관성)**: 이 패턴을 도입할 때 발생하는 가장 큰 과제는 분산된 여러 서비스에 걸쳐 복합적인 트랜잭션을 수행할 때 시스템 전체의 **비즈니스 정합성 및 데이터 일관성(Data Consistency)** 을 어떻게 유지할 것인가 하는 문제다.

#### 4. 분산 트랜잭션 처리 패턴(Distributed Transaction Pattern)

- 여러 개의 분산된 서비스에 걸쳐 비즈니스 로직이 수행될 때, 시스템 전체의 **데이터 일관성을 유지**하기 위한 방법이 필요하다.
- **전통적인 방법: 2단계 커밋(2-Phase Commit, 2PC)**
  - 분산 데이터베이스 환경에서 원자성(Atomicity)을 보장하기 위해, 모든 노드가 함께 커밋(Commit)되거나 롤백(Rollback)되는 강력한 일관성 보장 방식이다.
  - **한계점**: 각 데이터베이스에 **잠금(Lock)** 이 걸려 심각한 성능 저하가 발생한다. 네트워크 장애가 빈번한 클라우드 환경에서 특정 서비스의 트랜잭션 지연이 다른 정상 서비스까지 연쇄적인 대기 상태에 빠지게 만든다. 결국 2단계 커밋은 마이크로서비스의 핵심 철학인 **독립성과 자율성을 크게 훼손**한다.
- **MSA를 위한 해결책: 사가 패턴(Saga Pattern)**
  - 여러 분산된 서비스를 하나의 거대한 트랜잭션으로 강하게 묶지 않고, **각 서비스의 로컬 트랜잭션을 순차적으로 처리**하여 궁극적인 데이터 일관성(Eventual Consistency)을 맞추는 아키텍처 패턴이다.
  - 각 마이크로서비스는 자신의 로컬 데이터베이스를 업데이트한 후, **다음 트랜잭션을 트리거(Trigger)하는 이벤트나 메시지를 비동기로 발행**한다.
  - **보상 트랜잭션(Compensating Transaction)**: 처리 도중 특정 서비스에서 실패할 경우 데이터 정합성을 다시 맞춰야 한다. 이때 시스템 전체를 롤백하는 대신, 성공적으로 처리된 이전 트랜잭션들을 **의도적으로 되돌리는(취소하는) 보상 트랜잭션을 실행**하여 원래 상태로 복구한다.
  - **사가 패턴 적용 예시 (주문 서비스와 고객 서비스)**:
    1. **가주문 생성**: 주문 서비스는 미확정 상태의 가주문을 생성하고, 주문자 정보가 담긴 **'주문 생성됨' 이벤트**를 메시지 브로커로 발행한다.
    2. **신용 한도 확인**: 고객 서비스가 이벤트를 구독하여 다음 비즈니스를 처리한다. 한도가 충분하면 **'신용 승인됨' 이벤트**를, 부족하면 **'신용한도 초과됨' 이벤트**를 발행한다.
    3. **최종 상태 처리**: 주문 서비스는 고객 서비스가 발행한 이벤트를 확인해 최종 처리를 수행한다. '신용 승인됨'을 받으면 주문을 **'주문 승인' 상태로 업데이트**하고, '신용한도 초과됨'을 받으면 가주문을 취소하는 **보상 트랜잭션('주문 처리 취소')을 수행**한다.

#### 5. 데이터 일관성에 대한 생각의 전환: 결과적 일관성

- 모든 비즈니스 처리가 반드시 실시간성을 요구하는 것은 아니다. 데이터 일관성이 실시간으로 맞지 않더라도 일정 시점이 지났을 때 일관성을 만족하면 되는 개념을 **결과적 일관성(Eventual Consistency)** 이라고 한다.
- 결과적 일관성 개념은 시스템의 **고가용성을 극대화**한다. 마이크로서비스는 주로 사가 패턴과 이벤트 기반 비동기 통신을 적용하여 이를 구현한다.

- **CQRS 패턴(Command and Query Responsibility Segregation)**
  - **명령(Command)과 조회(Query)의 책임을 분리**하는 아키텍처 패턴이다.
  - 동일한 저장소에서 입력, 조회, 수정, 삭제를 모두 처리하던 기존 패러다임에서 벗어난 방식이다.
  - 사용자의 요청은 크게 시스템 상태를 변경하는 **명령**과 시스템 상태를 **조회**하는 것으로 나뉘며, 보통 조회 요청이 훨씬 많다.
  - 따라서 쓰기 모델과 읽기 모델을 분리하여 명령 서비스와 조회 서비스를 나누고, 더 나아가 **쓰기용 저장소와 조회용 저장소를 물리적으로 분리**할 수 있다.
  - **명령 측면 마이크로서비스**는 입력, 수정, 삭제를 처리하며 쓰기에 최적화된 **관계형 데이터베이스(RDBMS)** 를 사용한다.
  - **조회 측면 마이크로서비스**는 조회 성능이 뛰어난 **NoSQL(MongoDB, Elasticsearch 등)** 을 저장소로 사용하며, 인스턴스를 늘려(Scale-out) 대응하기 유리하다.
  - 이 구조에서는 명령 서비스에서 데이터가 변경될 때 조회 서비스와의 일관성이 깨질 수 있다. 이를 해결하기 위해 이벤트 주도 아키텍처를 도입하여, 명령 서비스가 데이터를 쓰면서 **이벤트를 발행**하고 조회 서비스가 이를 구독하여 **최신 상태로 동기화**한다.
- **API 조합과 CQRS 패턴을 활용한 서비스 연계**
  - 여러 하위 서비스의 데이터를 모아 상위 마이크로서비스(API 조합)를 만들면 하위 서비스에 강하게 의존하는 단점이 있다.
  - **CQRS 패턴**을 활용하면 각 하위 서비스가 변경 시점에 **이벤트를 발행**하고, 상위 서비스는 이 이벤트를 구독하여 자신의 저장소에 기록함으로써 데이터 일관성을 맞추고 조회 기능을 독립적으로 제공할 수 있다.

- **쓰기 최적화: 이벤트 소싱(Event Sourcing) 패턴**
  - 사가 패턴이나 CQRS 패턴에서 데이터 불일치를 막으려면, 저장소 기록 작업과 메시지 발행 작업이 분리되지 않고 **원자성(Atomicity)**을 가져야 한다.
  - 최종 상태를 계산하여 저장하는 대신, **상태 변경 트랜잭션(이벤트) 자체를 순서대로 저장하는 방식**을 **이벤트 소싱 기법**이라고 한다.
  - 상태 변경 이벤트를 계산하지 않고 그대로 **이벤트 저장소(Event Store)** 에 저장하므로 메시지 브로커와 데이터 저장소를 하나로 묶어 사용할 수 있다.
  - 현재 상태가 필요할 때는 저장된 상태 변경 트랜잭션을 처음부터 순차적으로 계산하여 복원한다.
  - 매번 처음부터 계산하는 것이 부담된다면 주기적으로 상태를 계산해 **스냅샷(Snapshot)** 으로 저장하고, 이후 발생한 트랜잭션만 추가로 계산하면 된다.
  - 특정 시점의 상태를 완벽히 재현할 수 있어 별도의 이력 로그 데이터를 남길 필요가 없다.
  - 이벤트는 한 번 발생하면 수정이나 삭제 없이 **입력(Append)만 되므로**, 동시성이나 데이터 정합성 문제에서 비교적 자유롭다.
