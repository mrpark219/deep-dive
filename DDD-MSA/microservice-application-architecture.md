# 마이크로서비스 애플리케이션 아키텍처

## 1. 비즈니스 로직은 어디에? - 관심사의 분리

- **비즈니스 로직**이란 보통 시스템의 목적인 비즈니스 영역의 업무 규칙, 흐름, 개념을 표현하는 용어다.
- 개발자의 역할은 문제 영역의 **비즈니스 로직을 분석하고 프로그래밍 언어로 잘 표현하는 것**이다.
- 설계 원칙 중 **관심사의 분리(Separation of Concerns)** 라는 원칙이 있다.
  - 이는 시스템의 각 영역이 처리하는 관심사가 분리되어 관리되어야 한다는 의미로, **시스템을 이해하고 변경하기 쉽게** 만든다.
- 기술과 비즈니스 로직을 분리하면 **복잡성이 낮아지고 유지보수성이 높아진다**.

### 1.1. 데이터베이스 중심 아키텍처의 문제점

- 특정 관계형 데이터베이스(RDBMS)에 의존하여 데이터 모델링을 수행하고, 이 **물리 테이블 모델을 중심으로 애플리케이션을 구현**하는 방식이다.
- 비즈니스 로직이 스프링 프레임워크(Spring Framework)의 서비스 객체가 아닌 **SQL 질의(Query)에 존재**하게 된다.
- 이 경우 특정 데이터베이스에 강하게 의존하므로 기술 변경이 어렵고, **시스템의 성능 또한 데이터베이스에 전적으로 의존**하게 된다.

## 2. 헥사고날 아키텍처와 클린 아키텍처

### 2.1. 레이어드 아키텍처

- **레이어드 아키텍처(Layered Architecture)** 를 구성하는 레이어(Layer)는 물리적인 장비를 뜻하는 티어(Tier)와 달리 **논리적인 개념**이다.
- 논리적 계층은 보통 화면을 처리하는 **프레젠테이션**, 비즈니스 흐름을 제어하는 **비즈니스 로직**, 데이터를 처리하는 **데이터 액세스(Data Access)** 등 3개로 구분한다.
- 레이어드 아키텍처는 다음과 같은 규칙을 따른다.
  - 상위 계층이 하위 계층을 호출하는 **단방향성을 유지**한다.
  - 상위 계층은 하위의 모든 계층을 알 필요 없이 바로 밑의 **근접 계층만 활용**한다.
  - 상위 계층이 하위 계층의 변경에 **영향을 받지 않아야** 한다.
  - 하위 계층은 자신을 호출하는 **상위 계층을 알지 못하게** 구성한다.
  - 계층 간 호출은 **인터페이스(Interface)를 통하는 것**이 바람직하다.
- 애플리케이션의 핵심인 비즈니스 로직은 **고수준 영역**이다. 따라서 **의존성 역전 원칙(DIP)**을 적용하여 데이터 액세스 계층의 인터페이스를 비즈니스 로직 계층으로 옮겨 외부 의존성을 줄여야 한다.

### 2.2. 헥사고날 아키텍처

- 레이어드 아키텍처에 DIP를 적용해도 한계가 존재한다. 현대 애플리케이션은 프레젠테이션이나 데이터 액세스 외에도 **훨씬 다양한 외부 인터페이스를 필요**로 하기 때문이다.
- 애플리케이션을 호출하는 다양한 외부 시스템과 연동되는 다수의 저장소가 존재한다.
- 기존의 단방향 계층 구조로는 이를 처리하기 어려워 **헥사고날 아키텍처(Hexagonal Architecture)** 가 등장했다.
- 이를 **포트 앤 어댑터 아키텍처(Ports and Adapters Architecture)** 라고도 부른다.
- 이 아키텍처는 고수준의 비즈니스 로직을 다루는 **내부 영역**과 인터페이스 처리를 담당하는 저수준의 **외부 영역**으로 시스템을 나눈다.
- 내부 영역은 외부 기술에 의존하지 않는 **순수한 비즈니스 로직을 표현**하며, 외부 영역과 통신하기 위한 **포트(Port)** 를 제공한다.
- 외부 영역은 외부 요청을 내부로 전달하는 **인바운드 어댑터(Inbound Adapter)** 와, 내부 비즈니스 로직이 외부 시스템을 호출할 때 사용하는 **아웃바운드 어댑터(Outbound Adapter)** 로 구성된다.
- 헥사고날 아키텍처의 핵심은 **고수준의 내부 영역이 저수준의 외부 구체 어댑터에 전혀 의존하지 않는다는 것**이며, 이는 내부 영역이 제공하는 포트를 통해 달성된다.
- 즉, 외부의 **아웃바운드 어댑터가 내부의 아웃바운드 포트에 의존**하여 구현되는 구조다.

### 2.3. 클린 아키텍처

- 소프트웨어는 행위 가치와 구조 가치를 가지는데, 장기적인 유지보수를 위해서는 **구조 가치가 더 중요**하다.
- 소프트웨어를 유연하게 유지하려면 **시스템의 핵심 구조를 외부 기술의 선택으로부터 오랫동안 보호**해야 한다.
- **클린 아키텍처(Clean Architecture)** 는 동심원 형태의 여러 영역으로 구성되며, 중심부부터 **엔티티(Entity), 유즈케이스(Use Case), 세부 사항(Details)** 으로 구분된다.
- **업무 규칙**이란 사업적 수익 창출이나 비용 절감을 위한 핵심 절차를 뜻하며, 이는 수동으로 처리할 수도 있지만 소프트웨어 시스템으로 자동화할 수 있다.
- 도메인의 핵심 업무 규칙은 데이터를 요구하며, 이 규칙과 데이터를 결합하여 만든 객체를 **엔티티(Entity)** 라고 부른다.
- 엔티티를 감싸는 **유즈케이스(Use Case)** 는 애플리케이션에 특화된 업무 규칙을 표현하고, **엔티티 내부의 로직을 호출하여 시스템의 전체 흐름을 제어**한다.
- 이때 엔티티 같은 고수준 영역은 상대적으로 저수준인 **유즈케이스 영역을 알지 못하게 설계**해야 한다.
- 유즈케이스 바깥을 감싸는 모든 영역(UI, DB, 웹 서버, 프레임워크, 통신 프로토콜 등)은 **세부 사항**에 해당한다.
- 이 세부 사항과 유즈케이스의 관계는 의존성 역전 원칙(DIP)을 활용하여 **플러그인처럼 유연하게 교체할 수 있도록 구성**해야 한다.

## 3. 마이크로서비스의 내부 구조 정의

### 3.1. 바람직한 마이크로서비스의 내부 아키텍처: 클린 마이크로서비스

- **관심사의 분리, 헥사고날 아키텍처, 클린 아키텍처**는 거대한 모노리스 소프트웨어를 통제하기 위해 등장한 개념이다.
- 마이크로서비스는 복잡한 모노리스를 여러 기능으로 쪼개어 시스템의 **복잡성을 완화**한다.
- 마이크로서비스는 자율적인 팀에 의해 다양한 기술과 언어를 사용하는 **폴리글랏(Polyglot) 구조**를 가질 수 있으므로, 내부 아키텍처 역시 다양하게 구성될 수 있다.
- 각 서비스는 목적에 맞는 **적절한 개발 언어, 저장소, 내부 아키텍처**를 정의하는 것이 바람직하다.
- 3가지 아키텍처의 핵심 원칙을 종합하면 다음과 같다.
  - 응집성을 높이고 다른 관심사 영역과의 **의존도를 낮춰야 한다**.
  - 비즈니스 로직(고수준) 영역을 세부 기술(저수준) 중심의 외부 영역으로부터 **철저히 분리**한다.
  - 고수준 영역은 저수준 영역에 의존하지 않아야 하며, 반드시 **저수준 영역이 고수준 영역에 의존**하게 설계해야 한다.
  - 저수준 영역의 교체나 확장이 **고수준 영역에 영향을 주어서는 안 된다**.
  - 자바(Java)처럼 인터페이스를 지원하는 언어는 저수준의 구체 클래스가 고수준의 추상 인터페이스에 의존하도록 **의존성 역전 원칙(DIP)** 을 적용한다.

### 3.2. 내부 영역 - 업무 규칙

- 업무 규칙을 정의하는 내부 영역에는 서비스 인터페이스, 구현체, 도메인, 리포지토리 인터페이스, 도메인 이벤트 인터페이스 등이 있다.
- 서비스 인터페이스는 외부가 내부의 세부 구현을 알지 못하게 막아주어 강력한 **정보 은닉 효과**를 제공한다.
- 리포지토리 및 각종 인터페이스는 외부와의 결합도를 낮추고 **의존성 역전 원칙(DIP)** 을 지원한다.
- **도메인**은 비즈니스 개념을 표현하고, **서비스**는 도메인을 활용해 시스템의 전체 흐름을 제어한다.

#### 트랜잭션 스크립트 패턴

- 비즈니스 개념을 표현하는 도메인 객체가 어떠한 **로직(행위)도 가지지 않는 패턴**이다.
- 모든 비즈니스 행위와 처리 책임이 **오직 서비스(Service)에 집중**된다.
- 서비스가 절차에 따라 처리를 수행하므로 점차 비대해지고, 도메인 객체는 단순한 **데이터 묶음(DTO)** 으로 전락한다.
- 절차지향적 프로그래밍과 유사하여 이해하기 쉽고, 데이터베이스 중심 설계에 익숙한 개발자가 적응하기 편하므로 **간단한 비즈니스를 처리**할 때 적합하다.

#### 도메인 모델 패턴

- 도메인 객체가 단순한 데이터를 넘어 고유한 **비즈니스 행위까지 포함**하는 패턴이다.
- 도메인 객체의 데이터는 자신이 제공하는 행위에 의해 안전하게 **은닉**된다.
- 서비스는 시스템 유즈케이스 흐름만 제어하고, 실제 비즈니스 로직 처리는 **도메인 객체에 위임**한다.
- 책임이 도메인으로 분산되어 **서비스가 단순하고 명료해진다**.
- 진정한 의미의 **객체지향 설계(Object-Oriented Design)** 모델이며, 도메인 주도 설계(DDD)의 **애그리거트 패턴**을 적용하기 좋은 기반이 된다.

#### 도메인 주도 설계의 애그리거트 패턴

- 업무가 복잡해질수록 객체 간의 다단계 참조가 발생하여 **도메인 모델의 결합도가 높아지고 경계가 모호해지는 문제**를 해결하기 위한 고급 패턴이다.
- 연관된 도메인 개념들을 데이터 일관성 유지를 위해 **최상위 엔티티(루트)를 중심으로 묶어낸 군집**이 바로 **애그리거트(Aggregate)** 다.
- 애그리거트를 일관되게 처리하기 위해 다음과 같은 엄격한 규칙을 적용한다.
  - 외부에서는 반드시 애그리거트의 대표인 **루트(Root) 엔티티만 참조**할 수 있다.
  - 내부 상세 클래스는 직접 참조하거나 수정할 수 없으며, **오직 루트를 통해서만 접근**해야 한다.
  - 다른 애그리거트를 참조할 때는 객체를 직접 연관 짓지 않고 **기본 키(ID)를 사용**하여 참조 결합도를 낮춘다.
  - 데이터 정합성을 위해 하나의 트랜잭션에서는 **오직 하나의 애그리거트만 수정**한다.

### 3.3. 외부 영역 - 세부사항

#### API 퍼블리싱 어댑터

- 외부 클라이언트에게 REST API를 제공하는 **인바운드 어댑터**다.
- 내부 서비스 인터페이스를 호출한 뒤, 엔티티를 직접 노출하지 않고 API 규격에 맞는 **DTO(Data Transfer Object)** 로 변환하여 응답한다.

#### API 프록시 어댑터

- 우리 시스템이 다른 외부 서비스의 API를 호출할 때 사용하는 **아웃바운드 어댑터**다.
- 내부 영역에 정의된 프록시 인터페이스를 구현하며, REST, 소켓, SOAP 등 **다양한 프로토콜**을 사용하여 외부와 통신한다.

#### 저장소 처리 어댑터

- 데이터를 물리적으로 저장하고 조회하기 위한 데이터 처리 메커니즘이다.
- 트랜잭션 스크립트 패턴을 쓸 때는 주로 **SQL 매핑 방식(MyBatis 등)** 을 사용하고, 도메인 모델 패턴을 적용할 때는 **ORM(JPA 등)** 방식을 많이 선택한다.

#### 도메인 이벤트 발행 어댑터

- 서비스 간 비동기 통신을 위해 이벤트를 외부 메시지 브로커 등으로 전송하는 **아웃바운드 어댑터**다.
- 시스템에서 발생한 핵심 상태 변경 사항(예: '주문 완료됨')을 담은 **도메인 이벤트(Domain Event)** 를 다른 서비스(컨슈머)에 전달하기 위해 발행한다.

#### 도메인 이벤트 핸들러

- 외부 시스템이 발행한 도메인 이벤트를 수신(구독)하여 **내부 영역으로 전달**하는 역할을 수행한다.
- 이벤트 종류와 상태에 따라 적절한 **내부 서비스 인터페이스를 호출**하여 후속 비즈니스 로직이 이어지도록 처리한다.
