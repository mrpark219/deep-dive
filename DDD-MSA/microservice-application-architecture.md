# 마이크로서비스 애플리케이션 아키텍처

## 1. 비즈니스 로직은 어디에? - 관심사의 분리

- **비즈니스 로직**이란 보통 시스템의 목적인 비즈니스 영역의 업무 규칙, 흐름, 개념을 표현하는 용어다.
- 개발자의 역할은 문제 영역의 **비즈니스 로직을 분석하고 프로그래밍 언어로 잘 표현하는 것**이다.
- 설계 원칙 중 **관심사의 분리(Separation of Concerns)** 라는 원칙이 있다.
  - 이는 시스템의 각 영역이 처리하는 관심사가 분리되어 관리되어야 한다는 의미로, **시스템을 이해하고 변경하기 쉽게** 만든다.
- 기술과 비즈니스 로직을 분리하면 **복잡성이 낮아지고 유지보수성이 높아진다**.

### 1.1. 데이터베이스 중심 아키텍처의 문제점

- 특정 관계형 데이터베이스(RDBMS)에 의존하여 데이터 모델링을 수행하고, 이 **물리 테이블 모델을 중심으로 애플리케이션을 구현**하는 방식이다.
- 비즈니스 로직이 스프링 프레임워크(Spring Framework)의 서비스 객체가 아닌 **SQL 질의(Query)에 존재**하게 된다.
- 이 경우 특정 데이터베이스에 강하게 의존하므로 기술 변경이 어렵고, **시스템의 성능 또한 데이터베이스에 전적으로 의존**하게 된다.

## 2. 헥사고날 아키텍처와 클린 아키텍처

### 2.1. 레이어드 아키텍처

- **레이어드 아키텍처(Layered Architecture)** 를 구성하는 레이어(Layer)는 물리적인 장비를 뜻하는 티어(Tier)와 달리 **논리적인 개념**이다.
- 논리적 계층은 보통 화면을 처리하는 **프레젠테이션**, 비즈니스 흐름을 제어하는 **비즈니스 로직**, 데이터를 처리하는 **데이터 액세스(Data Access)** 등 3개로 구분한다.
- 레이어드 아키텍처는 다음과 같은 규칙을 따른다.
  - 상위 계층이 하위 계층을 호출하는 **단방향성을 유지**한다.
  - 상위 계층은 하위의 모든 계층을 알 필요 없이 바로 밑의 **근접 계층만 활용**한다.
  - 상위 계층이 하위 계층의 변경에 **영향을 받지 않아야** 한다.
  - 하위 계층은 자신을 호출하는 **상위 계층을 알지 못하게** 구성한다.
  - 계층 간 호출은 **인터페이스(Interface)를 통하는 것**이 바람직하다.
- 애플리케이션의 핵심인 비즈니스 로직은 **고수준 영역**이다. 따라서 **의존성 역전 원칙(DIP)**을 적용하여 데이터 액세스 계층의 인터페이스를 비즈니스 로직 계층으로 옮겨 외부 의존성을 줄여야 한다.

### 2.2. 헥사고날 아키텍처

- 레이어드 아키텍처에 DIP를 적용해도 한계가 존재한다. 현대 애플리케이션은 프레젠테이션이나 데이터 액세스 외에도 **훨씬 다양한 외부 인터페이스를 필요**로 하기 때문이다.
- 애플리케이션을 호출하는 다양한 외부 시스템과 연동되는 다수의 저장소가 존재한다.
- 기존의 단방향 계층 구조로는 이를 처리하기 어려워 **헥사고날 아키텍처(Hexagonal Architecture)** 가 등장했다.
- 이를 **포트 앤 어댑터 아키텍처(Ports and Adapters Architecture)** 라고도 부른다.
- 이 아키텍처는 고수준의 비즈니스 로직을 다루는 **내부 영역**과 인터페이스 처리를 담당하는 저수준의 **외부 영역**으로 시스템을 나눈다.
- 내부 영역은 외부 기술에 의존하지 않는 **순수한 비즈니스 로직을 표현**하며, 외부 영역과 통신하기 위한 **포트(Port)** 를 제공한다.
- 외부 영역은 외부 요청을 내부로 전달하는 **인바운드 어댑터(Inbound Adapter)** 와, 내부 비즈니스 로직이 외부 시스템을 호출할 때 사용하는 **아웃바운드 어댑터(Outbound Adapter)** 로 구성된다.
- 헥사고날 아키텍처의 핵심은 **고수준의 내부 영역이 저수준의 외부 구체 어댑터에 전혀 의존하지 않는다는 것**이며, 이는 내부 영역이 제공하는 포트를 통해 달성된다.
- 즉, 외부의 **아웃바운드 어댑터가 내부의 아웃바운드 포트에 의존**하여 구현되는 구조다.

### 2.3. 클린 아키텍처

- 소프트웨어는 행위 가치와 구조 가치를 가지는데, 장기적인 유지보수를 위해서는 **구조 가치가 더 중요**하다.
- 소프트웨어를 유연하게 유지하려면 **시스템의 핵심 구조를 외부 기술의 선택으로부터 오랫동안 보호**해야 한다.
- **클린 아키텍처(Clean Architecture)** 는 동심원 형태의 여러 영역으로 구성되며, 중심부부터 **엔티티(Entity), 유즈케이스(Use Case), 세부 사항(Details)** 으로 구분된다.
- **업무 규칙**이란 사업적 수익 창출이나 비용 절감을 위한 핵심 절차를 뜻하며, 이는 수동으로 처리할 수도 있지만 소프트웨어 시스템으로 자동화할 수 있다.
- 도메인의 핵심 업무 규칙은 데이터를 요구하며, 이 규칙과 데이터를 결합하여 만든 객체를 **엔티티(Entity)** 라고 부른다.
- 엔티티를 감싸는 **유즈케이스(Use Case)** 는 애플리케이션에 특화된 업무 규칙을 표현하고, **엔티티 내부의 로직을 호출하여 시스템의 전체 흐름을 제어**한다.
- 이때 엔티티 같은 고수준 영역은 상대적으로 저수준인 **유즈케이스 영역을 알지 못하게 설계**해야 한다.
- 유즈케이스 바깥을 감싸는 모든 영역(UI, DB, 웹 서버, 프레임워크, 통신 프로토콜 등)은 **세부 사항**에 해당한다.
- 이 세부 사항과 유즈케이스의 관계는 의존성 역전 원칙(DIP)을 활용하여 **플러그인처럼 유연하게 교체할 수 있도록 구성**해야 한다.
