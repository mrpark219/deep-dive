# 설계와 아키텍처란?

## 1. 설계(Design) VS 아키텍처(Architecture)

- 흔히 **아키텍처**는 저수준의 세부 사항과는 분리된 **고수준의 무언가**를 가리킬 때 사용되는 반면, **설계**는 **저수준의 구조** 또는 결정 사항 등을 의미할 때가 많다.
- 하지만 저수준의 세부 사항과 고수준의 구조는 모두 소프트웨어 전체 설계의 **구성 요소**이다.
  - 개별적으로 존재할 수 없으며, 실제로 이 둘을 구분 짓는 **경계**는 뚜렷하지 않다.
- 고수준에서 저수준으로 향하는 **의사결정의 연속성**만이 있을 뿐이다.

## 2. 좋은 소프트웨어란?

- 소프트웨어 아키텍처의 **목표**는 필요한 시스템을 만들고 유지보수하는 데 투입되는 **인력**을 최소화하는 데 있다.
- "**코드**는 나중에 **정리**하면 돼. 시장에 출시하는 게 먼저야"라는 거짓말에 속은 개발자가 나중에 코드를 정리하는 경우는 거의 없다.
  - **시장**의 압박은 절대로 수그러들지 않기 때문이다.
- 이것이 수차례 반복되면 결국 엉망진창이 되고, **생산성**은 **0**을 향해 수렴하기 시작한다.
  - 엉망진창인 코드가 쌓이면 개발자의 생산성은 차츰 낮아지고, 코드가 엉망이 되는 추세는 멈추거나 수그러들지 않는다.
- **진실**은 엉망으로 만들면 깔끔하게 유지할 때보다 **항상 더 느리다**는 것이다.
