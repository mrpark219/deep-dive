# 패러다임 개요

## 1. 패러다임

- 패러다임에는 **구조적 프로그래밍**, **객체 지향 프로그래밍**, **함수형 프로그래밍** 3가지가 있다.

## 2. 구조적 프로그래밍 (Structured Programming)

- **최초로 적용**된 패러다임이다(최초로 만들어진 패러다임은 아니다).
- 무분별한 점프(goto 문장)가 프로그램 구조에 해롭다는 사실에 기반한다.
- **제어 흐름**의 **직접적인 전환**에 대해 규칙을 부여한다.
  - - 프로그램의 순서를 내키는 대로 점프하지 못하게 막고, 반드시 정해진 입구와 출구를 가진 **제어 구조(조건문, 반복문)** 만 사용하도록 강제한다.

## 3. 객체 지향 프로그래밍 (Object-Oriented Programming)

- 두 번째로 도입된 패러다임이지만, 구조적 프로그래밍보다 **먼저** 등장했다.
- **제어 흐름**의 **간접적인 전환**에 대해 규칙을 부여한다.
  - 함수 호출을 직접 연결(A가 B를 직접 호출)하는 대신, **인터페이스**라는 중간 벽을 두어 소스 코드의 의존 방향을 제어(**의존성 역전**)할 수 있게 만든다.

## 4. 함수형 프로그래밍 (Functional Programming)

- **가장 먼저** 만들어진 패러다임이다.
- 함수형 프로그래밍은 **할당문**에 대한 규칙을 부과한다(불변성).
  - 변수에 한 번 들어간 값이 **절대로** 바뀌지 않게 강제함으로써, 여러 곳에서 동시에 데이터를 수정할 **때** 발생하는 복잡한 부작용을 원천적으로 차단한다.

## 5. 결론

- 3가지 프로그래밍 패러다임은 프로그래머에게서 **권한**을 박탈한다.
- 각 패러다임은 부정적인 의도를 가지는 일종의 **추가적인 규칙**을 부과한다.
  - 즉, 패러다임은 무엇을 해야 할지를 말하기보다는 무엇을 **해서는 안 되는지**를 말해준다.
- 세 가지 패러다임과 아키텍처의 **세 가지 큰 관심사**(함수, 컴포넌트 분리, 데이터 관리)가 어떻게 서로 연관되는지 주목하면 좋다.
