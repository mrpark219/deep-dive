# 네트워크 예외

## 1. 연결 예외

### 1.1 java.net.UnknownHostException

- 입력한 호스트 이름이나 IP 주소를 시스템이 아예 찾을 수 없을 때 발생하는 예외이다.
- `999.999.999.999`처럼 존재하지 않는 IP를 입력하거나, `google.gogo`처럼 존재하지 않는 도메인 이름을 사용할 때 발생한다.

### 1.2. java.net.ConnectException: Connection refused

- **`Connection refused`** 메시지는 네트워크를 통해 대상 IP의 서버 컴퓨터 자체에는 도달했지만, 최종적인 통신 연결은 거절되었다는 뜻이다.
- 대상 IP의 서버는 켜져 있지만, 클라이언트가 접속을 시도한 **포트(PORT)** 를 서버 애플리케이션이 열어두지 않았을 때 주로 발생한다.
- 또는 네트워크 방화벽 등에서 해당 요청을 무단 연결로 인지하고 강제로 막을 때도 발생할 수 있다.

#### 윈도우 OS

- 윈도우 운영체제의 경우 기본 예외 메시지 끝에 `connect`가 추가로 붙어 **`java.net.ConnectException: Connection refused: connect`** 형태로 출력되는 차이가 있다.

#### TCP RST(Reset) 패킷

- 클라이언트의 접속을 거절할 때 서버 컴퓨터의 운영체제(OS)는 클라이언트에게 **TCP RST(Reset)** 패킷을 보낸다.
- 이 패킷은 현재 TCP 연결에 문제가 있으니 강제로 끊으라는 뜻이며, 클라이언트가 연결 시도 중 이 패킷을 받으면 즉시 연결을 해제하면서 해당 예외가 발생하게 된다.

## 2. Timeout 예외

```java
socket.connect(new InetSocketAddress("192.168.1.250", 45678), 1000);
```

- 클라이언트가 해당 IP로 연결 패킷을 보냈지만 서버가 존재하지 않거나, 서버에 문제가 생겨 응답 패킷을 보내지 못할 때 발생한다.
- 이때 무한정 기다리는 것을 방지하기 위해 운영체제(OS)에는 **연결 대기 타임아웃**이 기본적으로 설정되어 있다.
- OS별 기본 대기 시간은 윈도우 약 21초, 리눅스 75~180초, 맥(Mac) 약 75초 수준이다.
- 이 대기 시간이 지나면 **`java.net.ConnectException: Operation timed out`** 예외가 발생한다.
- 자바의 `Socket` 객체를 생성할 때 IP와 PORT를 전달하면 즉시 연결을 시도하지만, 인자 없이 빈 소켓만 생성하면 연결을 보류할 수 있다.
- 빈 소켓을 생성한 뒤 **`socket.connect(endpoint, timeout)`** 을 호출하면, 사용자가 밀리초(ms) 단위로 지정한 타임아웃 시간을 적용하여 연결을 시도할 수 있다.
- 연결 대상 정보는 `SocketAddress`의 자식 객체인 **`InetSocketAddress`**(IP, PORT 기반)를 사용하여 지정한다.
- 설정한 타임아웃 시간이 지나도 연결되지 않으면 **`java.net.SocketTimeoutException: Connect timed out`** 예외가 발생한다.

### 2.1. Read 타임아웃

```java
socket.setSoTimeout(3000); // 타임아웃 시간 설정
```

- 앞에서 설명한 '연결 타임아웃'이 TCP 접속 단계에서 발생한다면, **'Read 타임아웃(소켓 타임아웃)'** 은 연결이 성공한 이후 데이터를 읽는 과정에서 발생한다.
- 클라이언트가 요청을 보냈는데, 서버 트래픽 폭주 등의 이유로 응답이 계속 오지 않을 때 무한정 기다리는 것을 방지하기 위해 사용한다.
- **`socket.setSoTimeout()`** 메서드를 사용하여 응답을 기다릴 최대 시간(밀리초 단위)을 설정할 수 있다.
- 설정한 시간(예: 3초)이 지나도 데이터를 읽지 못하면 **`java.net.SocketTimeoutException: Read timed out`** 예외가 발생한다.
- 만약 Read 타임아웃 시간을 별도로 설정하지 않으면, `read()` 메서드는 서버의 응답이 올 때까지 **무한 대기(블로킹)** 상태에 빠지게 되므로 주의해야 한다.

## 3. TCP 연결 종료

- TCP 네트워크 통신에는 강제 종료와 정상 종료, 2가지 종류의 종료 방식이 존재한다.

### 3.1. TCP 정상 종료 과정 (FIN 패킷 교환)

- TCP 통신에서 양 종단(A, B)이 연결을 정상적으로 종료하려면 서로 **FIN(Finish)** 패킷을 주고받아야 한다.
- 한쪽(예: 서버)에서 연결 종료를 위해 `socket.close()`를 호출하면, TCP 계층에서 종료를 의미하는 **FIN 패킷**을 상대방(클라이언트)에게 전달한다.
- FIN 패킷을 받은 클라이언트의 운영체제(OS)는 확인 응답으로 **ACK 패킷**을 서버에 보낸다.
- 클라이언트 역시 완전히 연결을 끝내기 위해 즉시 `socket.close()`를 호출하여 서버 측으로 자신의 **FIN 패킷**을 보낸다.
- 마지막으로 서버의 OS가 클라이언트의 FIN 패킷에 대한 **ACK 패킷**을 응답하면, 서로 FIN 메시지 교환이 완료되며 TCP 연결이 정상 종료된다.

#### 입력 스트림별 EOF(End of File) 처리 방식

- 클라이언트가 서버의 데이터를 읽기 위해 대기하던 중 서버가 먼저 `close()`를 호출하여 FIN 패킷을 보내면, 클라이언트는 더 이상 읽을 데이터가 없음을 인지하게 된다.
- 이렇게 FIN 패킷을 받아 더는 읽을 데이터가 없는 상태를 **EOF(End of File)** 라고 부르며, 클라이언트가 메시지를 읽는 3가지 방식에 따라 반환 결과가 각각 다르다.
- **`inputStream.read()`**: 데이터를 1바이트(byte) 단위로 읽으며, EOF 도달 시 데이터가 없음을 의미하는 숫자 **`-1`** 을 반환한다.
- **`BufferedReader.readLine()`**: 데이터를 라인(`String`) 단위로 읽기 때문에 `-1`을 문자로 표현할 수 없어 **`null`** 을 반환한다.
- **`DataInputStream.readUTF()`**: `String` 단위로 데이터를 읽는 보조 스트림으로, 반환값 대신 **`EOFException`** 예외를 직접 발생시킨다.
- 이 세 가지 방식은 결과 형태만 다를 뿐 모두 **상대방이 FIN 메시지를 보내며 소켓 연결을 끊었다**는 동일한 의미를 가진다.
- 따라서 EOF(`-1`, `null`, `EOFException`)가 발생했다면 해당 소켓으로 다른 통신 작업을 시도해서는 안 되며, **즉시 내 쪽에서도 `close()`를 호출**하여 상대방에게 FIN 패킷을 보내고 소켓 연결을 완전히 끊어야 한다.

### 3.2. 강제 종료와 RST(Reset) 패킷

- TCP 통신 중 예상치 못한 문제가 발생하면 연결 상태를 강제로 초기화(무효화)하기 위해 **RST(Reset)** 패킷이 전송된다.
- 예를 들어, 서버가 종료를 위해 `close()`를 호출하여 **FIN 패킷**을 보냈는데, 클라이언트가 이를 무시하고 `write()`를 통해 데이터(**PUSH 패킷**)를 전송하는 비정상적인 상황이 발생할 수 있다.
- 서버는 정상적인 종료(FIN 패킷 수신)를 기대하는 상태에서 데이터가 들어오면 TCP 연결에 문제가 있다고 판단하여, 클라이언트에게 즉시 연결을 끊으라는 **RST 패킷**을 전송한다.
- 클라이언트 측에 RST 패킷이 도착했다는 것은 현재 세션이 강제 종료되었음을 의미하므로 해당 연결을 더 이상 사용해서는 안 된다.
- RST 패킷을 받은 후 클라이언트가 `read()`로 데이터를 읽으려 하면, Mac은 **`java.net.SocketException: Connection reset`**, 윈도우는 **`현재 연결은 사용자의 호스트 시스템의 소프트웨어에 의해 중단되었습니다`** 예외를 던진다.
- 마찬가지로 RST 패킷을 받은 후 `write()`로 데이터를 전송하려 하면, Mac은 **`java.net.SocketException: Broken pipe`**, 윈도우는 `read()` 상황과 동일한 예외를 던진다.

#### 참고 - 비정상 종료 관련 부가 정보

- **RST 패킷이 발생하는 대표적인 상황**: 앞선 예시처럼 TCP 스펙에 맞지 않는 순서로 메시지가 전달될 때, 수신 버퍼에 아직 다 읽지 않은 데이터가 남아있는데 연결을 종료할 때, 방화벽 등 네트워크 장비에서 연결을 강제로 차단할 때 주로 발생한다.
- **`java.net.SocketException: Socket is closed` 예외**: 상대방의 RST 패킷 전송과 무관하게, 내 쪽에서 이미 소켓을 닫은(`close()`) 이후에 실수로 `read()`나 `write()` 메서드를 다시 호출할 때 발생하는 예외이다.

#### 네트워크 예외 및 종료 상황 정리

- **정상 종료 (EOF 수신)**: 상대방이 정상적으로 연결을 종료한 경우 데이터를 읽으면 `-1`, `null`, `EOFException` 형태의 **EOF**가 발생하며, 이 경우 내 쪽에서도 소켓의 `close()`를 호출하여 완전히 연결을 끊어야 한다.
- **`Connection reset` 예외**: 강제 종료를 의미하는 RST 패킷을 받은 이후에 내 쪽에서 **`read()`** 메서드를 호출할 때 발생한다. (윈도우 환경: `현재 연결은 사용자의 호스트 시스템의 소프트웨어에 의해 중단되었습니다`)
- **`Broken pipe` 예외**: 강제 종료를 의미하는 RST 패킷을 받은 이후에 내 쪽에서 **`write()`** 메서드를 호출할 때 발생한다. (윈도우 환경은 `현재 연결은 사용자의 호스트 시스템의 소프트웨어에 의해 중단되었습니다`)
- **`Socket is closed` 예외**: 상대방의 연결 종료 여부와 상관없이, **자신이 직접 소켓을 닫은 이후**에 다시 `read()` 또는 `write()` 메서드를 호출할 때 발생한다.
