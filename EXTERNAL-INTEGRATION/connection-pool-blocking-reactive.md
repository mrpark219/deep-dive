# 커넥션 풀 산정 및 블로킹 vs 리액티브 모델 비교

## 1. 커넥션 풀 붕괴 메커니즘

- 외부 서비스 지연이 발생했을 때 자원의 점유 시간이 길어지고, 이는 커넥션 풀을 빠르게 고갈시킨다.
- 외부 서비스 지연이 없더라도 풀 크기가 부족한 상태에서 많은 요청이 들어오면 요청을 정상적으로 처리할 수 없다.

### 1.1. 핵심 자원별 점유 구간

| 자원                       | 점유 구간               | 점유 시간                                                                  | 비고                                               |
| :------------------------- | :---------------------- | :------------------------------------------------------------------------- | :------------------------------------------------- |
| **Tomcat 스레드**          | 요청 시작 ~ 응답 완료   | $W_{\text{total}} \approx W_{\text{app}} + W_{\text{ext}} + W_{\text{db}}$ | I/O 대기 동안 스레드 블로킹 발생                   |
| **HTTP 커넥션 (외부 API)** | 각 외부 API 호출 동안만 | $W_{\text{ext}}$                                                           | 외부 지연 시 가장 먼저 고갈                        |
| **HikariCP (DB 커넥션)**   | 짧은 트랜잭션 동안만    | $W_{\text{db}}$                                                            | 외부 호출을 트랜잭션 밖으로 빼면 DB 풀을 잡지 않음 |

> **💡 수식 기호 정의**
>
> - $W_{\text{total}}$: 전체 요청 처리 시간
> - $W_{\text{app}}$: 애플리케이션 내부 로직 연산 시간
> - $W_{\text{ext}}$: 외부 API(PG, 포인트 등) 호출 대기 시간
> - $W_{\text{db}}$: DB 쿼리 실행 및 트랜잭션 처리 시간

### 1.2. 커넥션 풀 고갈 시 붕괴 흐름

1. **외부 지연 발생**: 외부 API 처리 지연 발생 시 HTTP 커넥션 점유 시간($W_{\text{ext}}$)이 급증한다.
2. **HTTP Pool 고갈**: 설정된 HTTP 커넥션 풀(예: `max-total=5`)이 순식간에 가득 찬다.
3. **Tomcat 스레드 병목**: HTTP 커넥션을 얻지 못한 요청들이 Tomcat 스레드를 잡은 채 대기(Pending) 상태에 빠진다.
4. **시스템 전체 마비**: 타임아웃 초과 및 서킷 브레이커(Circuit Breaker) 발동으로 정상적인 요청까지 연쇄 실패한다. (`PENDING` 처리 및 성공률 급감)

## 2. 처리량 및 자원 기반 커넥션/스레드 풀 크기 산정

### 2.1. 산정 공식: Little's Law

- 경험적 수치가 아닌 시스템 처리량과 지연 시간을 바탕으로 수학적으로 적정 커넥션 풀 크기를 계산한다.
- $L = \lambda \times W$
  - **$L$ (필요한 풀 크기)**: 시스템 내에 동시 존재해야 하는 요청 및 자원 수 (In-flight Requests)
  - **$\lambda$ (목표 처리량)**: 초당 처리 요청 수 (TPS, Throughput)
  - **$W$ (평균 지연 시간)**: 1개 요청을 처리하는 데 걸리는 시간 (Latency, Sec)

### 2.2. 자원별 핵심 의미 및 산정 기준

- 시스템 요소마다 지연 시간($W$)의 주체와 영향도가 다르므로, Little's Law를 적용하는 기준을 구분하여 설정한다.

| 자원               | 산정 공식 / 기준          | 역할 및 핵심 특징                                                                                                            |
| :----------------- | :------------------------ | :--------------------------------------------------------------------------------------------------------------------------- |
| **HTTP 풀**        | `maxPerRoute ≥ λ × W_ext` | **외부 API 호출용 커넥션**<br>외부 시스템 지연($W_{\text{ext}}$)이 길므로, 외부가 수용 가능한 TPS 상한 내에서 넉넉하게 설정  |
| **Hikari (DB)**    | `≥ λ × W_db`              | **RDB 연동용 커넥션**<br>DB 쿼리 지연($W_{\text{db}}$)은 상대적으로 매우 짧아 적은 풀 크기로도 디커플링 가능                 |
| **Tomcat threads** | `≥ λ × W_total`           | **요청 처리 스레드**<br>전체 지연($W_{\text{total}}$) 동안 점유되며, 외부 I/O 지연 발생 시 가장 먼저 고갈되는 병목 지점      |
| **Bulkhead (ext)** | `= ext 동시 호출 상한`    | **장애 격리 벽 (Resilience4j 등)**<br>외부 호출 동시 실행 수를 제한하며, HTTP 풀보다 작을 경우 이 지점이 먼저 병목 역할을 함 |

> **💡 실무 세팅 (여유율 고려 예시)**
>
> - 계산 결과가 $L = 25 \times 0.4\text{s} = 10$개이더라도, 순간 부하 및 커넥션 획득 대기를 고려해 **30개** 수준으로 여유를 둔다 (`max-total: 30`, `max-per-route: 30`).
> - 풀을 적정 수준으로 증설하면 외부 지연 상황에서도 요청 대기열을 유연하게 수용할 수 있다.
